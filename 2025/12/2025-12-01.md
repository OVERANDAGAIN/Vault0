## **一、Fig. 9 分析（论文原意）**

这幅图（Fig. 9）展示的是：
**当链路发生中断后，一个 LTP block 是如何从中断中恢复的整个过程的“细节视图”。**

横轴是 **Data Delivery Time**（数据送达时间），纵轴是 **Sequence Offset**（序列偏移量），代表数据段在 block 中的位置。

图上出现几个关键事件点：

---

### **① 左上角的红色 “R ×”：Last CP retransmission**

* “R ×” 表示 **Checkpoint（CP） segment 的重传**。
* 图中写的是 “Last CP retransmission”，意思是：
  **最后一次 CP 重传动作发生在这里。**

为什么会出现 CP 重传？
— 因为之前链路中断导致原本的 CP 没有送到接收端，所以必须靠 **CP 的 RTO 超时机制**触发重传。

这与 IV-A 的 Case 1-B 完全一致。

---

### **② 底部中间的事件：RS acknowledgment in response to CP**

当接收端最终收到 CP 后，会产生一个 **Report Segment（RS）**。
这是 LTP 可靠传输中的关键机制：

* 接收端用 RS 告诉发送端：
  “哪些 segments 我收到了，哪些丢了，需要重传哪些。”

在图上，这个 RS 是在 **81.5s–81.6s 左右的位置**返回的。

这说明：

* CP 的重传终于成功
* 接收端马上触发 RS
* 系统恢复进入“正常重传流程”

---

### **③ 中间的紫色竖线：T1**

论文中 T1 表示：

> **从接收到 RS 到开始 retransmission（重传）之间的时间间隔。**

这是 sender 收到 RS 后，在协议栈内部处理 RS、准备重传所需的延时。

它很短，但依然是实际 LTP 行为的一部分。

---

### **④ 右上位置：Retransmission starts（RR ×）**

这里出现红色 “RR ×”，表示：

> 发送端开始正式发送 **需要重传的数据 segments**。

这些 segments 是 RS 中标记为“丢失”的部分。

由于链路中断导致部分 segment 丢失，所以必须重发。

这一部分与 IV-A 的 Case 1-A 重传机制一致，即：

* RS 报告缺失段
* 发送端依照 RS 重发所有丢失 segments
* 最后一个重传段带有新的 CP 标记

---

### **⑤ 右上角的黑色 ×：RA in response to RS**

这表示：

> 接收端在收到 RS acknowledgment（RA）时发送回应。

RA 是用来确认 RS 已收到的。
它是最终完成“本轮重传流程”的确认信号。

---

#  **Fig. 9 整体含义总结**

Fig. 9 清晰展示了链路中断发生后，LTP block 的恢复流程：

1. 原 CP 因链路中断丢失
2. RTO 触发 CP 的重传（图左）
3. 接收端收到 CP → 发 RS（图中）
4. 发送端处理 RS → T1 延时
5. 重传之前中断期间丢失的 segments（图右）
6. 最后发送 RA → 完成本次恢复流程（图右上）

---

#  **二、演讲稿


现在我来解释 Fig. 9。这张图展示的是一个 LTP block 在链路中断之后，是如何一步一步恢复传输的完整细节。它对应的是我们前面讲过的 Section IV-A 中的机制，只不过这里做了更细致的时间轴展开。

图的左上角我们可以看到一个红色的 “R” 标记，那是最后一次 CP 的重传。由于链路中断，原来的 CP 没有送到接收端，所以发送端必须经过一个 RTO 周期之后重新传一次 CP。这就是为什么我们看到 CP 重传出现在中断恢复后的第一个时间点。

CP 成功送达之后，接收端会立即回复一条 RS 报告，这个 RS 会告诉发送端，哪些 segment 已经收到了，哪些是在中断期间丢失的。图中在大约 81.5 到 81.6 秒的位置就出现了这一条 RS acknowledgment，这是恢复流程中的关键步骤。

接下来，中间的紫色竖线标记了一个时间间隔 T1，这是发送端接收到 RS 后进行处理的内部延迟，也就是说它需要一点时间来准备要重传的 segment。

等到准备完成之后，右边我们就可以看到重传正式开始的标志，也就是图中的 “RR”。这一组红色的标记表示发送端开始重发中断期间丢失的数据段。最后，当接收端成功收到这些重传的数据后，会再发一条 RA，确认 RS 的传递也成功结束。

所以整张图告诉我们一个实际的重要事实：
链路中断并不会让一个 block 完全失败，但会触发一整套额外的重传流程，包括 CP 重传、RS 反馈、再到数据段的补发。所有这些额外动作都会显著拉长整个 block 的传输时间。

Fig. 11 是论文的重要图，展示了链路中断对 **goodput（有效吞吐率）** 的影响，它结合了 Fig. 10 的结论，但展示的是另一个角度。

## **一、Fig. 11 分析**

### 图 11 展示什么？

它比较了两种实验设置（Set 1 和 Set 2）下：

* **解析模型预测的 goodput**（蓝线 / 红线）
* **实验真实测量的 goodput**（蓝圈 / 红方块）

横轴是链路中断持续时间
纵轴是 goodput（bytes/s）

**最终目的是：展示中断越长，goodput 越低，并呈现同样的“阶梯下降”规律。**

---

### Goodput 为什么呈“阶梯下降”？

这与 Fig. 10 的“阶梯增长”现象一致，只不过：

* Fig. 10 显示：中断越长，额外延迟呈**阶梯上升**
* Fig. 11 显示：goodput 越低，是**阶梯下降**

本质原因一样：

> **链路中断导致的延迟不是线性增加，而是随着 CP 重传的次数阶梯上升。
> 延迟上升是阶梯状 → goodput 下降也是阶梯状。**

Goodput = 文件大小 / 文件传输时间
当时间以阶梯式增加，goodput 就会以阶梯式下降。

---

### 为什么有两个数据集（Set 1 和 Set 2）？

Set 1 / Set 2 的区别在链路配置不同：

* Set 1：传输条件稍差，或 block 较大、传播时延较大
* Set 2：条件较好（论文原文中 Set 2 的 goodput 明显更高）

你可以看到：

* **红线（Set 2）整体高于蓝线（Set 1）**
  → 表明 Set 2 拥有更好的链路性能
* 但两者的趋势完全一致
  → 随着中断持续时间增加，goodput 逐级下降
  → 每下降一级，都对应一次 CP 超时重传

---

### 模型 vs 实验：高度吻合

无论 Set 1 还是 Set 2：

* 解析模型（线条）
* 实验数据（圈/方块）

都非常接近，误差极小。

说明：

> **模型不仅能预测时间延迟，也能非常准确地预测吞吐率变化。**

---

#  图 11 的核心意义

1. Goodput 随中断时间 **阶梯式下降**
2. 每一个台阶都对应一次 **CP 超时重传（RTO）**
3. 实验验证了解析公式（28）–（37）整体的正确性
4. 模型适用于不同实验条件（Set 1、Set 2 都匹配）
5. 说明 LTP 的性能瓶颈在于“CP 是否在中断中被丢弃”

## **二、口语化演讲稿**

---

现在我们来看 Fig. 11。这张图展示的是链路中断对 goodput，也就是有效吞吐率的影响。横轴表示中断持续时间，纵轴表示 goodput。图里包括两种设置：蓝色代表 Set 1，红色代表 Set 2，而且每一种设置都有模型预测值和实验测量值。

你会注意到，goodput 的下降曲线并不是一条平滑的斜线，而是呈现一种非常典型的阶梯状下降模式。这一点和我们在 Fig. 10 中看到的“时间开销阶梯增长”是完全一致的。原因在于，只要链路中断覆盖了 CP 的位置，发送端就必须进入一次完整的 RTO 超时重传，而每一次 CP 的超时都会显著拉长整个 block 的传输时间。因为 goodput 等于文件大小除以总传输时间，所以当时间呈现“台阶式增加”的时候，goodput 就会呈现“台阶式下降”。

另外你还会发现，无论是 Set 1 还是 Set 2，模型预测的曲线和实验测量的数据点几乎是重合的。这说明我们在 IV-B 中建立的解析模型不仅能够准确预测延迟，还能够精确捕捉吞吐率随中断变化的趋势。而不同的设置之间，Set 2 整体的 goodput 会更高一点，但两者的下降模式完全一致，说明这个模型在不同链路配置下都成立。

因此，Fig. 11 进一步验证了一个非常重要的事实：链路中断对 LTP 的性能影响不是线性的，而是由 CP 的重传机制主导的阶梯式变化；而我们的理论模型能够非常准确地预测这种行为。

---

## **一、Fig.12 分析**

### 图 12 表达的核心问题是什么？

图 12 展示了：

> **在不同的信道误码率（BER）下，链路中断持续时间如何影响整个文件的传输时间。**

这里使用了 Set 1 的实验环境，BER 有三种：

* **10⁻⁶（蓝色）**
* **5×10⁻⁶（红色）**
* **10⁻⁵（紫色）**

每一种 BER 都包括：

* 模型预测（实线）
* 实验测量（点/方块）

---

#  观察现象 1：所有曲线都是“阶梯状”增长

和 Fig.10 的结论一致，文件传输时间随着中断持续时间增加不是直线，而是一级一级往上“跳”。

原因完全一样：

> **每一个台阶都是一次 CP 超时重传。**

当链路中断覆盖到 CP 时：

* CP 丢失
* 接收端不发 RS
* 发送端等待一个完整 RTO
* 文件传输时间突然增加一个 RTO 的量

这就是阶梯的来源。

---

#  观察现象 2：BER 越高，曲线整体越“抬高”

三条曲线的垂直位置关系是：

* 蓝色（BER = 10⁻⁶）最低
* 红色（BER = 5×10⁻⁶）中间
* 紫色（BER = 10⁻⁵）最高

这完全符合 IV-B 的解析模型：

> **BER 越大 → segment 丢失概率越大 → 重传次数越多 → 文件传输时间自然变长。**

特别是：

* RDS 丢失导致更多补发（影响 T_seg_total）
* CP 丢失导致更多 RTO（影响 T_CP_wait）
* RS 丢失导致额外 RTO（影响 T_RS_wait）

由于 CP/RS 在误码时的代价巨大，因此 BER 一旦升高，纵向差距明显拉大。

---

#  观察现象 3：BER 越高，阶梯的“高度”也越明显

原因：

* 在较高 BER 下，**even without disruption** 就可能发生 CP 的误码 → RTO
* 当链路中断和误码 *叠加* 时，会形成 **更频繁、更高的阶梯**

也就是说：

> **链路中断 + 误码不是简单相加，而是相互放大。**

这正是 IV-B 建模的动机：
链路中断与误码是 “joint effect”，必须一起分析。

---

#  观察现象 4：模型与实验高度吻合

你可以看到：

* 蓝圈 vs 蓝线 → 几乎重叠
* 黑星 vs 红线 → 接近
* 黑方块 vs 紫线 → 高度一致

这说明：

> **解析模型（公式 28–37）成功捕捉了 LTP 的真实行为。**

特别是：

* CP RTO 的模型吻合度非常高
* 随中断时长的阶梯行为精确预测
* BER 的影响趋势与实验完全一致

---

#  图 12 的核心意义（论文级别总结）

1. **证明模型不仅解释链路中断，也能解释误码 BER 的影响**
2. **链路中断造成的是“阶梯式”时间损失，随着 BER增加被进一步放大**
3. **误码越高 → CP 重传次数越多 → 阶梯越高 → 文件传输时间显著增加**
4. **实验与模型高度吻合 → 模型可用于真实 ISS 场景性能预测**

#  **二、演讲稿**

下面是一段适合在课堂或答辩中“一字不改直接朗读”的稿子：

---

现在我来讲 Fig.12。这张图是本篇论文最关键的实验结果之一，因为它把链路中断和信道误码的影响同时放在一张图里面，让我们看到它们如何共同决定整个文件的传输时间。

横轴是链路中断的持续时间，纵轴是文件完整传输所需要的时间。图中有三条颜色不同的曲线，分别对应三种不同的信道误码率：10⁻⁶、5×10⁻⁶ 和 10⁻⁵，每一条曲线都包括模型预测和实际实验的数据。

这里一个非常重要的现象是：所有的曲线都是阶梯状的，而不是连续上升的。这和我们在 Fig.10 中看到的现象完全一致，就是说链路中断的影响并不是线性增加，而是当中断覆盖了 CP 的位置时，会触发一次完整的 RTO 超时重传，从而让传输时间突然跳高一截。所以每一个台阶，其实对应的都是“CP 又被迫重传了一次”。

同时你会看到，随着误码率的提高，这三条曲线整体都往上抬了一段距离。例如，紫色曲线，误码率最高，它的整个传输时间明显比其他颜色都要更长。这很好理解，因为误码越高，segment 丢失越多，重传次数也越多，尤其是 CP 和 RS 被误码破坏时，会引起 RTO，导致传输时间成倍增加。

更关键的是，无论是误码为 10⁻⁶ 还是 10⁻⁵，模型预测的曲线和实验的数据点几乎完全吻合。这进一步说明，我们在 IV-B 中推导的公式体系非常准确地捕捉到了 LTP 在真实网络条件下的表现，无论是链路中断、误码导致的 segment 重传、还是 CP 的超时行为，模型都可以量化预测。

因此，Fig.12 清晰展示了一个结论：链路中断和信道误码并不是独立作用，而是相互放大的。其中，CP 的丢失是造成阶梯上升的根本原因，而误码率越高，这种阶梯越高、越频繁，从而导致整个文件传输时间显著增加。
