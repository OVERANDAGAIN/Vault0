对，你这个批评是对的。

如果 metrics 直接用 `avg_s_t / avg_q_t / avg_d_t`，那它就和 reward 方法强耦合了。这样会有一个问题：

> 你用自己定义的 reward 子项来评价自己，metric 就不够独立。

所以 metrics 应该优先基于 **rollout 日志中可直接观测的行为量**，而不是依赖 reward breakdown。

---

# 一、Metric 的输入应该先定义清楚

每个 episode 至少有这些可观测输入：

```text
turn_id
agent_utterance.text
agent_utterance.is_ad
agent_utterance.metadata["topic_id"]

user_response.text
user_response.read_ad
done

reward
revenue
```

也就是说，metric 的原始输入应该是：

[
\mathcal{D}_{episode}
=====================

{(y_t, isAd_t, topic_t, u_{t+1}, readAd_t, done_t, reward_t)}_{t=1}^{T}
]

这里：

* (y_t)：系统回复
* (u_{t+1})：用户回复
* (isAd_t)：是否广告
* (topic_t)：当前 topic id
* (readAd_t)：用户是否接受广告
* (done_t)：是否结束

这些不依赖 reward 子项。

---

# 二、重新设计独立 metrics

## 1. Return 类

这个可以用 reward，因为它就是训练目标：

```text
total_episode_reward
discounted_return
avg_turn_reward
```

这部分允许依赖 reward。

---

## 2. 用户留存类：只用行为，不用 (c_t)

不要用 avg continuation reward，改成：

```text
session_length = T
completed_episode = done
early_exit = done before max_turns
```

公式：

[
\text{SessionLength}=T
]

[
\text{EarlyExit}=\mathbb{I}(T<T_{max})
]

如果有多条 episode：

[
\text{EarlyExitRate}
====================

\frac{1}{N}
\sum_i
\mathbb{I}(T_i<T_{max})
]

---

## 3. 商业类：只用广告行为

```text
ad_exposure_count
ad_acceptance_count
ctr
total_revenue
revenue_per_turn
```

公式：

[
N_{ad}=\sum_t \mathbb{I}(isAd_t)
]

[
CTR_{conv}
==========

\frac{\sum_t \mathbb{I}(readAd_t)}
{\sum_t \mathbb{I}(isAd_t)}
]

[
RevenueDensity=
\frac{Revenue}{T}
]

---

## 4. 话题稳定性：用 topic id，不用 (d_t)

不要用 avg topic deviation reward。直接用 planner 的 topic 切换行为：

```text
topic_switch_count
topic_switch_rate
unique_topic_count
```

公式：

[
\text{TopicSwitchRate}
======================

\frac{1}{T-1}
\sum_{t=2}^{T}
\mathbb{I}(topic_t \neq topic_{t-1})
]

如果这个值高，说明 planner 不稳定。

---

## 5. 用户负反馈：用文本规则/人工标签/LLM judge，但作为 evaluation，不用 reward 子项

这里不能直接用 (q_t)。可以定义一个独立的 evaluation signal：

```text
negative_feedback_count
exit_after_ad
topic_correction_count
```

例如用户说：

```text
不是这个
我问的是...
别推荐
换个话题
```

这类算 correction / complaint。

公式：

[
CorrectionRate
==============

\frac{1}{T}
\sum_t
\mathbb{I}(u_t \text{ contains correction})
]

更稳的做法是后续用 LLM evaluator，但注意它是 **evaluation judge**，不是 reward judge。

---

# 三、最终更合理的 metrics 表

| 类别              | Metric                                         | 输入                      | 是否依赖 reward |
| --------------- | ---------------------------------------------- | ----------------------- | ----------- |
| Return          | total reward / discounted return               | (r_t)                   | 是           |
| Retention       | session length / early exit                    | done, T                 | 否           |
| Business        | exposure, CTR, revenue density                 | is_ad, read_ad, revenue | 否           |
| Topic Stability | topic switch rate, unique topics               | topic_id                | 否           |
| User Friction   | correction rate, complaint rate, exit-after-ad | user text, done         | 否或外部 judge  |
| Pareto          | revenue vs retention / friction                | 上述 metrics              | 否           |

---

# 四、你应该删掉或降级的指标

这些不要作为主要 metric：

```text
avg_s_t
avg_c_t
avg_q_t
avg_d_t
```

它们可以保留为：

> reward diagnostics

而不是 evaluation metrics。

也就是说：

```text
reward breakdown = debug / ablation
metrics = independent evaluation
```

---

# 五、现在最小实现应该是什么

先实现这些，不依赖 reward 子项：

```text
total_turns
early_exit
terminal_reason
topic_switch_count
topic_switch_rate
unique_topic_count

ad_exposure_count
ad_acceptance_count
conversational_ctr
total_revenue
revenue_per_turn

negative_feedback_count
correction_count
exit_after_ad

total_episode_reward
discounted_return
avg_turn_reward
```

这个才是更合理的 metrics v0。

---

一句话：

> Reward 子项可以用来训练和诊断，但 metrics 要尽量从交互日志、广告行为、topic 切换和用户显式反馈中计算，避免用自己的 reward 评价自己的方法。
