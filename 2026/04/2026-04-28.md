先别继续深挖 reward。现在正确顺序是：

> **先把 mock user、metrics、pipeline 跑通，再回来升级 reward。**

原因很简单：==更成熟的 reward 需要数据来源。现在没有稳定 mock user、没有 rollout 数据、没有 metrics，继续设计 reward 很容易变成纸面优化。==

你明天汇报可以这样定位：

> 当前 reward 已完成 v0 接入，重点是模块化接口和六项结构落地；但其中语义类 reward 仍是 proxy/fallback，后续需要用模型化 estimator 升级。

---

更成熟 reward 后续可以分三步做。

第一步，保留当前六项结构，但把 estimator 分层：

| reward 项              | 当前 v0                  | 后续升级                                |
| --------------------- | ---------------------- | ----------------------------------- |
| (s_t) satisfaction    | simulator satisfaction | LLM judge / BERT 回归                 |
| (c_t) continuation    | 是否退出                   | continuation predictor              |
| (a_t) ad acceptance   | read_ad / exposure     | 用户兴趣分类器、conversion intent           |
| (q_t) annoyance       | metadata               | 情绪/反感分类器                            |
| (o_t) ad overload     | 规则                     | 保持规则                                |
| (d_t) topic deviation | 占位                     | embedding cosine / topic classifier |

第二步，用 LLM judge 先生成伪标签。
不用马上训练 BERT。先让 LLM 对每轮输出：

```json
{
  "task_satisfaction": 0.7,
  "ad_acceptance": 0.2,
  "annoyance": 0.0,
  "topic_deviation": 0.1,
  "explanation": "..."
}
```

这能快速把 reward 从规则变成语义估计。

第三步，后续再蒸馏成小模型。
也就是：

```text
LLM judge 标注 rollout 数据
        ↓
训练 BERT/DeBERTa reward estimator
        ↓
替换当前 estimator
```

---

明天不要说“reward 已经成熟”。说：

> “Reward v0 已经完成接入，当前核心贡献是结构化设计和可替换 estimator 接口。下一步会把语义类子项从 fallback estimator 升级为 LLM-judge 或 BERT-based reward model。”

你现在下午到明晚的优先级应该是：

1. **让 reward breakdown 在 rollout 里正常输出**
2. **实现 2–3 个 mock user baseline**
3. **补 metrics report**
4. **如果还有时间，再做一个 LLMJudgeEstimator 的接口占位，不一定真的大规模使用**

也就是说，reward 暂时做到“可运行、可解释、可替换”就够了。不要试图今天把它做成研究最终版。
