对，你这个纠正是对的。**Metrics 不能依赖我们自定义的 reward，也不能依赖 reward breakdown。**
如果 metrics 里还放 `total_reward / discounted_return / avg_s_t / avg_q_t`，那其实是在用方法内部定义去评价方法本身，不够独立。

所以这里要重新定义：

> **Evaluation metrics 的输入只来自 episode log、agent action、广告事件、用户行为事件和退出信息。**

也就是说，metrics 不是“reward 统计”，而是“交互结果统计”。

---

# 一、Metrics 的原始输入

一个 episode 的日志可以抽象为：

$$\mathcal{E} = \{(y_t, u_t, ad_t, adID_t, click_t, done_t)\}_{t=1}^{T}$$

其中：

| 变量         | 含义          | 来源                        |
| ---------- | ----------- | ------------------------- |
| $T$        | 对话轮数        | episode log               |
| $y_t$      | 第 $t$ 轮系统回复 | agent log                 |
| $u_t$      | 第 $t$ 轮用户回复 | simulator / real user log |
| $ad_t$     | 是否插广告       | agent action              |
| $adID_t$   | 广告 ID       | agent metadata            |
| $click_t$  | 用户是否点击/接受广告 | simulator / user event    |
| $done_t$   | 是否退出        | environment               |
| $t_{exit}$ | 退出轮数        | episode log               |

这个定义是干净的，不依赖 reward。

# 二、重新设计独立 Metrics

## 1. 对话留存类：Conversation Retention Metrics

这一类只回答：

> 用户聊了多久？是否提前退出？

### 1.1 Session Length

$$\text{SessionLength} = T$$

含义：一个 episode 中实际发生的对话轮数。

如果设最大轮数 $T_{\max}$，可以定义归一化留存：

$$\text{NormalizedRetention} = \frac{T}{T_{\max}}$$

---

### 1.2 Early Exit

$$\text{EarlyExit} = \mathbb{I}(T < T_{\max})$$

多 episode 下：

$$\text{EarlyExitRate} = \frac{1}{N} \sum_{i=1}^{N} \mathbb{I}(T_i < T_{\max})$$

这里不判断退出原因，只看是否提前结束。

---

### 1.3 Exit Turn

$$t_{exit} = T$$

用于分析用户是在前期、中期还是后期退出。

---

## 2. 广告曝光与转化类：Ad Delivery / Conversion Metrics

这一类完全来自广告事件，不依赖 reward。

### 2.1 Ad Exposure Count

$$N_{ad} = \sum_{t=1}^{T} \mathbb{I}(ad_t = 1)$$

---

### 2.2 Ad Exposure Rate / Ad Load

$$\text{AdLoad} = \frac{N_{ad}}{T}$$

这就是对话中的广告密度。

---

### 2.3 Ad Click / Acceptance Count

$$N_{click} = \sum_{t=1}^{T} \mathbb{I}(click_t = 1)$$

这里 `click` 可以是真实点击，也可以是 simulator 的 `read_ad` / accept event。

---

### 2.4 Conversational CTR

$$\text{CTR}_{conv} = \frac{N_{click}}{N_{ad}}$$

如果 $N_{ad}=0$，CTR 记为 0 或 undefined。汇报中建议显示为 `N/A` 更严谨。

---

### 2.5 Revenue

如果每个广告有价值 $v(adID_t)$，则：

$$\text{Revenue} = \sum_{t=1}^{T} \mathbb{I}(click_t=1) \cdot v(adID_t)$$

---

### 2.6 Revenue Density

$$\text{RevenueDensity} = \frac{\text{Revenue}}{T}$$

这个可以对应你说的“单位轮次转化/单位轮次收益”。

---

## 3. 广告压力类：Ad Pressure Metrics

这一类衡量广告是否插得太密。

### 3.1 Consecutive Ad Length

$$\text{MaxConsecutiveAds} = \max \text{连续 } ad_t=1 \text{ 的长度}$$

---

### 3.2 Windowed Ad Load

给定窗口大小 $k$：

$$\text{AdLoad}_k(t) = \frac{1}{k} \sum_{i=t-k+1}^{t} \mathbb{I}(ad_i=1)$$

---

### 3.3 Frequency Cap Violation Rate

如果设定阈值 $\rho$，比如最近 5 轮广告占比不能超过 0.4：

$$\text{FreqCapViolationRate} = \frac{1}{T} \sum_{t=1}^{T} \mathbb{I}(\text{AdLoad}_k(t) > \rho)$$

这个是广告行业里比较自然的 frequency capping 思路。

---

## 4. 话题轨迹稳定性：Topic Trajectory Metrics

这类只依赖 agent metadata 里的 topic id，不用语义 reward。

### 4.1 Topic Switch Count

$$N_{switch} = \sum_{t=2}^{T} \mathbb{I}(topic_t \neq topic_{t-1})$$

---

### 4.2 Topic Switch Rate

$$\text{TopicSwitchRate} = \frac{N_{switch}}{T-1}$$

过高说明 planner 不稳定，容易 topic oscillation。

---

### 4.3 Unique Topic Count

$$N_{topic} = |\{topic_t\}_{t=1}^{T}|$$

过高说明对话过于发散。

---

### 4.4 Average Segment Length

如果 topic id 连续相同视为一个 segment：

$$\text{AvgSegmentLength} = \frac{T}{N_{segment}}$$

其中 $N_{segment}=N_{switch}+1$。

这个指标可以直接衡量 segment 是否过短、频繁跳转。

---

## 5. 用户显式摩擦：User Friction Metrics

这类不能用 reward 的 $q_t$，但可以基于用户文本里的**显式行为模式**。第一版可以规则，后续可以人工/外部 judge。

### 5.1 Correction Count

用户把话题拉回原目标，例如：

* “我问的是……”
* “不是这个”
* “回到……”
* “I was asking about...”
* “Actually, I want...”

定义：

$$N_{corr} = \sum_{t=1}^{T} \mathbb{I}(u_t \text{ contains correction})$$

---

### 5.2 Correction Rate

$$\text{CorrectionRate} = \frac{N_{corr}}{T}$$

这在你当前系统很重要，因为用户经常把 Python 话题拉回来。

---

### 5.3 Complaint Count

用户明确表达不满：

* “不要广告”
* “别推销”
* “无关”
* “annoying”
* “not related”

$$N_{complaint} = \sum_{t=1}^{T} \mathbb{I}(u_t \text{ contains complaint})$$

---

### 5.4 Exit After Ad

$$\text{ExitAfterAd} = \mathbb{I}(done_T=1 \land ad_{T}=1)$$

或更宽松：

$$\text{ExitWithinKAfterAd} = \mathbb{I}(\exists t, ad_t=1 \land T-t \le k)$$

用于衡量广告是否诱发退出。

# 三、最终推荐 Metrics 集合

现在建议先固定这套，不要再引入 reward 子项。

| 类别          | Metric                            | 依赖输入              |
| ----------- | --------------------------------- | ----------------- |
| Retention   | Session Length (T)                | episode log       |
| Retention   | Normalized Retention (T/T_{\max}) | episode log       |
| Retention   | Early Exit                        | done, $T_{\max}$  |
| Ads         | Ad Exposure Count                 | $ad_t$            |
| Ads         | Ad Load                           | $ad_t, T$         |
| Ads         | Conversational CTR                | $click_t, ad_t$   |
| Ads         | Revenue                           | $click_t, adID_t$ |
| Ads         | Revenue Density                   | revenue, $T$      |
| Ad Pressure | Max Consecutive Ads               | $ad_t$            |
| Ad Pressure | Frequency Cap Violation Rate      | $ad_t$            |
| Topic       | Topic Switch Rate                 | topic id          |
| Topic       | Unique Topic Count                | topic id          |
| Topic       | Avg Segment Length                | topic id          |
| Friction    | Correction Rate                   | $u_t$             |
| Friction    | Complaint Count                   | $u_t$             |
| Exit        | Exit Turn                         | done              |
| Exit        | Exit After Ad                     | done, $ad_t$      |

# 四、Pareto 怎么做

Pareto 也不要用 reward。
用独立指标组合：

$$\mathbf{M}(\pi) = \big[
\text{RevenueDensity},
\text{NormalizedRetention},
-\text{CorrectionRate},
-\text{ComplaintRate},
-\text{TopicSwitchRate}
\big]$$

或者简化为二维：

### Revenue–Retention

$$x = \text{NormalizedRetention}, \quad y = \text{RevenueDensity}$$

### Revenue–Friction

$$x = \text{CorrectionRate} + \text{ComplaintRate}, \quad y = \text{RevenueDensity}$$

### Revenue–Topic Stability

$$x = \text{TopicSwitchRate}, \quad y = \text{RevenueDensity}$$

理想策略不是 reward 高，而是：

> revenue 高，同时 retention 高、friction 低、topic switch 低。

# 五、你这段可以写进文档的核心表述

> Metrics should be computed from observable interaction logs rather than reward components.
> Given an episode log consisting of system responses, user responses, ad exposure events, ad ids, click/acceptance events, topic ids, and termination signals, we evaluate a policy from four independent perspectives: retention, monetization, ad pressure, and topic/friction stability.
> Reward is used for training or internal optimization, while metrics are used for method-independent evaluation.

中文就是：

> 指标体系应从可观测交互日志中计算，而不是依赖 reward 子项。Reward 用于训练或内部优化，metrics 用于独立评价策略效果。

这才是严谨的。
