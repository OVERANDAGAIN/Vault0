

**Metrics 不能依赖我们自定义的 reward
如果 metrics 里还放 `total_reward / discounted_return / avg_s_t / avg_q_t`，那其实是在用方法内部定义去评价方法本身，不够独立。


>  metrics 的输入只来自 episode log、广告事件、用户行为事件和退出信息


# 一、Metrics 的原始输入

一个 episode 的日志可以抽象为：

$$\mathcal{E} = \{(y_t, u_t, ad_t, adID_t, click_t, done_t)\}_{t=1}^{T}$$
其中：

| 变量         | 含义          | 来源                        |
| ---------- | ----------- | ------------------------- |
| $T$        | 对话轮数        | episode log               |
| $y_t$      | 第 $t$ 轮系统回复 | agent log                 |
| $u_t$      | 第 $t$ 轮用户回复 | simulator / real user log |
| $ad_t$     | 是否插广告       | agent action              |
| $adID_t$   | 广告 ID       | agent metadata            |
| $click_t$  | 用户是否点击/接受广告 | simulator / user event    |
| $done_t$   | 是否退出        | environment               |
| $t_{exit}$ | 退出轮数        | episode log               |

这个定义是干净的，不依赖 reward。

# 二、设计独立 Metrics

## 1. Rule-based metrics

第一类是放能直接从 log 里算出来的。

这些指标不需要模型判断，也不依赖 reward。


### 1. 1对话留存类：Conversation Retention 

> 用户聊了多久？是否提前退出？

#### 1.1.1 Session Length

$$\text{SessionLength} = T$$

含义：一个 episode 中实际发生的对话轮数。

如果设最大轮数 $T_{\max}$，可以定义归一化留存：

$$\text{NormalizedRetention} = \frac{T}{T_{\max}}$$

---

#### 1.1.2 Early Exit

$$\text{EarlyExit} = \mathbb{I}(T < T_{\max})$$

多 episode 下：

$$\text{EarlyExitRate} = \frac{1}{N} \sum_{i=1}^{N} \mathbb{I}(T_i < T_{\max})$$

看是否提前结束。

---

#### 1.1.3 Exit Turn

$$t_{exit} = T$$

用于分析用户是在前期、中期还是后期退出。

---

### 1.2. 广告曝光与转化类：Ad Delivery / Conversion Metrics


#### 1.2.1 Revenue

每个广告有价值 $v(adID_t)$，则：

$$\text{Revenue} = \sum_{t=1}^{T} \mathbb{I}(click_t=1) \cdot v(adID_t)$$

---

#### 1.2.2 Revenue Density

$$\text{RevenueDensity} = \frac{\text{Revenue}}{T}$$

“单位轮次转化/单位轮次收益”。

---

### 1.3. 话题轨迹稳定性：Topic Trajectory Metrics

这类只依赖 agent metadata 里的 topic id，不用语义

#### 1.3.1 Topic Switch Count

$$N_{switch} = \sum_{t=2}^{T} \mathbb{I}(topic_t \neq topic_{t-1})$$

---

#### 1.3.2 Topic Switch Rate

$$\text{TopicSwitchRate} = \frac{N_{switch}}{T-1}$$

过高说明 planner 不稳定，容易 topic oscillation。

## 2. Behavioral metrics

第二层看用户行为结果。

比如：

* click / accept
* continue / stop
* correction
* complaint
* repeated clarification
* topic pull-back

这一层比 rule-based 更接近用户真实反馈，但也不完全等于“用户满意”。
例如用户继续聊，可能是真的感兴趣，也可能只是还没完成任务；用户没点击广告，也不一定是不接受。

所以它不能直接当 ground truth， LLM judge 接近真实交互结果。

---

### 2.1. Judge-based metrics

第三层才用 judge。

原因是有些东西确实没法只靠 log 算，比如：

* 这条广告是否自然
* 是否破坏原任务
* 是否和当前 segment 匹配
* 回复有没有推进用户目标
* topic shift 是否合理

这些维度需要语义判断，所以可以用 LLM judge / classifier / embedding。

但这里要注意：judge-based metrics 只能作为 diagnostic，不应该成为最核心的 primary metrics。
否则 metrics 和 reward 又会重叠。

### 2.2. 用户显式摩擦：User Friction Metrics

这类不能用 reward 的 $q_t$，但可以基于用户文本里的**显式行为模式**。第一版可以规则，后续可以人工/外部 judge。

#### 2.2.1 Correction Count

用户把话题拉回原目标，例如：

* “我问的是……”
* “不是这个”
* “回到……”
* “I was asking about...”
* “Actually, I want...”

定义：

$$N_{corr} = \sum_{t=1}^{T} \mathbb{I}(u_t \text{ contains correction})$$
---

#### 2.2.2 Complaint Count

用户明确表达不满：

* “不要广告”
* “别推销”
* “无关”
* “annoying”
* “not related”

$$N_{complaint} = \sum_{t=1}^{T} \mathbb{I}(u_t \text{ contains complaint})$$
