## “Planner 优化方案与 baseline 设计”

主要讲三点就够：

### 1. 为什么现在要看 Planner

现在 reward 和 mock user 已经能形成交互闭环，所以接下来真正需要优化的是 planner：它决定什么时候继续话题、什么时候切换话题、什么时候插广告。也就是说，planner 是后续 policy learning 的核心。

### 2. Planner 可能的几类方案

几个候选方向：

```text
Rule-based planner baseline
```

先用规则做一个可解释 baseline，比如根据用户满意度、广告距离、topic depth 决定是否插广告。

```text
Contextual bandit
```

如果先不考虑长期多步影响，可以把每一轮看成一次 action selection，在 topic / ad candidates 中选择当前最优动作。

```text
RL / policy gradient
```

如果考虑长期 reward，就把整个 episode 看成 trajectory，用累计 reward 训练 planner。

```text
LLM-as-planner with reward-guided selection
```

保留现在 LLM planner，但在 prompt 或 candidate scoring 里加入 reward 预测，让 planner 显式考虑 user value 和 business value。

# 1. CRS 中的 RL Policy Learning

这一组最基础，说明 **Conversational Recommender System 本来就可以被建模成 RL policy learning 问题**。

| 文献                                                                                                            |      年份 / 类型 | 大致内容                                                                                                                   | 和 REC-DIAL Planner 的关系                                                                                   |
| ------------------------------------------------------------------------------------------------------------- | -----------: | ---------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **Conversational Recommender System**                                                                         | 2018 / arXiv | 把对话系统和推荐系统整合进 deep RL 框架，目标是构建 personalized conversational recommendation agent，并优化 session-level utility。([arXiv][1]) | 可作为“多轮推荐 = sequential decision problem”的基础引用。你们可以类比它的 ask/recommend policy，只是你们多了广告动作。                   |
| **Estimation–Action–Reflection: Towards Deep Interaction Between Conversational and Recommender Systems**     |  2020 / WSDM | EAR 把 CRS 拆成 Estimation、Action、Reflection 三阶段，重点解决问什么、何时推荐、如何根据用户反馈更新的问题。([Penn State][2])                             | 很适合支撑“Planner 不只是生成下一句，而是要根据用户反馈更新状态并做 action decision”。                                                 |
| ==**UNICORN: Unified Conversational Recommendation Policy Learning via Graph-based Reinforcement Learning**== | 2021 / SIGIR | 统一建模 CRS 中三个决策：何时问/推荐、问什么属性、推荐什么 item，并用 graph-based RL 学习统一策略。([arXiv][3])                                            | 和你们最像：UNICORN 是 when/what/which，你们是 continue/switch/insert-ad/choose-ad。是 Planner policy learning 的核心参考。 |

**这组文献可以支撑的论点：**

> REC-DIAL 的 Planner 可以被定义为一个 high-level conversational recommendation policy。它不是普通 response generator，而是根据用户状态、对话历史和候选内容选择下一步系统行为。

# 2. 广告插入 / 推荐广告联合优化

这一组对你们最有特色的地方最重要，因为你们不是普通 CRS，而是 **广告支持型多轮对话推荐系统**。

| 文献                                                                                                 |     年份 / 类型 | 大致内容                                                                                                            | 和 REC-DIAL Planner 的关系                                                                                       |
| -------------------------------------------------------------------------------------------------- | ----------: | --------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| ==**RAM: Jointly Learning to Recommend and Advertise**==                                           |  2020 / KDD | 提出 two-level RL framework：第一层生成推荐列表以优化长期用户体验，第二层插入广告，平衡即时广告收益和广告对长期用户体验的负面影响；广告层决定是否插广告、插哪个广告、插在哪里。([arXiv][4]) | 和你们高度相关。它的 “whether / which / where to insert ad” 可以对应你们的 “是否插广告 / 插哪个广告 / 在哪轮或哪个 topic context 插入”。         |
| ==**DEAR: Deep Reinforcement Learning for Online Advertising Impression in Recommender Systems**== | 2021 / AAAI | 用 DQN 解决推荐系统中的广告插入问题，同时决定是否插入广告、插哪个广告、插入位置，并优化长期收益。([arXiv][5])                                                 | 可以作为你们广告 Planner 的直接类比。DEAR 的 list position 在你们这里可以改成 dialogue turn / topic transition point。                |
| **Ad-load Balancing via Off-policy Learning in a Content Marketplace**                             | 2024 / WSDM | 研究广告负载平衡，用 off-policy contextual bandit 从 logged bandit feedback 学习策略，目标是同时提升用户满意度相关指标和广告收益。([arXiv][6])        | 支撑你们近期 baseline：先做 contextual bandit，再做完整 RL。它也支撑把 ad exposure、session position、user heterogeneity 放进 state。 |

**这组文献可以支撑的论点：**

> 广告推荐中的核心难点不是单纯最大化广告收益，而是同时考虑广告收益和用户体验。REC-DIAL 可以把传统推荐列表中的 ad insertion 问题迁移到多轮对话场景中。

# 3. Hierarchical RL / Planner-Speaker 分层

你们现在已经有：

```text
Planner：决定做什么
Speaker：决定怎么说
```

所以 hierarchical policy learning 这组文献非常适合支撑最终架构。

| 文献                                                                                                                      |      年份 / 类型 | 大致内容                                                                                                                               | 和 REC-DIAL Planner 的关系                                                                            |
| ----------------------------------------------------------------------------------------------------------------------- | -----------: | ---------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| **Towards Hierarchical Policy Learning for Conversational Recommendation with Hypergraph-based Reinforcement Learning** | 2023 / IJCAI | 提出 Director-Actor hierarchical CRS：Director 决定 ask/recommend 这类 option，Actor 再选择具体 primitive action，例如问哪个属性或推荐哪个 item。([arXiv][7]) | 和你们的两级 Planner 很像：高层决定 continue/switch/insert_ad，低层选择 topic/ad candidate。                         |
| **Modelling Hierarchical Structure between Dialogue Policy and Natural Language Generator with Option Framework**       |  2021 / ICLR | 用 option framework 建模 dialogue policy 和 NLG 的层级关系，把 dialogue act 看成 option，把词生成看成 primitive action，并用 HRL 训练。([OpenReview][8])     | 直接对应你们的 Planner/Speaker 分离问题。可以支撑“Planner action 需要被 Speaker 严格实现，并引入 realization/compliance 约束”。 |
| **HiCR: Confident Action Decision via Hierarchical Policy Learning for Conversational Recommendation**                  |   2023 / WWW | 将多轮 CRS 的 action decision 建模为 hierarchical policy learning 问题，强调复杂动作空间下分层决策的必要性。([ACM Digital Library][9])                         | 可作为补充文献，支撑“复杂 action space 不适合直接 flat policy”。                                                    |

**这组文献可以支撑的论点：**

> REC-DIAL 的 action space 包括 dialogue act、topic choice、ad choice 和 language realization，因此用 hierarchical planner 比 flat planner 更合理。

# 4. Reward / Multi-objective Optimization

你们现在的 reward 是：

```text
user_value + business_value - penalties
```

所以 multi-objective reward、reward shaping、reward learning 都是后续要关注的方向。

| 文献                                                                                   |        年份 / 类型 | 大致内容                                                                                                                                                                 | 和 REC-DIAL Planner 的关系                                                                        |
| ------------------------------------------------------------------------------------ | -------------: | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| **Multi-Objective Intrinsic Reward Learning for Conversational Recommender Systems** | 2023 / NeurIPS | 指出 RL-based CRS 很依赖 handcrafted reward，而手工 reward 可能和用户真实意图不一致；提出从用户交互中学习 intrinsic reward，并把 reward learning 建成 multi-objective bi-level optimization。([arXiv][10]) | 很适合支撑你们 reward 后续升级：当前用 RewardCalculator，之后可以做 learned planner-level reward。                  |
| **Reward Shaping for User Satisfaction in a REINFORCE Recommender**                  |   2022 / arXiv | 关注如何通过 satisfaction-related signal 进行 reward shaping，使 RL recommender 不只优化点击，还优化用户满意度。                                                                               | 可作为补充方向。你们 mock user 已经有 satisfaction，因此可以先用 oracle satisfaction，后续训练 satisfaction predictor。 |
| **Multi-objective Recommender Systems Survey**                                       |  2023 / Survey | 总结推荐系统中的多目标优化问题，例如用户价值、平台价值、长期目标、短期目标、多 stakeholder 目标。                                                                                                              | 适合放在 introduction / problem motivation，支撑“用户价值和商业价值天然冲突”。                                     |

**这组文献可以支撑的论点：**

> REC-DIAL 的 reward 不应只优化广告收益，也不应只优化用户满意度，而应该显式建模 user value、business value 和 negative experience penalty 的多目标权衡。

# 5. LLM Planner / Goal-guided Planning

这组和你们现在的 LLMPlanner 最相关。它们说明：**可以不直接训练整个 LLM，而是在 LLM 外接一个可训练 Planner 或 goal planner**。

| 文献                                                                                                                 |               年份 / 类型 | 大致内容                                                                                                                                                                               | 和 REC-DIAL Planner 的关系                                                              |
| ------------------------------------------------------------------------------------------------------------------ | --------------------: | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| ==**PPDPP: Plug-and-Play Policy Planner for Large Language Model Powered Dialogue Agents**==                       |           2024 / ICLR | 提出一个可插拔 dialogue policy planner，用于增强 LLM-powered dialogue agents 的策略规划能力；训练方式包括 supervised fine-tuning、基于 goal-oriented AI feedback 的 RL，以及 LLM self-play simulation。([arXiv][11]) | 和你们非常相关。可以支撑“Speaker 可以继续用 LLM，Planner 作为可训练外接模块”。                                  |
| **ChatCRS: Incorporating External Knowledge and Goal Guidance for LLM-based Conversational Recommender Systems**   | 2025 / NAACL Findings | 指出 LLM 在 domain-specific CRS 中存在 grounding 和 goal guidance 不足，提出结合 external knowledge retrieval 和 goal-guided planning 的 CRS 框架。([ACL Anthology][12])                              | 支撑你们当前发现：原始 LLMPlanner 更偏用户帮助，商业策略和目标规划弱，因此需要显式 Planner。                            |
| **User Simulator-Guided Multi-Turn Preference Optimization for Reasoning LLM-based Conversational Recommendation** |          2026 / arXiv | 提出 multi-turn preference optimization 框架，包含 user simulator、retriever 和 reasoning LLM-based recommender。([arXiv][13])                                                               | 这是比较新的方向，和你们 mock user + multi-turn optimization 接近；适合作为未来 work，而不是当前 baseline 主文献。 |

**这组文献可以支撑的论点：**

> LLM 本身不一定擅长稳定执行长期策略目标，因此 REC-DIAL 可以采用 “trainable planner + LLM speaker” 的结构：Planner 优化策略，Speaker 负责语言实现。

> 我把相关文献分成几条线。第一类是 CRS 的 RL policy learning，例如 Sun & Zhang 2018、EAR 和 UNICORN，它们说明多轮推荐可以被建模为高层策略学习问题。第二类是推荐广告联合优化，例如 RAM 和 DEAR，它们和我们最接近，因为它们研究是否插广告、插哪个广告以及如何平衡广告收益和用户体验。第三类是 hierarchical policy learning，例如 DAHCR 和 HDNO，可以支撑我们把 Planner 和 Speaker 分开，形成高层 action 和低层语言实现。第四类是 reward 和 multi-objective optimization，例如 Multi-Objective Intrinsic Reward Learning for CRS，可以支撑我们后续从 handcrafted reward 走向 learned reward。第五类是 LLM planner，例如 PPDPP 和 ChatCRS，可以支撑我们在 LLM agent 外接一个可训练 Planner。最后，user simulator 相关文献可以支撑我们用 Persona-based Mock User 作为 rollout 环境来训练 Planner。
