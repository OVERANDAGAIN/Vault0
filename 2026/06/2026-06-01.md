好的。我们从 **Formulation 总框架** 开始定版，不先展开所有 state 字段。先把最关键的问题解决：

> **REC-DIAL 的一个 RL step 到底是什么？**

我的建议是：**以 agent decision turn 为一个 RL step**。也就是：

```text
user utterance / initial query
        ↓
Monitor 解析用户反馈与上下文
        ↓
StateBuilder 构造压缩 state
        ↓
Planner 选择 structured high-level action
        ↓
Speaker 把 action 实现成自然语言
        ↓
Mock User 回复
        ↓
RewardCalculator 计算 reward
        ↓
进入下一轮 state
```

形式化为：

```text
s_t → a_t → y_t → u_{t+1} → r_t, s_{t+1}
```

其中：

```text
s_t      = Planner 看到的压缩状态
a_t      = Planner 输出的结构化动作
y_t      = Speaker 生成的系统回复
u_{t+1}  = Mock User 的下一轮回复
r_t      = 当前轮 reward
s_{t+1}  = 根据用户反馈更新后的下一轮状态
```

---

# 1. 为什么用 agent decision turn 作为 RL step？

因为在 REC-DIAL 里，真正受 policy 控制的是 **Planner 的决策**，不是用户回复，也不是 Speaker 的自由生成。

所以一个 transition 应该覆盖完整的：

```text
Agent decision → Agent response → User feedback → Reward / next state
```

而不是把 user turn 和 agent turn 都拆成独立 RL step。

原因有三点。

第一，RL action 应该对应 agent 可控变量。用户说什么是环境反馈，不是 agent action。RL recommender 的基本形式也是 agent 在 state 下选择 action，然后根据用户反馈获得 reward 并转移到新 state。RL recommender survey 中也采用这种形式：agent 选择推荐 action 后，环境基于用户反馈产生 reward 和 state transition。([arXiv][1])

第二，已有 CRS 里的 RL policy 通常也是在系统轮次做决策。Sun & Zhang 的 CRS 使用 belief tracker 从用户 utterance 中抽取 facet-value pairs，并维护为 agent 的 memory state；agent 根据这个 state 决定下一步是询问 facet 还是推荐。([arXiv][2])

第三，REC-DIAL 的 User response 更适合作为 **observation / feedback**，而不是 action。Wu et al. 的 dialog-based interactive recommendation 明确指出，自然语言反馈只能部分反映用户偏好，因此适合用 POMDP 视角建模。([Enlighten Publications][3])

因此我们可以把 REC-DIAL 写成：

```text
The underlying problem is a POMDP, while the implemented Planner operates on a compressed belief state.
```

中文就是：

```text
理论上，REC-DIAL 是部分可观测的序列决策问题；
工程上，Planner 基于 Monitor 和 StateBuilder 构造出的 belief state 做决策。
```

# 2. 初版 formulation 的主结构

建议正式文档中先写成这个形式：

```text
We formulate REC-DIAL as a partially observable sequential decision-making problem.
Each episode consists of a multi-turn conversation between an agent and a persona-based mock user.
At each agent decision turn, the Planner receives a compressed belief state and outputs a structured high-level action.
The Speaker realizes this action into natural language.
The Mock User then generates a response, which serves as raw feedback.
The Monitor and RewardCalculator transform the raw feedback into updated state signals and scalar reward.
```

中文版本：

```text
我们将 REC-DIAL 形式化为一个部分可观测的序列决策问题。
每个 episode 是 agent 与 persona-based mock user 之间的一次多轮对话。
在每个 agent decision turn，Planner 接收由 Monitor 和 StateBuilder 构造的压缩 belief state，并输出一个结构化高层动作。
Speaker 将该动作实现为自然语言回复。
随后 Mock User 产生下一轮用户反馈。
用户反馈本身不是 reward，而是 raw feedback；Monitor 和 RewardCalculator 将其解释为下一轮状态信号和当前轮标量 reward。
```

# 3. REC-DIAL 的 MDP / POMDP 定义

我们可以先把它写成：

```text
M = (S, A, O, T, R, γ)
```

其中：

| 符号  | REC-DIAL 中的含义                          |
| --- | -------------------------------------- |
| `S` | Planner 可见的压缩 belief state             |
| `A` | Planner 的结构化高层动作空间                     |
| `O` | 用户回复、广告点击/接受/拒绝、退出等 observation        |
| `T` | Mock User 和系统状态更新共同形成的 transition      |
| `R` | RewardCalculator 根据用户反馈和系统行为计算的 reward |
| `γ` | discount factor，用于优化长期收益               |

这里我建议不要把它强行写成完全可观测 MDP。因为用户真实满意度、真实广告容忍度、真实任务目标都是 latent 的。Wu et al. 的 dialog-based interactive recommendation 也强调，用户自然语言反馈只能给出 partial observation，而不是完整偏好状态。([Enlighten Publications][3])

所以更严谨的写法是：

```text
Latent user state:
    z_t = true user preference, satisfaction, patience, ad tolerance, task goal

Observation:
    o_t = user utterance, explicit feedback, click / accept / reject / quit events

Belief state:
    s_t = compressed estimate of z_t and dialogue context
```

Planner 实际用的是 `s_t`，不是 `z_t`。

# 4. 一个 RL transition 的推荐定义

正式定义可以写成：

```text
At turn t, the system observes the latest user utterance u_t and historical interaction log H_t.
The Monitor produces structured user-state signals m_t.
The StateBuilder constructs a compressed planner state s_t.
The Planner samples or selects a structured action a_t ~ π(a | s_t).
The Speaker realizes a_t into a natural-language response y_t.
The Mock User returns the next user response u_{t+1} and interaction events e_{t+1}.
The RewardCalculator computes r_t.
The next state s_{t+1} is constructed from the updated history, ad log, and monitor signals.
```

用公式表示：

```text
m_t = Monitor(u_t, H_t, E_t, p)

s_t = StateBuilder(p, H_t^summary, u_t, E_t, m_t)

a_t ~ π(a | s_t)

y_t = Speaker(a_t, s_t, H_t)

u_{t+1}, e_{t+1}, done_t = MockUser(y_t, H_t, p)

r_t = R(s_t, a_t, y_t, u_{t+1}, e_{t+1})

s_{t+1} = StateBuilder(p, H_{t+1}^summary, u_{t+1}, E_{t+1}, m_{t+1})
```

其中：

```text
H_t = dialogue history
E_t = ad exposure / ad interaction history
p   = user profile
m_t = Monitor output
```

# 5. 这个 formulation 与已有工作的关系

这里我们可以在文档中明确写出我们不是凭空设计，而是结合四类工作。

## 5.1 RL-based CRS

Sun & Zhang 的 CRS 用 belief tracker 从用户 utterance 中抽取 facet-value pairs，并把它们维护为 agent 的 memory state；policy 在每轮根据 state 决定是询问 facet 还是推荐。([arXiv][2])

对应到 REC-DIAL：

```text
facet belief state
→ dialogue belief / preference state

ask facet / recommend
→ ask clarification / recommend organic / insert ad / switch topic
```

## 5.2 RL-based online advertising

DEAR 把推荐场景中的广告插入建成 RL 问题，并把广告决策拆成三个相关子问题：是否插广告、插哪个广告、插到哪个位置。([AAAI][4]) DEAR 的动机也正是广告收益和用户体验之间的权衡：广告插入不当或过频会降低用户体验，而广告过少会减少广告收益。([arXiv][5])

对应到 REC-DIAL：

```text
whether to insert ad
→ action_type = INSERT_AD or not

which ad
→ ad_id / ad_candidate_id

where to insert
→ dialogue timing / delivery mode / topic context
```

## 5.3 POMDP interactive recommendation

Wu et al. 说明，在 dialog-based interactive recommendation 中，用户可以用自然语言表达反馈，但这些反馈只能部分呈现用户偏好，因此需要在部分可观测环境下追踪用户偏好。([Enlighten Publications][3])

对应到 REC-DIAL：

```text
user response is not the true state
user response is observation
Monitor estimates belief state from observation
```

## 5.4 Modular CRS framework

CRSLab 强调 CRS 系统需要统一、可扩展、模块解耦的框架，并实现了多个 CRS 模型和数据集接口。([arXiv][6]) 这支持我们把 REC-DIAL 拆成 Mock User、Monitor、Planner、Speaker、RewardCalculator，而不是让一个大模型同时承担所有职责。

对应到 REC-DIAL：

```text
Monitor = state estimation
Planner = decision making
Speaker = language realization
RewardCalculator = feedback-to-reward mapping
Mock User = environment/user simulator
```

## 5.5 Multi-turn planning

SAPIENT 是 2025 年 NAACL 的多轮 CRS planning 工作，它指出已有 RL-based CRS 可能因为 greedy action selection 或 sampling strategy 而产生非最优的对话规划，并提出用 MCTS-based planner 改善 multi-turn planning。([ACL Anthology][7])

对应到 REC-DIAL：

```text
初版不一定直接使用 MCTS
但 formulation 应该保留长期规划空间
即 Planner 优化 episode-level return，而不是单轮 reward
```

# 6. 我建议我们接下来按这个顺序推进

第一步已经基本确定：

```text
RL step = agent decision turn
Transition = Planner action → Speaker response → User feedback → Reward / next state
Theoretical view = POMDP
Implementation view = belief-state MDP
```

接下来应该依次定：

```text
1. State space：Planner 到底看到什么
2. MonitorOutput：哪些字段由 Monitor 负责输出
3. Action space：Planner 可以做哪些动作
4. PlannerAction schema：action_type + target + intent + constraints
5. Reward function：哪些直接观测，哪些 proxy，如何组合
6. Episode termination：什么时候 done
7. Baseline planner：rule-based / contextual bandit / DQN / policy gradient
8. Open issues：哪些设计还需要实验或数据验证
```

我们下一步先进入 **State Space 设计**。这部分最关键，因为它直接决定 Monitor 输出什么、Planner 输入什么，以及后续 action/reward 是否能闭环。
