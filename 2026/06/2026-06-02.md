
# Step 1 核心结论

REC-DIAL 的 RL step 定义为：

> **一次 agent decision turn。**

也就是说，Planner 在看到当前用户回复和压缩后的上下文后，做一次高层决策；Speaker 将这个决策实现成自然语言；Mock User 对这句话产生下一轮反馈；RewardCalculator 再根据用户反馈和系统行为计算当前 reward。

![[Pasted image 20260604070158.png|226]]

系统在每轮根据 dialogue state 选择 action，再根据用户反馈获得 reward 和下一个 state。

# Slide 1：Episode 怎么定义

一个 episode 是一次完整的用户-agent 多轮对话，从用户初始需求开始，到对话结束为止。

在 REC-DIAL 中，episode 开始于：

```text
Start:
- sample / load user profile
- generate initial user query
```

结束条件可以包括：

```text
用户主动离开
达到最大轮数
任务完成
系统主动结束
严重负反馈 / 满意度过低
```


```ad-tip
**当前代码里已经有其中两个比较明确的结束条件：**
1. UserSimulator 返回 EXIT_ACTION
2. turn_count 达到 max_turns
   
 
 .
 

`DialogueEnv.step()` 里会根据 `user_response.text == EXIT_ACTION` 判断 `done`，
`run_simulation.py` 里会用 `turn_count < max_turns` 控制 episode 长度。
```

## episode 长度

我们不应该把 episode 定义成固定长度，因为多轮对话的自然结束点不同。
1. 有些用户可能很快完成需求，
2. 有些用户会继续探索，
3. 有些用户会因为广告过载或低满意度提前离开。

这符合的目标：REC-DIAL 优化的是 **长期 episode-level return**，而不是单轮收益。

# Slide 2：Turn 怎么定义

## Dialogue turn 用于日志展示

日志层面，一个 turn 可以理解成：

```text
Agent response + User response
```

或者从用户视角看：

```text
User utterance + Agent response
```


## RL step 用于训练和决策

RL 层面，一个 step 应该是：

```text
Planner 做一次决策
→ Speaker 生成一句系统回复
→ User 产生下一轮反馈
→ 得到 reward 和 next state
```


## 为什么要区分？


用户回复更合理的角色是：

```text
observation / feedback
```

POMDP interactive recommendation 相关工作也明确指出，自然语言用户反馈只能部分反映用户真实偏好，因此用户回复更适合作为 partial observation，而不是完整 state 或 action。

([Enlighten Publications][2])

# Slide 4：为什么不采用其他 step 定义

## 推荐 PPT 标题

**Why Agent Decision Turn Is the Right RL Step**

## 方案 A：把 user turn 和 agent turn 都当成 step

这个方案不推荐。

问题是：

```text
用户回复不是 agent 可控 action
用户 turn 没有 policy decision
会导致 transition 和 reward attribution 不清楚
```

如果把 `user_t` 也当成一个 RL step，那么我们会遇到一个问题：这个 step 的 action 是什么？谁在优化它？答案并不清楚。

所以这个方案不适合。
