
# Step 1 核心结论

REC-DIAL 的 RL step 应该定义为：

> **一次 agent decision turn。**

也就是说，Planner 在看到当前用户回复和压缩后的上下文后，做一次高层决策；Speaker 将这个决策实现成自然语言；Mock User 对这句话产生下一轮反馈；RewardCalculator 再根据用户反馈和系统行为计算当前 reward。

可以简化成：

```text
User feedback / initial query
        ↓
Monitor + StateBuilder
        ↓
Planner chooses structured action
        ↓
Speaker generates agent response
        ↓
Mock User responds
        ↓
RewardCalculator computes reward
        ↓
Next state
```

也就是：

```text
state_t → planner_action_t → speaker_response_t → user_feedback_{t+1} → reward_t, state_{t+1}
```

这个定义和 RL-based conversational recommender 的基本形式一致：系统在每轮根据 dialogue state 选择 action，再根据用户反馈获得 reward 和下一个 state。Sun & Zhang 的 CRS 工作也是用 belief tracker 维护对话状态，然后 policy 决定是 ask facet 还是 recommend。([arXiv][1])

# Slide 1：Episode 怎么定义

## 推荐 PPT 标题

**Episode Definition: One Complete Simulated Conversation**

## 这一页要表达的结论

一个 episode 是一次完整的用户-agent 多轮对话，从用户初始需求开始，到对话结束为止。

在 REC-DIAL 中，episode 开始于：

```text
user_profile + initial_user_query
```

结束条件可以包括：

```text
用户主动离开
达到最大轮数
任务完成
系统主动结束
严重负反馈 / 满意度过低
```

当前代码里已经有其中两个比较明确的结束条件：

```text
1. UserSimulator 返回 EXIT_ACTION
2. turn_count 达到 max_turns
```

`DialogueEnv.step()` 里会根据 `user_response.text == EXIT_ACTION` 判断 `done`，`run_simulation.py` 里也会用 `turn_count < max_turns` 控制 episode 长度。

## 建议写在 PPT 上

```text
Episode = one complete interaction session between Agent and Mock User

Start:
- sample / load user profile
- generate initial user query

End:
- user exits
- max turns reached
- task completed
- severe dissatisfaction / failure state
```

## 设计理由

我们不应该把 episode 定义成固定长度，因为多轮对话的自然结束点不同。
有些用户可能很快完成需求，
有些用户会继续探索，
有些用户会因为广告过载或低满意度提前离开。

这也符合我们的目标：REC-DIAL 优化的是 **长期 episode-level return**，而不是单轮收益。


## 当前代码映射

当前代码里：

```text
ProfileSimulator.generate_initial_query()
→ 生成 episode 初始用户 query

DialogueEnv.reset()
→ 重置环境历史和 simulator state

DialogueEnv.step()
→ 每轮执行 agent response 后的用户反馈生成

SimulatorState.check_exit()
→ 根据 patience / max_turns / satisfaction 判断是否退出
```

这里要注意：当前代码里的 `reward = 0.0` 只是占位。后续 formulation 需要正式定义 per-step reward。

# Slide 2：Turn 怎么定义

## 推荐 PPT 标题

**Turn Definition: Logging Turn vs RL Decision Step**

## 这一页要表达的结论

REC-DIAL 里最好区分两种 turn：

```text
1. Dialogue turn：用于日志展示
2. RL step：用于训练和决策
```

它们不完全一样。

## Dialogue turn

日志层面，一个 turn 可以理解成：

```text
Agent response + User response
```

或者从用户视角看：

```text
User utterance + Agent response
```

这主要用于保存 conversation log、可视化和人工检查。

## RL step

RL 层面，一个 step 应该是：

```text
Planner 做一次决策
→ Speaker 生成一句系统回复
→ User 产生下一轮反馈
→ 得到 reward 和 next state
```

也就是：

```text
agent decision turn = one RL step
```

## 为什么要区分？

因为在 RL 中，action 必须是 agent 可控的东西。用户说什么不是 agent action，而是 environment feedback。

所以不能把：

```text
user says something
```

也当成一个 RL action step。

用户回复更合理的角色是：

```text
observation / feedback
```

POMDP interactive recommendation 相关工作也明确指出，自然语言用户反馈只能部分反映用户真实偏好，因此用户回复更适合作为 partial observation，而不是完整 state 或 action。([Enlighten Publications][2])

## 建议写在 PPT 上

```text
Dialogue turn:
- for logging and visualization

RL step:
- for decision making and training
- one Planner decision followed by one User feedback

Key choice:
User response is raw feedback / observation, not agent action.
```

# Slide 3：一个 RL step 的完整闭环

## 推荐 PPT 标题

**RL Step in REC-DIAL: Agent Decision → User Feedback**

## 这一页要表达的结论

一个 RL step 应该覆盖完整的：

```text
Planner action → Speaker response → Mock User feedback → Reward / next state
```

不要把 Planner 决策和用户反馈拆开，否则 reward credit assignment 会变得混乱。

## Step 具体流程

第 `t` 步开始时，系统已经看到了当前用户输入：

```text
u_t
```

然后：

```text
1. Monitor 分析 u_t 和近期历史，输出用户状态信号
2. StateBuilder 构造 PlannerState_t
3. Planner 根据 PlannerState_t 选择 structured action_t
4. Speaker 根据 action_t 生成系统回复 y_t
5. Mock User 根据 y_t 生成下一轮用户反馈 u_{t+1}
6. RewardCalculator 计算 reward_t
7. 系统更新历史、广告曝光记录和下一轮 state
```

可以写成：

```text
PlannerState_t
→ PlannerAction_t
→ AgentUtterance_t
→ UserResponse_{t+1}
→ Reward_t, PlannerState_{t+1}
```

## 建议写在 PPT 上

```text
One RL step = one agent decision cycle

Input:
- latest user utterance
- compressed history
- monitor signals
- ad exposure history
- candidate pool

Output:
- structured planner action
- user feedback
- scalar reward
- next planner state
```

## 当前代码映射

当前代码已经基本有这个流程：

```text
DialogueAgent.act(history)
    → Monitor.evaluate_turn()
    → Planner.update_monitor_signals()
    → Planner.should_insert_ad()
    → Planner.select_next_topic()
    → Speaker.generate_utterance()

DialogueEnv.step(bot_utterance)
    → Simulator.generate_response()
    → return history, reward, done, info
```

也就是说，当前代码结构已经支持我们把 RL step 定义为 agent decision turn。

但当前实现还有三个问题：

```text
1. Planner 输入还不是统一的 PlannerState
2. Planner 输出还不是统一的 PlannerAction
3. Env.step() 里的 reward 目前还是 0.0，占位性质很强
```

这三个问题会在后面的 state、action、reward 设计中解决。

# Slide 4：为什么不采用其他 step 定义

## 推荐 PPT 标题

**Why Agent Decision Turn Is the Right RL Step**

## 方案 A：把 user turn 和 agent turn 都当成 step

这个方案不推荐。

问题是：

```text
用户回复不是 agent 可控 action
用户 turn 没有 policy decision
会导致 transition 和 reward attribution 不清楚
```

如果把 `user_t` 也当成一个 RL step，那么我们会遇到一个问题：这个 step 的 action 是什么？谁在优化它？答案并不清楚。

所以这个方案不适合。

---

## 方案 B：把 token generation 当成 RL step

这个方案也不推荐作为初版。

如果每个 token 都是一个 action，那么问题会变成：

```text
language-model-level RL
```

这更接近 PPO / GRPO / RLHF 的设定，而不是当前 REC-DIAL 的模块化 Planner 设定。

我们当前的目标是先训练或设计 **高层 Planner**，不是直接优化 Speaker 的每个 token。

因此 token-level RL 暂时不作为 formulation 初版重点。

---

## 方案 C：把 user-agent-user 作为完整 transition

这个方案本质上和我们推荐的方案兼容，但表述上容易混乱。

更清楚的说法是：

```text
RL step = agent decision turn
transition = agent action followed by user feedback
```

也就是说，RL step 的核心 action 是 PlannerAction，但这个 step 的 transition 必须包含之后的用户反馈。

最终推荐写法：

```text
One RL step corresponds to one agent decision turn.
The transition is completed after the Mock User responds.
```

# Step 1 的正式设计结论

可以在最终文档中写成下面这一段：

> In REC-DIAL, we define one RL step as one agent decision turn. At each step, the system observes the latest user utterance and the compressed interaction context. The Monitor estimates user-state signals, and the StateBuilder constructs a PlannerState. The Planner then selects a structured high-level action, which is realized by the Speaker into a natural-language response. The Mock User generates the next user response and interaction events. The RewardCalculator computes the scalar reward, and the next PlannerState is constructed from the updated dialogue history, ad exposure log, and Monitor signals.

中文版本：

> 在 REC-DIAL 中，我们将一个 RL step 定义为一次 agent decision turn。每一步开始时，系统看到当前用户回复和压缩后的对话上下文；Monitor 估计用户状态信号，StateBuilder 构造 PlannerState；Planner 基于该状态选择一个结构化高层动作；Speaker 将动作实现为自然语言回复；Mock User 随后产生下一轮用户反馈和交互事件；RewardCalculator 根据用户反馈、广告事件和状态变化计算当前 reward；最后系统更新历史、广告记录和下一轮 PlannerState。

# Step 1 的模块接口结论

从 formulation 角度，Step 1 最终确定的接口链路是：

```text
u_t, H_t, E_t, p
        ↓
Monitor
        ↓
PlannerState_t
        ↓
Planner
        ↓
PlannerAction_t
        ↓
Speaker
        ↓
AgentUtterance_t
        ↓
Mock User / Environment
        ↓
UserResponse_{t+1}, events_{t+1}, done_t
        ↓
RewardCalculator
        ↓
Reward_t, PlannerState_{t+1}
```

其中：

| 符号 / 对象              | 含义                  |
| -------------------- | ------------------- |
| `u_t`                | 当前用户回复或初始 query     |
| `H_t`                | 当前对话历史              |
| `E_t`                | 广告曝光与交互历史           |
| `p`                  | 用户画像                |
| `PlannerState_t`     | Planner 可见的压缩状态     |
| `PlannerAction_t`    | Planner 输出的结构化动作    |
| `AgentUtterance_t`   | Speaker 生成的系统自然语言回复 |
| `UserResponse_{t+1}` | Mock User 对系统回复的反馈  |
| `events_{t+1}`       | 广告接受、点击、拒绝、退出等事件    |
| `Reward_t`           | 当前轮训练 reward        |
| `done_t`             | episode 是否结束        |

# Step 1 对当前代码的改造要求

这一页可以放在 formulation 后面，说明“当前代码如何对齐未来设计”。

## 当前代码已有

```text
DialogueAgent.act()
→ 已经承担 Monitor → Planner → Speaker 的控制流程

DialogueEnv.step()
→ 已经承担 AgentUtterance → UserResponse 的环境反馈流程

ProfileSimulator
→ 已经维护 hidden user state，例如 patience, satisfaction, ads_seen

Monitor
→ 已经输出 satisfaction, switch_intent, ad_resistance 等信号
```

## 需要改造

```text
1. 新增 PlannerState
当前 state 分散在 history、current_topic_id、topic_history、cached_signals、simulator state 和 ad log 中。

2. 新增 PlannerAction
当前 Planner 输出被拆成 topic selection 和 ad insertion decision，后续需要统一。

3. 新增 RewardCalculator
当前 Env.step() 里的 reward = 0.0，需要变成正式 reward breakdown。

4. 新增 trajectory log
后续每一步应该记录：
PlannerState_t, PlannerAction_t, AgentUtterance_t, UserResponse_{t+1}, Reward_t, PlannerState_{t+1}, done_t
```
