
# 总体分 6 步

## Step 1：确定 RL step / episode / transition

这一部分是 formulation 的地基，先定清楚：

```text
一个 episode 是什么？
一个 turn 是什么？
一个 RL step 是什么？
user response 在 formulation 里是什么？
Planner action 到 reward 的闭环怎么定义？
```

核心结论大概率是：

```text
RL step = agent decision turn
transition = Planner action → Speaker response → Mock User feedback → reward / next state
user response = raw observation / feedback, not reward itself
```

这一部分可以做成 1–2 页 PPT。

---

## Step 2：设计 Planner State

这是最重要的一步，因为它决定：

```text
Monitor 要输出什么
Planner 要输入什么
RewardCalculator 需要读哪些信号
后续训练时 state vector 怎么构造
```

我们会重点回答：

```text
完整 history 为什么不能直接作为 state？
哪些信息必须进入 state？
哪些信息只是日志，不进入 Planner？
哪些是 observable，哪些是 inferred proxy？
哪些 state 来自 Monitor，哪些来自 ad log / user profile / history summary？
```

这一部分会形成一个核心表格：

| State block | 字段 | 来源 | 含义 | 为什么需要 |
| ----------- | -- | -- | -- | ----- |

比如：

```text
user_profile
dialogue_belief
monitor_signals
ad_context
history_summary
last_action
candidate_context
```

这是整个 formulation 里最需要仔细打磨的部分。

---

## Step 3：设计 MonitorOutput

这一步和 state 紧密相关，但要单独做，因为你们会议里特别强调 Monitor 和 Planner 解耦。

重点是把这件事说清楚：

```text
Monitor 不做决策。
Monitor 只做状态估计。
```

例如：

```text
不输出 should_insert_ad
而输出 ad_resistance_score, fatigue_score, topic_saturation_score
```

这一部分我们会明确：

```text
Monitor 输出哪些 score
每个 score 代表什么
取值范围是什么
是否需要 confidence
这些 score 如何进入 Planner state
哪些 score 可以进入 reward
```

这部分可以做成 1–2 页 PPT。

---

## Step 4：设计 Planner Action Space 和 PlannerAction schema

这一步回答：

```text
Planner 到底能做什么？
动作是扁平的还是层级的？
是直接选自然语言，还是选 structured action？
371 个 topic/ad candidates 怎么进入动作空间？
```

我建议主张：

```text
Planner action = hierarchical structured action
```

也就是：

```text
action_type + target candidate + intent + constraints
```

例如：

```json
{
  "action_type": "INSERT_AD",
  "target": {
    "topic_id": "travel_budget",
    "ad_id": "ad_219"
  },
  "intent": {
    "goal": "provide relevant sponsored option"
  },
  "constraints": {
    "must_disclose_sponsored": true,
    "avoid_hard_selling": true
  }
}
```

这里会重点参考：

```text
Sun & Zhang CRS: ask / recommend
DEAR: 是否插广告 + 哪个广告 + 插入位置
SAPIENT: action_type + candidate binding
```

---

## Step 5：设计 Reward 和优化目标

这一步回答：

```text
Planner 最终优化什么？
reward 包含哪些部分？
哪些是直接可观测，哪些是 proxy？
如何避免消极策略？
```

重点不是写复杂公式，而是把 reward 的组成讲清楚：

```text
User value
Task progress
Business / ad value
Risk / annoyance penalty
Terminal outcome
```

还要重点讲几个 degenerate policy：

```text
完全不插广告
过度插广告
只讨好用户但不推进任务
只安全闲聊
频繁切话题
```

这一部分对你们很关键，因为 reward 会决定后续 Planner 学到什么。

---

## Step 6：整合成最终 formulation 文档 / PPT 骨架

最后一步把前面内容整合成：

```text
1. Problem formulation
2. RL step definition
3. State design
4. Monitor output
5. Planner input
6. Planner action space
7. Speaker input
8. Transition and reward
9. Baseline planner choices
10. Open questions
11. References
```

这一版就可以直接作为初版提交材料。后面如果要变成正式 PPT，我可以再帮你压缩成每页 slide 的标题、要点和讲稿。

# 什么时候给我代码库最合适？

我建议在 **Step 2 之前或 Step 2 进行中** 给我代码库。

原因是：State 设计不能完全脱离你们已有实现。代码里很可能已经有这些东西：

```text
user_profile 的具体字段
mock_user 的输入输出
RewardCalculator 当前有哪些 reward components
Planner 现在的输入输出格式
Speaker 现在接收什么参数
Monitor 是否已经有雏形
topic/ad candidate JSON 的结构
episode log 怎么存
```

如果我在没有看代码的情况下先完整设计 state/action，很容易出现两种问题：

第一，设计得太理想化，和现有框架对不上。

第二，漏掉已有代码里已经实现或隐含的字段，导致重复设计。

所以更高效的路线是：

```text
现在先定 Step 1：RL step 和 transition
然后你给我代码库
我基于代码库做 Step 2：State + MonitorOutput + Action schema
再继续 Step 3-6
```

换句话说，代码库最好在 **我们进入 State Space 具体字段设计之前**提供。
