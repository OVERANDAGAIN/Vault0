
# Step 2：PlannerState 设计

REC-DIAL 的 PlannerState 应该设计成一个 **decision-oriented belief state**。

也就是说，它不是为了完整记录对话，而是为了回答：

> 在当前局面下，Planner 下一步应该采取哪类高层动作？

---

# 1. PlannerState 的基本定位

REC-DIAL 中，Planner 看到的 state 应该是：

```text
structured belief state
+ compressed dialogue memory
+ recent local context
```

更准确地说：

```text
PlannerInput = PlannerState + MemoryContext + CandidatePool + Constraints
```

其中：

```text
PlannerState:
    决策所需的结构化状态

MemoryContext:
    压缩历史 + 最近几轮原文

CandidatePool:
    当前可选 topic / item / ad candidates

Constraints:
    安全规则、广告频率上限、最大轮数等硬约束
```

**State 描述当前局面，MemoryContext 提供语义依据，CandidatePool 提供可选对象，Constraints 限制动作边界。**

---

# 2. 为什么不能直接用完整 history？

完整 history 有三个问题。

第一，它太长。随着轮数增加，直接把完整对话塞给 Planner 会让输入不稳定，也不适合后续做 RL 训练。

第二，它噪声太多。很多历史句子对当前决策没有直接价值，真正有价值的是用户需求、偏好、拒绝、满意度变化、广告反馈等结构化信息。

第三，它不利于 credit assignment。如果 Planner 每轮都读完整 history，很难判断当前 action 的效果来自哪里。

这也是多数相关工作的共同做法：不是保留完整历史，而是用某种压缩状态表示历史影响。



但是，**完全不保留最近几轮原文也不合适**。因为 REC-DIAL 是自然语言对话，最近几轮有很多局部语义信息，例如指代、省略、语气、刚刚的拒绝、刚刚的广告反应。这些信息很难全部被结构化字段覆盖。

所以我建议采用：

> **structured state + rolling summary + recent K turns** 的混合设计。

---

# 3. 历史应该怎么进入 Planner？

历史信息分三层进入系统。

## 第一层：Structured Belief

这是 PlannerState 的主体，是给 RL / rule-based Planner 用的。

它不保存原文，而保存压缩后的状态变量，例如：

```text
用户需求是否清楚
当前是否有推荐机会
用户是否疲劳
广告负担是否过高
上一步动作是否失败
```

这部分可以向量化，后续可以直接接 rule-based、contextual bandit、DQN 或 policy gradient。

---

## 第二层：Rolling Summary

这是长期历史摘要，用于保留完整对话的长期语义。

它应该维护：

```text
用户当前目标
稳定偏好
明确拒绝
已经推荐过什么
已经解释过什么
用户对广告的历史态度
未解决的问题
不要重复的内容
对话目前处于什么方向
```

Rolling Summary 不是每轮把完整 history 重新总结一遍，而是更合理地做 **增量更新**：

```text
old_summary + latest agent action + latest user response
→ updated_summary
```

这样更稳定，也更容易控制信息丢失。

---

## 第三层：Recent K Turns

最近几轮原文仍然需要保留。建议初版保留最近 **2–3 个 user-agent exchange**。

原因是：

```text
最近几轮保留局部语境；
rolling summary 保留长期语境；
structured state 保留决策变量。
```

Recent K Turns 主要给 LLM Planner 和 Speaker 用，尤其用于：

```text
保持回复连贯；
理解代词和省略；
识别用户刚刚的语气；
避免重复上一句话；
让 Speaker 能自然接话。
```

对纯 RL Planner 来说，recent turns 可以不直接作为高维文本输入，而是通过 embedding 或 Monitor 提取后的结构化变量进入 state。

---

# 4. 最终推荐的 PlannerState 结构

我建议 Step 2 最终采用 **5 个核心状态块**：

```text
1. User Need & Preference Belief
2. Interaction Status
3. Ad Opportunity & Burden
4. Action Feedback Memory
5. Safety & Hard Constraints
```

这五块是按 **决策功能** 分的，不是按代码模块分的。

---

# 1. 先修正一个关键表述

之前说：

```text id="cn9jaj"
Planner 的输入是 Monitor 的输出
```

这个说法不够准确。

应该改成：

```text id="d9e1ah"
Planner 的输入是由 Monitor 输出参与构造的 PlannerState。
```

---

# 2. Monitor 应该输出什么？

Monitor 不应该输出完整 PlannerState。它应该只负责 **当前轮用户反馈解析**。

也就是说，Monitor 主要回答：

```text id="sf5fj8"
用户这一轮表达了什么？
用户对上一轮系统行为的反应如何？
用户是否满意 / 不满 / 疲劳 / 反感广告？
用户是否提出新偏好或新约束？
用户是否想换话题或结束？
是否涉及敏感或不适合广告的内容？
```

所以更合理的 `MonitorOutput` 是：

```json id="q5h2no"
{
  "user_intent": "recommendation | exploration | information_seeking | comparison | complaint | casual_chat | unknown",

  "preference_delta": {
    "positive": [],
    "negative": [],
    "constraints": {},
    "open_needs": []
  },

  "feedback_to_last_action": {
    "feedback_type": "positive | neutral | negative | ignored | explicit_rejection",
    "target": "last_response | topic | recommendation | ad | unknown"
  },

  "user_state_signals": {
    "satisfaction_level": "low | medium | high",
    "engagement_level": "low | medium | high",
    "fatigue_level": "low | medium | high",
    "switch_intent_level": "low | medium | high",
    "ad_resistance_level": "low | medium | high",
    "termination_risk_level": "low | medium | high"
  },

  "safety_signals": {
    "sensitive_content_flag": false,
    "ad_allowed": true,
    "policy_constraint_type": "none | no_ad | safe_response | avoid_recommendation"
  },

  "confidence": {
    "overall": 0.75
  },

  "evidence": {
    "brief_reason": "User gave a positive response and added a budget constraint."
  }
}
```

这才是 Monitor 合理的输出范围。

Monitor 不应该直接输出：

```text id="w7295g"
should_insert_ad
should_switch_topic
should_recommend
next_action_type
```

这些是 Planner 的职责。

---

# 3. 那 Step 2 的五类核心 state 是否合理？

五类 state 仍然合理，但要改清楚：**它们不是全部由 Monitor 直接输出的，而是由 StateBuilder 综合生成的。**

更准确的五类是：

```text id="d5ux66"
1. User Need & Preference Belief
2. Interaction Status
3. Ad Opportunity & Burden
4. Action Feedback Memory
5. Safety & Hard Constraints
```

但每一类的来源不同。

| PlannerState block            | 主要来源                                                      | Monitor 是否负责          |
| ----------------------------- | --------------------------------------------------------- | --------------------- |
| User Need & Preference Belief | BeliefTracker / DialogueMemory + Monitor preference_delta | Monitor 只提供本轮增量       |
| Interaction Status            | Monitor user_state_signals + turn info                    | Monitor 是主要来源         |
| Ad Opportunity & Burden       | AdMatcher + AdLog + Monitor ad_resistance                 | Monitor 只负责用户反感部分     |
| Action Feedback Memory        | LastActionLog + Monitor feedback_to_last_action           | Monitor 负责解释用户反馈      |
| Safety & Hard Constraints     | Safety classifier / rules + Monitor safety_signals        | Monitor 可参与，但硬约束由系统维护 |

这个表非常重要。它能避免误解：**PlannerState 不是 MonitorOutput 的同义词。**

---

# 4. 重新定义模块串联

更合理的模块链路是：

```text id="xlxooh"
UserResponse + recent_turns
        ↓
Monitor
        ↓
MonitorOutput

MonitorOutput + old DialogueMemory
        ↓
BeliefTracker / MemoryUpdater
        ↓
Updated DialogueMemory

AdLog + CandidatePool + current belief
        ↓
AdMatcher / CandidateEvaluator
        ↓
AdOpportunitySummary

MonitorOutput + DialogueMemory + AdOpportunitySummary + LastActionLog + Constraints
        ↓
StateBuilder
        ↓
PlannerState

PlannerState + CandidatePool + MemoryContext
        ↓
Planner
        ↓
PlannerAction
```

这条链路更清楚。

它也符合 DEAR 的启发：广告 RL 的 state 不只是“用户当前反馈”，还包含广告历史、推荐上下文和当前候选列表表示；DEAR 的 action 同时处理是否插广告、插哪个广告、插到哪个位置，说明广告决策需要上下文、历史负担和候选广告共同参与，而不是单靠一个 Monitor 信号。([arXiv][2])

# 5. 重新给出 Step 2 的 PlannerState

现在这个版本会更合理。

## 5.1 User Need & Preference Belief

这部分不应该由 Monitor 单独决定。Monitor 只抽取本轮用户新表达的偏好，BeliefTracker 负责累积成稳定 belief。

```json id="2wpbch"
"user_need_belief": {
  "current_intent": "exploration",
  "need_clarity": "medium",
  "preference_confidence": "low",
  "response_readiness": "medium",
  "exploration_value": "high",
  "recommendation_opportunity": "medium",
  "positive_preferences": ["budget-friendly"],
  "negative_preferences": ["luxury"],
  "constraints": {},
  "open_needs": ["specific category", "budget range"]
}
```

来源：

```text id="sox44b"
current_intent:
    主要来自 Monitor

positive_preferences / negative_preferences / constraints:
    来自 DialogueMemory，Monitor 每轮提供增量

need_clarity / preference_confidence / response_readiness:
    由 BeliefTracker / StateBuilder 根据 memory + MonitorOutput 估计

recommendation_opportunity:
    由 StateBuilder 综合 need belief + candidate availability + interaction status 得出
```

这里的关键是：**Monitor 不直接维护完整用户需求，它只更新本轮观察。**

---

## 5.2 Interaction Status

这部分 Monitor 参与最多，因为它主要是对用户当前状态的估计。

```json id="nbnkwd"
"interaction_status": {
  "turn_stage": "middle",
  "topic_stage": "developing",
  "engagement_level": "high",
  "satisfaction_level": "medium",
  "fatigue_level": "low",
  "switch_intent_level": "low",
  "termination_risk_level": "low"
}
```

来源：

```text id="jz6c62"
engagement_level / satisfaction_level / fatigue_level / switch_intent_level / termination_risk_level:
    主要来自 Monitor

turn_stage:
    来自 episode turn counter

topic_stage:
    来自 DialogueMemory / topic tracker，不应该只由 Monitor 判断
```

这里要避免让 Monitor 输出 `topic_ended` 这种决策型变量。更好的做法是让 Monitor 输出 `switch_intent_level`，再让 StateBuilder 根据 topic depth、topic summary、用户反馈等生成 `topic_stage`。

---

## 5.3 Ad Opportunity & Burden

这部分不能主要靠 Monitor。Monitor 只能判断用户是否反感广告。真正的广告机会来自 AdMatcher / CandidateEvaluator。

```json id="5u686f"
"ad_opportunity_burden": {
  "ad_candidate_relevance": "medium",
  "ad_value_level": "medium",
  "recent_ad_load": "low",
  "ad_resistance_level": "low",
  "ad_cooldown_status": "satisfied",
  "last_ad_outcome": "none"
}
```

来源：

```text id="7xnlsy"
ad_candidate_relevance:
    来自 AdMatcher / CandidateEvaluator

ad_value_level:
    来自 ad metadata / revenue proxy

recent_ad_load:
    来自 AdLog

ad_resistance_level:
    来自 Monitor

ad_cooldown_status:
    来自 AdLog + rule constraint

last_ad_outcome:
    来自 LastActionLog + Monitor feedback_to_last_action
```

这里一定要说明：**Monitor 不负责判断“有没有好广告”，它只负责判断用户是否可能抵触广告。**

广告机会是候选广告系统的输出；广告负担是广告历史和用户反感共同决定的。

---

## 5.4 Action Feedback Memory

这部分由 LastActionLog 和 Monitor 共同生成。

```json id="r2gyyo"
"action_feedback_memory": {
  "last_action_type": "guide_exploration",
  "last_target_type": "topic",
  "last_feedback_type": "positive",
  "recent_failure_count": 0,
  "recent_repetition_level": "low",
  "recent_positive_signal": true
}
```

来源：

```text id="8kxfwp"
last_action_type / last_target_type:
    来自 PlannerAction log

last_feedback_type:
    来自 Monitor 对用户回复的解释

recent_failure_count / recent_repetition_level:
    来自 trajectory log / StateBuilder
```

Monitor 在这里的作用是把用户回复解释成：

```text id="m10xz7"
positive / neutral / negative / explicit_rejection / ignored
```

但它不负责维护完整历史统计。

---

## 5.5 Safety & Hard Constraints

这部分要尽量独立，不应该完全交给 Monitor。

```json id="j8wjq9"
"safety_constraints": {
  "sensitive_content_flag": false,
  "ad_allowed": true,
  "must_avoid_topics": [],
  "policy_constraint_type": "none"
}
```

来源：

```text id="rgyknj"
sensitive_content_flag:
    Monitor / safety classifier

ad_allowed:
    rules + sensitive content + ad policy

must_avoid_topics:
    safety rules + user explicit rejection

policy_constraint_type:
    constraint manager
```

也就是说，Monitor 可以检测敏感内容，但最终是否禁止广告，应由 ConstraintManager 统一决定。

# 6. 这时 Planner 的输入到底是什么？

现在可以明确写成：

```json id="qg2e9j"
{
  "planner_state": {
    "user_need_belief": {},
    "interaction_status": {},
    "ad_opportunity_burden": {},
    "action_feedback_memory": {},
    "safety_constraints": {}
  },

  "memory_context": {
    "rolling_summary": {},
    "recent_turns": []
  },

  "candidate_pool": {
    "topic_candidates": [],
    "organic_candidates": [],
    "ad_candidates": []
  },

  "constraints": {
    "max_turns": 12,
    "min_ad_gap": 2,
    "max_ads_per_episode": 3
  }
}
```

所以严格来说：

```text id="hjh6mj"
Planner 的直接输入不是 MonitorOutput。
Planner 的直接输入是 PlannerInput。
MonitorOutput 是 PlannerInput 的上游组成部分。
```

如果你们团队想保留“Planner 输入来自 Monitor”的说法，可以改成更安全的表述：

> Planner uses the state signals estimated by Monitor, together with dialogue memory and candidate context.

不要写：

> Planner input is Monitor output.

这句话容易让人误解为 Monitor 承担了全部 state construction。

# 7. 那 MonitorOutput 是否还应该包含五类核心信息？

不应该。

MonitorOutput 应该更小、更清楚。推荐只包含四类：

```text id="v5dlrf"
1. User intent and preference delta
2. Feedback to last action
3. User state signals
4. Safety / ad-resistance signals
```

即：

```json id="6w1tx3"
{
  "user_intent": "...",

  "preference_delta": {
    "positive": [],
    "negative": [],
    "constraints": {},
    "open_needs": []
  },

  "feedback_to_last_action": {
    "feedback_type": "positive | neutral | negative | ignored | explicit_rejection",
    "target": "recommendation | ad | topic | whole_response | unknown"
  },

  "user_state_signals": {
    "satisfaction_level": "low | medium | high",
    "engagement_level": "low | medium | high",
    "fatigue_level": "low | medium | high",
    "switch_intent_level": "low | medium | high",
    "ad_resistance_level": "low | medium | high",
    "termination_risk_level": "low | medium | high"
  },

  "safety_signals": {
    "sensitive_content_flag": false,
    "ad_allowed_hint": true
  }
}
```

这个 MonitorOutput 是可行的，因为它只需要看：

```text id="vko5ka"
latest user response
recent turns
last planner action
possibly rolling summary
```

它不需要知道完整广告池，也不需要直接做策略选择。

# 8. History summary 和 recent turns 在这个接口中的位置

这里也要进一步明确。

## Rolling summary

Rolling summary 由 MemoryUpdater 维护，不应该是 Monitor 每轮临时输出。

它包含：

```text id="b7zc59"
长期用户目标
稳定偏好
明确拒绝
已经推荐过的东西
已解决 / 未解决的问题
广告反馈历史
不要重复的内容
```

Monitor 可以读取 rolling summary 辅助判断，但不应该独自维护 summary。

---

## Recent turns

Recent turns 是 Planner 和 Speaker 的局部上下文。建议保留最近 2–3 个 exchange。

它的作用不是替代 structured state，而是补充：

```text id="5b4mkm"
指代解析
局部语气
刚刚说过什么
避免重复
让 Speaker 自然接话
```

所以在接口上：

```text id="v77z20"
PlannerState:
    结构化决策变量

MemoryContext:
    rolling summary + recent turns
```

不要把 recent turns 混进 PlannerState 的结构化字段里。

---

# 9. 修正后的 Step 2 最终结论

这版 Step 2 应该这样定：

> REC-DIAL 的 Planner 输入不是原始 MonitorOutput，而是由 StateBuilder 构造的 PlannerInput。Monitor 负责从当前用户回复中抽取本轮意图、偏好增量、对上一动作的反馈、满意度/疲劳/广告反感等用户状态信号；BeliefTracker 负责维护长期用户需求和偏好 belief；AdMatcher 和 AdLog 负责提供广告机会与广告负担；LastActionLog 负责维护动作反馈记忆；ConstraintManager 负责提供安全和硬约束。StateBuilder 将这些信息合并为 PlannerState。Planner 再基于 PlannerState、MemoryContext、CandidatePool 和 Constraints 选择结构化动作。

更短一点：

```text id="5jbd3o"
MonitorOutput = per-turn user feedback interpretation

PlannerState = decision-oriented belief state

PlannerInput = PlannerState + MemoryContext + CandidatePool + Constraints
```

这才是清楚、可行、模块边界合理的接口设计。
