# Step 2：PlannerInput 设计总结

REC-DIAL 中，Planner 的输入不应该是完整对话历史，也不应该只是 Monitor 的输出。更合理的设计是：

```text
PlannerInput_t = BeliefState_t + DialogueContext_t + AdContext_t + Constraints_t
```

其中：

```text
BeliefState:
    系统当前对用户需求和偏好的理解。

DialogueContext:
    历史摘要、最近几轮对话、上一轮动作反馈。

AdContext:
    当前广告候选、广告负担、用户广告态度。

Constraints:
    安全、业务、广告频率等硬约束。
```

这四块是 Planner 下一步决策所需的最小充分信息。它们比“直接输入完整 history”更稳定，也比把 Monitor 的所有打分项直接堆给 Planner 更清楚。

---

# 1. PlannerInput 为什么不是 MonitorOutput

Monitor 的职责是解释当前用户回复，而不是构造完整 Planner 输入。

也就是说，Monitor 负责回答：

```text
用户这一轮表达了什么？
用户对上一轮动作是什么反馈？
用户有没有新增偏好或限制？
用户是否拒绝广告？
是否出现敏感内容？
```

MonitorOutput 可以是：

```json
{
  "intent": "seek_recommendation",
  "preference_update": {
    "positive": ["budget-friendly"],
    "negative": [],
    "constraints": {
      "budget": "low"
    },
    "open_questions": []
  },
  "feedback_to_last_action": {
    "type": "positive",
    "target": "clarification",
    "brief_reason": "User answered the clarification and gave a budget preference."
  },
  "user_ad_attitude": "neutral",
  "sensitive_content": false
}
```

但 Planner 需要的不只是这些。它还需要长期记忆、最近上下文、广告候选、广告历史和硬约束。所以需要一个 StateBuilder：

```text
MonitorOutput
+ previous DialogueContext
+ AdLog / CandidatePool
+ LastActionLog
+ Constraints
→ StateBuilder
→ PlannerInput
```


---

# 2. BeliefState：用户需求和偏好理解

BeliefState 只负责描述用户当前目标和偏好，不要塞太多主观评分。

推荐最小结构：

```json
{
  "current_intent": "explore | ask_info | seek_recommendation | compare | complain | casual",
  "user_goal": "用户当前主要目标的简短描述",
  "positive_preferences": [],
  "negative_preferences": [],
  "constraints": {},
  "open_questions": []
}
```

各字段含义：

| 字段                     | 作用                          |
| ---------------------- | --------------------------- |
| `current_intent`       | 判断用户当前是在探索、提问、要推荐、比较、抱怨还是闲聊 |
| `user_goal`            | 保持系统回复围绕用户目标                |
| `positive_preferences` | 用于 topic / item / ad 匹配     |
| `negative_preferences` | 避免重复用户明确拒绝的内容               |
| `constraints`          | 预算、地点、时间、类别等限制              |
| `open_questions`       | 尚未解决但可能影响下一步的问题             |

这里不要把逻辑写成“信息不够就问，信息够就推荐”。REC-DIAL 是开放式对话，不一定需要锁定唯一 item。更准确的表述是：

> BeliefState helps the Planner decide whether to explore, clarify, suggest, recommend, advertise, repair, or close.

也就是说，BeliefState 不是为了判断“是否足够推荐”，而是为了支持多种高层动作选择。

# 3. DialogueContext：history 怎么进入 Planner

完整 history 不直接作为 Planner 输入，但历史信息不能丢。建议用三层结构：

```text
DialogueContext = rolling_summary + recent_turns + last_action_feedback
```

## 3.1 Rolling Summary：长期摘要

Rolling summary 是对长期历史的压缩，不是逐句总结。它只保留对后续决策有影响的信息。

建议总结以下内容：

```text
1. 用户当前目标
2. 稳定偏好
3. 明确拒绝
4. 已经推荐或解释过的内容
5. 已解决的问题
6. 未解决的问题
7. 广告反馈历史
8. 不要重复的内容
```

推荐结构：

```json
{
  "rolling_summary": "用户正在寻找预算友好的日本旅行住宿建议，偏好交通方便，不想要豪华酒店和人太多的区域。",
  "resolved_points": [
    "已确认预算敏感",
    "已确认交通便利很重要"
  ],
  "unresolved_points": [
    "还未确定具体城市",
    "还未确定旅行时间",
    "还未确定住宿类型"
  ],
  "do_not_repeat": [
    "豪华酒店推荐",
    "已经解释过的基础预算建议"
  ],
  "ad_feedback_summary": "目前尚未展示广告，用户没有表达广告反感。"
}
```

### Rolling Summary 的更新方式

不要每轮从完整 history 重新总结。更合理的是增量更新：

```text
previous_summary
+ last_planner_action
+ last_agent_response
+ latest_user_response
+ monitor_output
→ updated_summary
```

更新规则可以是：

```text
如果用户新增偏好：
    加入 stable_preferences 或 constraints。

如果用户明确拒绝：
    加入 explicit_rejections / do_not_repeat。

如果系统已经回答某个问题：
    从 unresolved_points 移到 resolved_points。

如果用户重新提出某个问题：
    保留为 unresolved_points，说明之前没有解决好。

如果用户接受或拒绝广告：
    更新 ad_feedback_summary。

如果系统给过某个推荐：
    加入 do_not_repeat 或 already_recommended。
```

这样 summary 会比较稳定，也便于调试。

---

## 3.2 Recent Turns：最近 2–3 轮原文

除了 rolling summary，还应该保留最近几轮原文。建议初版保留最近 **2–3 个 exchange**，也就是最近 4–6 条 utterance。

原因是 rolling summary 解决长期记忆，但 recent turns 解决局部语义：

```text
指代关系：例如“那第二个呢？”
语气变化：用户是否刚刚不耐烦或认可
上下文承接：Speaker 如何自然接话
避免重复：刚刚系统说了什么
```

推荐结构：

```json
{
  "recent_turns": [
    {
      "role": "assistant",
      "text": "你更看重住宿价格、交通便利，还是周边安静？"
    },
    {
      "role": "user",
      "text": "价格和交通最重要。"
    }
  ]
}
```

这里的原则是：

```text
rolling_summary 负责长期记忆；
recent_turns 负责短期上下文；
BeliefState 负责结构化决策变量。
```

三者不能互相替代。

---

## 3.3 Last Action Feedback：上一轮动作反馈

Planner 需要知道用户当前反馈是针对什么产生的。这个信息对 reward credit assignment 很重要。

推荐结构：

```json
{
  "last_action": {
    "action_type": "ask_clarification",
    "target": "travel preference"
  },
  "last_user_feedback": {
    "type": "positive | neutral | negative | explicit_rejection | ignored",
    "target": "clarification | recommendation | ad | topic | whole_response",
    "brief_reason": "用户补充了价格和交通偏好。"
  }
}
```

如果上一轮是广告，而用户说“不想看广告”，那么 Planner 下一步应该知道这是对 `insert_ad` 的负反馈，而不是泛泛地认为用户“不高兴”。

# 4. AdContext：广告上下文

AdContext 单独存在，因为 REC-DIAL 不是普通对话推荐，而是广告支持型系统。

它回答三个问题：

```text
当前有没有合适广告？
最近广告是否过多？
用户对广告态度如何？
```

推荐最小结构：

```json
{
  "recent_ad_load": "none | low | medium | high",
  "last_ad_outcome": "none | accepted | clicked | ignored | rejected",
  "user_ad_attitude": "unknown | tolerant | neutral | resistant",
  "available_ads": [
    {
      "ad_id": "travel_2",
      "topic": "travel accommodation",
      "relevance": "high",
      "value": "medium"
    }
  ]
}
```

其中：

| 字段                 | 含义                  |
| ------------------ | ------------------- |
| `recent_ad_load`   | 最近广告负担是否过重          |
| `last_ad_outcome`  | 上一次广告是否被接受、点击、忽略或拒绝 |
| `user_ad_attitude` | 用户当前对广告的态度估计        |
| `available_ads`    | 当前可选广告及其相关性和价值      |

这里参考 DEAR 的思想：广告决策不能只考虑收益，也要考虑广告对用户体验的影响。REC-DIAL 中的广告“位置”不是推荐列表 slot，而是对话时机与表达方式。([arxiv.org](https://arxiv.org/abs/1909.03602))

# 5. Constraints：硬约束

Constraints 是动作边界，不是 Planner 自己慢慢学出来的偏好。

推荐结构：

```json
{
  "allowed_actions": [
    "guide_exploration",
    "ask_clarification",
    "recommend",
    "soft_ad_bridge",
    "insert_ad",
    "switch_topic",
    "close"
  ],
  "blocked_actions": [],
  "ad_allowed": true,
  "safety_mode": "normal",
  "max_turns_remaining": 6,
  "ad_cooldown_satisfied": true
}
```

典型规则：

```text
如果 sensitive_content = true:
    ad_allowed = false
    blocked_actions 加入 insert_ad。

如果刚刚插过广告:
    ad_cooldown_satisfied = false
    blocked_actions 加入 insert_ad。

如果用户明确拒绝广告:
    短期内 block insert_ad。

如果接近最大轮数:
    提高 close / summarize 的优先级。
```

这样可以避免 Planner 在不合适的情况下输出危险或不合规动作。

# 7. 对应模块链路

最终链路可以写成：

```text
UserResponse_t + RecentTurns + LastAction
        ↓
Monitor
        ↓
MonitorOutput

MonitorOutput + Previous DialogueContext
        ↓
MemoryUpdater / BeliefTracker
        ↓
Updated BeliefState + Updated DialogueContext

CandidatePool + AdLog + BeliefState
        ↓
AdMatcher / CandidateEvaluator
        ↓
Updated AdContext

MonitorOutput + AdLog + Safety Rules + Business Rules
        ↓
ConstraintManager
        ↓
Constraints

BeliefState + DialogueContext + AdContext + Constraints
        ↓
StateBuilder
        ↓
PlannerInput_t
```

这条链路的关键是：

```text
Monitor 只解释当前用户反馈；
MemoryUpdater 维护长期记忆；
AdMatcher 评估广告候选；
ConstraintManager 生成硬约束；
StateBuilder 组装 PlannerInput；
Planner 基于 PlannerInput 输出结构化动作。
```

# 8. Step 2 最终定稿表述

可以直接写进文档：

> REC-DIAL does not use the full dialogue history as the Planner input. Instead, each turn constructs a compact PlannerInput consisting of four parts: BeliefState, DialogueContext, AdContext, and Constraints. BeliefState summarizes the system’s current understanding of the user’s goal, preferences, constraints, and open needs. DialogueContext combines a rolling summary, recent 2–3 exchanges, and feedback to the last action, so that the system keeps both long-term memory and local conversational coherence. AdContext summarizes available ads, ad relevance/value, recent ad load, and user ad attitude. Constraints specify hard action boundaries such as safety restrictions, ad cooldown, maximum turns, and business rules. MonitorOutput only provides per-turn interpretation, and StateBuilder integrates it with memory, ad logs, candidate evaluation, and constraints to form the final PlannerInput.

中文版本：

> REC-DIAL 不直接将完整对话历史作为 Planner 输入，而是在每一轮构造一个紧凑的 PlannerInput。PlannerInput 由四部分组成：BeliefState、DialogueContext、AdContext 和 Constraints。BeliefState 表示系统当前对用户目标、偏好、限制和未解决需求的理解；DialogueContext 由滚动摘要、最近 2–3 轮对话和上一轮动作反馈组成，用于同时保留长期记忆和局部语境；AdContext 总结当前可用广告、广告相关性/价值、近期广告负担以及用户广告态度；Constraints 表示安全限制、广告冷却、最大轮数和业务规则等硬约束。MonitorOutput 只提供当前轮用户反馈的解释，最终由 StateBuilder 将其与对话记忆、广告日志、候选评估和约束信息整合成 PlannerInput。
