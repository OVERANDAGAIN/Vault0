
# Step 2：PlannerState 设计

REC-DIAL 的 PlannerState 应该设计成一个 **decision-oriented belief state**。

也就是说，它不是为了完整记录对话，而是为了回答：

> 在当前局面下，Planner 下一步应该采取哪类高层动作？

---

# 1. PlannerState 的基本定位

REC-DIAL 中，Planner 看到的 state 应该是：

```text
structured belief state
+ compressed dialogue memory
+ recent local context
```

更准确地说：

```text
PlannerInput = PlannerState + MemoryContext + CandidatePool + Constraints
```

其中：

```text
PlannerState:
    决策所需的结构化状态

MemoryContext:
    压缩历史 + 最近几轮原文

CandidatePool:
    当前可选 topic / item / ad candidates

Constraints:
    安全规则、广告频率上限、最大轮数等硬约束
```

**State 描述当前局面，MemoryContext 提供语义依据，CandidatePool 提供可选对象，Constraints 限制动作边界。**

---

# 2. 为什么不能直接用完整 history？

完整 history 有三个问题。

第一，它太长。随着轮数增加，直接把完整对话塞给 Planner 会让输入不稳定，也不适合后续做 RL 训练。

第二，它噪声太多。很多历史句子对当前决策没有直接价值，真正有价值的是用户需求、偏好、拒绝、满意度变化、广告反馈等结构化信息。

第三，它不利于 credit assignment。如果 Planner 每轮都读完整 history，很难判断当前 action 的效果来自哪里。

这也是多数相关工作的共同做法：不是保留完整历史，而是用某种压缩状态表示历史影响。



但是，**完全不保留最近几轮原文也不合适**。因为 REC-DIAL 是自然语言对话，最近几轮有很多局部语义信息，例如指代、省略、语气、刚刚的拒绝、刚刚的广告反应。这些信息很难全部被结构化字段覆盖。

所以我建议采用：

> **structured state + rolling summary + recent K turns** 的混合设计。

---

# 3. 历史应该怎么进入 Planner？

历史信息分三层进入系统。

## 第一层：Structured Belief

这是 PlannerState 的主体，是给 RL / rule-based Planner 用的。

它不保存原文，而保存压缩后的状态变量，例如：

```text
用户需求是否清楚
当前是否有推荐机会
用户是否疲劳
广告负担是否过高
上一步动作是否失败
```

这部分可以向量化，后续可以直接接 rule-based、contextual bandit、DQN 或 policy gradient。

---

## 第二层：Rolling Summary

这是长期历史摘要，用于保留完整对话的长期语义。

它应该维护：

```text
用户当前目标
稳定偏好
明确拒绝
已经推荐过什么
已经解释过什么
用户对广告的历史态度
未解决的问题
不要重复的内容
对话目前处于什么方向
```

Rolling Summary 不是每轮把完整 history 重新总结一遍，而是更合理地做 **增量更新**：

```text
old_summary + latest agent action + latest user response
→ updated_summary
```

这样更稳定，也更容易控制信息丢失。

---

## 第三层：Recent K Turns

最近几轮原文仍然需要保留。建议初版保留最近 **2–3 个 user-agent exchange**。

原因是：

```text
最近几轮保留局部语境；
rolling summary 保留长期语境；
structured state 保留决策变量。
```

Recent K Turns 主要给 LLM Planner 和 Speaker 用，尤其用于：

```text
保持回复连贯；
理解代词和省略；
识别用户刚刚的语气；
避免重复上一句话；
让 Speaker 能自然接话。
```

对纯 RL Planner 来说，recent turns 可以不直接作为高维文本输入，而是通过 embedding 或 Monitor 提取后的结构化变量进入 state。

---

# 4. 最终推荐的 PlannerState 结构

我建议 Step 2 最终采用 **5 个核心状态块**：

```text
1. User Need & Preference Belief
2. Interaction Status
3. Ad Opportunity & Burden
4. Action Feedback Memory
5. Safety & Hard Constraints
```

这五块是按 **决策功能** 分的，不是按代码模块分的。

---

# 5. User Need & Preference Belief

这一块回答：

> 用户现在想要什么？系统对用户需求的理解程度如何？当前是否已经能给出有价值回应？

注意，这里不要设计成传统 CRS 的“信息够不够”。REC-DIAL 不一定要锁定一个唯一 item。用户需求不完整时，也可以做引导式探索、初步建议、方向性推荐，而不是机械追问。

所以我建议不用简单的：

```text
uncertainty_level
task_progress_level
```

而是改成：

```text
need_clarity
preference_confidence
response_readiness
exploration_value
recommendation_opportunity
```

推荐结构：

```json
"user_need_belief": {
  "current_intent": "recommendation | exploration | information_seeking | comparison | casual_chat | complaint | unknown",
  "need_clarity": "low | medium | high",
  "preference_confidence": "low | medium | high",
  "response_readiness": "low | medium | high",
  "exploration_value": "low | medium | high",
  "recommendation_opportunity": "low | medium | high",
  "positive_preferences": [],
  "negative_preferences": [],
  "constraints": {},
  "open_needs": []
}
```

几个变量的含义：

```text
need_clarity:
    用户需求本身是否清楚。

preference_confidence:
    系统对用户偏好的把握是否可靠。

response_readiness:
    当前是否已经可以给出有帮助的回应。

exploration_value:
    继续引导用户表达偏好是否有价值。

recommendation_opportunity:
    当前是否适合进行推荐动作。
```

这样设计后，Planner 不再是：

```text
信息不够 → ask
信息够 → recommend
```

而是：

```text
需求模糊但可引导 → GUIDE_EXPLORATION
需求模糊但可以先帮忙 → PROVIDE_PROVISIONAL_SUGGESTION
偏好比较清楚且机会成熟 → RECOMMEND_ORGANIC
推荐机会高且广告负担低 → SOFT_AD_BRIDGE / INSERT_AD
```

这个设计更符合开放式 REC-DIAL。

---

# 6. Interaction Status

这一块回答：

> 当前对话状态是否适合继续推进、换话题、插广告、修复体验或收尾？

推荐结构：

```json
"interaction_status": {
  "turn_stage": "early | middle | late",
  "topic_stage": "new | developing | saturated",
  "engagement_level": "low | medium | high",
  "satisfaction_level": "low | medium | high",
  "fatigue_level": "low | medium | high",
  "switch_intent_level": "low | medium | high",
  "termination_risk_level": "low | medium | high"
}
```

这部分解决的是对话节奏问题。

例如：

```text
turn_stage = early:
    通常不适合强插广告，更适合理解需求或提供轻量建议。

topic_stage = saturated:
    适合总结、换话题，或做自然过渡。

fatigue_level = high:
    不适合继续追问，也不适合插广告。

satisfaction_level = low:
    优先 repair，而不是推荐或广告。
```

SAPIENT 这类多轮 CRS planning 工作强调，单纯 greedy action selection 可能导致非最优的多轮规划，因此状态里必须能反映对话阶段和长期规划风险，而不只是当前能否推荐。([ACL Anthology][2])

---

# 7. Ad Opportunity & Burden

这是 REC-DIAL 相对普通 CRS 的关键扩展。

这一块回答：

> 当前是不是一个合适的广告时机？如果插广告，收益和体验风险如何？

推荐结构：

```json
"ad_opportunity_burden": {
  "ad_candidate_relevance": "none | low | medium | high",
  "ad_value_level": "none | low | medium | high",
  "recent_ad_load": "low | medium | high",
  "ad_resistance_level": "low | medium | high",
  "ad_cooldown_status": "satisfied | not_satisfied",
  "last_ad_outcome": "none | accepted | clicked | ignored | rejected"
}
```

这里有两个核心概念：

```text
Ad Opportunity:
    有没有高相关、高价值、适合当前话题的广告。

Ad Burden:
    用户最近是否已经被广告打扰，是否表现出反感，是否需要冷却。
```

DEAR 的关键思想就是广告插入不能只看收益，还要考虑广告对用户体验的负面影响；它把在线广告决策拆成是否插广告、插哪个广告、插到哪个位置。REC-DIAL 中没有推荐列表 slot，但对应的是“是否插广告、插哪个广告、在什么对话时机和话术下插入”。([arXiv][3])

所以 state 中必须有广告机会和广告负担，而不是只记录 `ads_seen`。

---

# 8. Action Feedback Memory

这一块回答：

> 最近系统做了什么？用户反馈如何？有没有需要避免重复或修复的地方？

推荐结构：

```json
"action_feedback_memory": {
  "last_action_type": "guide_exploration | ask_clarification | provisional_suggestion | recommend_organic | soft_ad_bridge | insert_ad | repair | switch_topic | close",
  "last_target_type": "none | topic | item | ad",
  "last_feedback_type": "positive | neutral | negative | explicit_rejection | ignored",
  "recent_failure_count": 0,
  "recent_repetition_level": "low | medium | high",
  "recent_positive_signal": true
}
```

这部分不是完整 history，而是为 credit assignment 服务。

例如：

```text
last_action_type = insert_ad
last_feedback_type = explicit_rejection
```

这意味着下一步应该降低广告动作概率，可能需要 repair 或回到用户任务。

如果没有这部分，Planner 只知道用户当前情绪不好，但不知道不满来自广告、推荐失败、话题切换还是重复表达。

---

# 9. Safety & Hard Constraints

这一块回答：

> 当前有哪些动作不能做，或者必须保守处理？

推荐结构：

```json
"safety_constraints": {
  "sensitive_content_flag": false,
  "ad_allowed": true,
  "must_avoid_topics": [],
  "policy_constraint_type": "none | no_ad | safe_response | avoid_recommendation"
}
```

这部分应该主要作为 hard constraint，而不是让 RL 自己学。

例如：

```text
ad_allowed = false
→ Planner 不能输出 INSERT_AD。

policy_constraint_type = safe_response
→ Planner 必须选择安全处理类动作。
```

---

# 10. MemoryContext：历史摘要和最近几轮如何设计

除了 structured PlannerState，我建议显式保留一个 `MemoryContext`。它不一定全部进入 RL state vector，但应该作为 Planner / Speaker 的上下文输入。

推荐结构：

```json
"memory_context": {
  "rolling_summary": {
    "user_goal_summary": "...",
    "stable_preferences": [],
    "explicit_rejections": [],
    "resolved_points": [],
    "unresolved_points": [],
    "ad_feedback_summary": "...",
    "do_not_repeat": []
  },
  "recent_turns": [
    {
      "speaker": "user",
      "text": "..."
    },
    {
      "speaker": "assistant",
      "text": "..."
    }
  ]
}
```

## rolling_summary 应该总结什么？

重点不是总结所有句子，而是总结 **对未来决策有影响的信息**：

```text
用户目标：
    用户当前主要想解决什么问题。

稳定偏好：
    用户反复表达或明确认可的偏好。

明确拒绝：
    用户不想要什么、反感什么。

已解决内容：
    哪些问题已经回答过，避免重复。

未解决内容：
    还有哪些需求没有覆盖。

广告反馈：
    用户是否接受、忽略、拒绝过广告。

禁止重复：
    哪些建议或广告已经给过，不要再来。
```

## recent_turns 保留几轮？

初版建议保留最近 **2–3 个 exchange**，也就是最近 4–6 条 utterance。

原因是：

```text
1 个 exchange 太少，容易丢失局部语境；
超过 3 个 exchange 对 Planner 来说通常冗余；
完整 history 又太长，不适合稳定训练。
```

可以设成超参数：

```text
recent_k_exchange = 2 或 3
```

后续实验中再调。

## rolling_summary 和 recent_turns 的关系

两者不是替代关系。

```text
rolling_summary:
    负责长期记忆。

recent_turns:
    负责局部连贯性和语气。
```

例如用户说：

```text
“那第二个呢？”
```

这句话只有放在 recent turns 里才知道“第二个”指什么。Rolling summary 很可能会丢掉这种局部指代。

但如果用户五轮前说：

```text
“我不想看太贵的。”
```

这应该进入 rolling summary，而不依赖 recent turns。

# 12. 这个 state 如何串联其他模块？

## Monitor → PlannerState

Monitor 不直接输出决策，而是从用户回复中提取或估计：

```text
satisfaction_level
engagement_level
fatigue_level
switch_intent_level
ad_resistance_level
sensitive_content_flag
last_feedback_type
```

Monitor 的输出进入：

```text
interaction_status
ad_opportunity_burden
action_feedback_memory
safety_constraints
```

---

## Memory / BeliefTracker → PlannerState

Memory 或 BeliefTracker 负责把历史更新成：

```text
user_need_belief
rolling_summary
recent_turns
```

它应该处理：

```text
用户新表达的偏好；
用户新拒绝的内容；
当前需求是否更清楚；
是否出现新的 open needs；
哪些内容已解决。
```

---

## CandidateRetriever / AdMatcher → PlannerInput

候选池不属于 state 本体，但它和 state 一起传给 Planner。

例如：

```json
"candidate_pool": {
  "topic_candidates": [],
  "organic_candidates": [],
  "ad_candidates": []
}
```

然后 state 里只放摘要：

```text
ad_candidate_relevance
ad_value_level
recommendation_opportunity
```

---

## Planner → Speaker

Planner 不输出自然语言，而输出结构化 action。

Speaker 接收：

```text
PlannerAction
+ memory_context.recent_turns
+ rolling_summary
+ style constraints
```

这样 Speaker 能自然表达，但高层决策仍然可控。

---

## RewardCalculator ← State / Action / User Feedback

RewardCalculator 使用：

```text
previous planner_state
planner_action
agent utterance
user response
next planner_state
events
```

来计算 reward。

例如：

```text
如果 action = insert_ad
并且 last_feedback_type = explicit_rejection
→ ad penalty

如果 action = guide_exploration
并且 need_clarity 从 low 变 medium
→ exploration reward

如果 action = recommend_organic
并且 satisfaction_level 上升
→ user value reward
```

这说明 state 设计也直接支持 reward credit assignment。

# Step 2 最终结论

REC-DIAL 的 state 设计不应照搬传统 CRS 的“信息收集到足够 → 推荐”逻辑。更合理的是把 PlannerState 设计成 **下一步动作选择的 readiness state**。

它包括五类核心信息：

```text
1. User Need & Preference Belief
2. Interaction Status
3. Ad Opportunity & Burden
4. Action Feedback Memory
5. Safety & Hard Constraints
```

历史信息不直接完整进入 state，而是拆成：

```text
rolling summary:
    保留长期决策记忆

recent K turns:
    保留局部语境和自然语言连贯性

structured state:
    保留可训练、可决策、可解释的状态变量
```

这样设计后，REC-DIAL 的 Planner 不再只是判断 ask/recommend，而是可以在以下动作之间进行长期收益权衡：

```text
guide exploration
ask light clarification
provide provisional suggestion
recommend organic content
soft ad bridge
insert ad
repair experience
switch topic
summarize / close
```

这比传统 slot-filling CRS 更适合 REC-DIAL 的开放式、多目标、广告支持型对话推荐设定。
