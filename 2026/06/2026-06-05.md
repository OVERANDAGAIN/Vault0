# Step 2：PlannerInput / State Formulation

## 2.1 设计目标

在 REC-DIAL 中，Planner 的输入不应是完整对话历史，也不应等同于 Monitor 的直接输出。完整历史过长且噪声较多，MonitorOutput 也只是对当前用户反馈的局部解释，二者都不能直接作为稳定的决策状态。

因此，我们将 Planner 的输入定义为一个经过压缩和结构化后的决策上下文：

$$I_t = (B_t, D_t, A_t, K_t)$$

其中：

| 符号    | 名称              | 含义                              |
| ----- | --------------- | ------------------------------- |
| $B_t$ | BeliefState     | 系统当前对用户目标、偏好、约束和未解决需求的理解        |
| $D_t$ | DialogueContext | 对话历史的压缩表示，包括长期摘要、最近几轮原文和上一轮动作反馈 |
| $A_t$ | AdContext       | 当前广告候选、广告负担、用户广告态度和历史广告反馈       |
| $K_t$ | Constraints     | 安全、广告频率、最大轮数和业务规则形成的硬约束         |

这里的 $I_t$ 可以视为 Planner 在第 $t$ 个 decision turn 的输入。若后续训练 RL policy，则可以将其进一步向量化为：

$$s_t = \phi(I_t)$$

其中 $\phi(\cdot)$ 是状态编码函数。对于 rule-based Planner 或 LLM Planner，则可以直接使用结构化 JSON 形式的 $I_t$。

---

## 2.2 PlannerInput 与 MonitorOutput 的关系

Monitor 的职责不是直接生成 PlannerInput，而是解释当前用户反馈。记当前用户回复为 $u_t$，最近 $K$ 轮对话为 $R_t^K$，上一轮 Planner 动作为 $a_{t-1}$，则 Monitor 输出为：

$$m_t = M(u_t, R_t^K, a_{t-1})$$

其中 $m_t$ 包含本轮用户反馈的结构化解释，例如：

$$m_t = (
\text{intent}_t,
\Delta {P_t}^+,
\Delta {P_t}^-,
\Delta C_t,
\text{feedback}_{t-1},
\text{ad_attitude}_t,
\text{safety}_t
)$$

具体含义如下：

| 变量                       | 含义                             |
| ------------------------ | ------------------------------ |
| $\text{intent}_t$        | 当前用户意图，例如探索、询问信息、寻求推荐、比较、抱怨、闲聊 |
| $\Delta P_t^+$           | 本轮新增的正向偏好                      |
| $\Delta P_t^-$           | 本轮新增的负向偏好或拒绝                   |
| $\Delta C_t$             | 本轮新增的约束条件，例如预算、地点、时间、类别        |
| $\text{feedback}_{t-1}$  | 用户对上一轮系统动作的反馈                  |
| $\text{ad-attitude}_{t}$ | 用户当前对广告的态度估计                   |
| $\text{safety}_t$        | 敏感内容或安全约束信号                    |
|                          |                                |

因此，MonitorOutput 是 **per-turn feedback interpretation**，而不是完整 state。最终 PlannerInput 由 StateBuilder 统一构造：

$$I_t = \text{StateBuilder}(m_t, B_{t-1}, D_{t-1}, A_{t-1}, K_{t-1}, \mathcal{C}_t)$$

其中 $\mathcal{C}_t$ 表示当前候选集合，包括 topic candidates、organic recommendation candidates 和 ad candidates。候选集合通常不作为核心 state 的一部分，而是作为动作目标选择的辅助输入。

---

## 2.3 BeliefState ($B_t$)

BeliefState 表示系统当前对用户需求和偏好的理解。它不是完整用户画像，也不是完整对话历史，而是对用户当前任务状态的压缩表示。

我们定义：

$$B_t = (i_t, g_t, P_t^+, P_t^-, C_t, Q_t)$$

其中：

| 符号      | 名称                   | 含义                   |
| ------- | -------------------- | -------------------- |
| $i_t$   | current_intent       | 当前用户意图               |
| $g_t$   | user_goal            | 当前用户目标的简短描述          |
| $P_t^+$ | positive_preferences | 用户已表达的正向偏好           |
| $P_t^-$ | negative_preferences | 用户已表达的负向偏好、拒绝或不想要的内容 |
| $C_t$   | constraints          | 用户给出的约束条件            |
| $Q_t$   | open_questions       | 尚未解决、但可能影响后续回复的问题    |

示例：

```json
{
  "current_intent": "seek_recommendation",
  "user_goal": "寻找预算友好的日本旅行住宿建议",
  "positive_preferences": ["budget-friendly", "near public transportation"],
  "negative_preferences": ["luxury hotels", "crowded areas"],
  "constraints": {
    "location": "Japan",
    "budget": "low"
  },
  "open_questions": ["具体城市", "旅行时间", "住宿类型"]
}
```

BeliefState 的更新由 MonitorOutput 中的 preference update 驱动：

$$B_t = U_B(B_{t-1}, m_t)$$

其中 $U_B$ 是 BeliefUpdater。它负责将当前用户回复中的新增偏好、拒绝、约束和未解决需求合并到已有 belief 中。

这里需要强调：BeliefState 并不是为了判断“信息是否足够从而推荐唯一 item”。REC-DIAL 是开放式对话推荐系统，不是传统 slot-filling 任务。BeliefState 的作用是帮助 Planner 判断当前更适合采取哪类高层动作，例如引导探索、轻量澄清、给出初步建议、正式推荐、插入广告、修复体验或结束对话。

---

## 2.4 DialogueContext ($D_t$)

由于完整 dialogue history 不适合作为 Planner 输入，我们将历史信息压缩为 DialogueContext：

$$D_t = (S_t, R_t^K, F_{t-1})$$

其中：

| 符号        | 名称                   | 含义                    |
| --------- | -------------------- | --------------------- |
| $S_t$     | rolling_summary      | 对长期历史的压缩摘要            |
| $R_t^K$   | recent_turns         | 最近 $K$ 个 exchange 的原文 |
| $F_{t-1}$ | last_action_feedback | 上一轮系统动作及用户反馈          |

### 2.4.1 Rolling Summary ($S_t$)

Rolling summary 保存长期有效的对话信息，而不是逐句复述历史。它应当只保留对未来决策有影响的信息，包括：

$$S_t =
(
\text{goal_summary},
\text{stable_preferences},
\text{explicit_rejections},
\text{resolved_points},
\text{unresolved_points},
\text{ad_feedback_summary},
\text{do_not_repeat}
)$$

具体来说：

| 字段                    | 作用                  |
| --------------------- | ------------------- |
| `goal_summary`        | 当前用户目标的简短总结         |
| `stable_preferences`  | 用户较稳定、反复表达或明确认可的偏好  |
| `explicit_rejections` | 用户明确拒绝的内容           |
| `resolved_points`     | 已经回答或解决的需求          |
| `unresolved_points`   | 仍然没有解决的问题           |
| `ad_feedback_summary` | 用户历史上对广告的接受、忽略或拒绝情况 |
| `do_not_repeat`       | 已经推荐过或用户不希望重复的内容    |

Rolling summary 采用增量更新，而不是每轮重新总结完整历史：

$$S_t = U_S(S_{t-1}, a_{t-1}, y_{t-1}, u_t, m_t)$$

其中：

| 符号        | 含义                  |
| --------- | ------------------- |
| $S_{t-1}$ | 上一轮 rolling summary |
| $a_{t-1}$ | 上一轮 PlannerAction   |
| $y_{t-1}$ | 上一轮系统自然语言回复         |
| $u_t$     | 当前用户回复              |
| $m_t$     | MonitorOutput       |

更新规则可以描述为：

* 用户新增偏好时，加入 `stable_preferences` 或 `constraints`。
* 用户明确拒绝某类内容时，加入 `explicit_rejections` 和 `do_not_repeat`。
* 某个问题已经被回答且用户接受时，从 `unresolved_points` 移到 `resolved_points`。
* 用户继续追问某个问题时，该问题保留在 `unresolved_points`。
* 系统展示广告后，根据用户接受、点击、忽略或拒绝更新 `ad_feedback_summary`。
* 系统已经推荐过的 topic、item 或 ad 加入 `do_not_repeat`，防止重复。

示例：

```json
{
  "rolling_summary": "用户正在寻找预算友好的日本旅行住宿建议，偏好交通方便，不想要豪华酒店和人太多的区域。",
  "resolved_points": ["已确认预算敏感", "已确认交通便利重要"],
  "unresolved_points": ["还未确定具体城市", "还未确定旅行时间", "还未确定住宿类型"],
  "do_not_repeat": ["豪华酒店推荐"],
  "ad_feedback_summary": "尚未展示广告，用户没有表达广告反感。"
}
```

### 2.4.2 Recent Turns ($R_t^K$)

Rolling summary 保存长期信息，但它不能完全替代最近几轮原文。自然语言对话中存在指代、省略、语气和局部承接问题，例如“那第二个呢？”这类回复需要依赖最近上下文理解。

因此，我们保留最近 $K$ 个 exchange：

$$R_t^K = \text{Tail}_K(H_t)$$

其中 $H_t$ 是截至当前的对话历史，$K$ 是超参数。初版建议：

$$K = 2 \text{ 或 } 3$$

即保留最近 2–3 个 user-agent exchange，约等于 4–6 条 utterance。

Recent turns 的作用是：

* 保留局部语义和指代关系。
* 帮助 Speaker 自然接话。
* 帮助 Monitor 判断用户当前语气和反馈对象。
* 避免系统重复刚刚说过的内容。

### 2.4.3 Last Action Feedback ($F_{t-1}$)

Last action feedback 记录上一轮 PlannerAction 以及用户对该动作的反馈：

$$F_{t-1} = (a_{t-1}, \text{feedback}_{t-1})$$

示例：

```json
{
  "last_action": {
    "action_type": "ask_clarification",
    "target": "travel preference"
  },
  "last_user_feedback": {
    "type": "positive",
    "target": "clarification",
    "brief_reason": "用户补充了价格和交通偏好。"
  }
}
```

这部分用于支持 credit assignment。比如上一轮 action 是 `insert_ad`，而当前用户明确拒绝广告，那么系统可以把负反馈归因到上一轮广告动作，而不是简单地认为用户整体不满意。

## 2.5 AdContext ($A_t$)

AdContext 是 REC-DIAL 相比普通对话推荐系统必须额外建模的部分。它表示当前广告机会和广告负担。

定义：

$$A_t = (L_t^{ad}, O_t^{ad}, \alpha_t^{ad}, \mathcal{A}_t)$$

其中：

| 符号              | 名称               | 含义       |
| --------------- | ---------------- | -------- |
| $L_t^{ad}$      | recent_ad_load   | 最近广告负担   |
| $O_t^{ad}$      | last_ad_outcome  | 上一次广告结果  |
| $\alpha_t^{ad}$ | user_ad_attitude | 用户当前广告态度 |
| $\mathcal{A}_t$ | available_ads    | 当前可用广告候选 |

示例：

```json
{
  "recent_ad_load": "none",
  "last_ad_outcome": "none",
  "user_ad_attitude": "neutral",
  "available_ads": [
    {
      "ad_id": "travel_2",
      "topic": "travel accommodation",
      "relevance": "high",
      "value": "medium"
    }
  ]
}
```

AdContext 回答三个问题：

1. 当前是否有和用户需求相关的广告？
2. 最近广告是否过多，是否会造成用户负担？
3. 用户当前是否表现出广告接受、忽略或抵触？

这里可以参考广告插入类 RL 工作的思想：广告决策不能只最大化广告收益，也要考虑广告对用户体验的负面影响。对 REC-DIAL 而言，广告“位置”不再是推荐列表中的 slot，而是对话中的时机和表达方式。

---

## 2.6 Constraints ($K_t$)

Constraints 表示当前动作空间中的硬约束。它不是用户偏好，也不是 reward，而是 Planner 在当前轮必须遵守的边界条件。

定义：

$$K_t = (\mathcal{U}_t, \mathcal{B}_t, \eta_t^{ad}, \rho_t, n_t, c_t^{ad})$$

其中：

| 符号              | 名称                    | 含义        |
| --------------- | --------------------- | --------- |
| $\mathcal{U}_t$ | allowed_actions       | 当前允许的动作集合 |
| $\mathcal{B}_t$ | blocked_actions       | 当前禁止的动作集合 |
| $\eta_t^{ad}$   | ad_allowed            | 当前是否允许广告  |
| $\rho_t$        | safety_mode           | 当前安全模式    |
| $n_t$           | max_turns_remaining   | 剩余最大轮数    |
| $c_t^{ad}$      | ad_cooldown_satisfied | 广告冷却是否满足  |

示例：

```json
{
  "allowed_actions": [
    "guide_exploration",
    "ask_clarification",
    "recommend",
    "soft_ad_bridge",
    "insert_ad",
    "switch_topic",
    "close"
  ],
  "blocked_actions": [],
  "ad_allowed": true,
  "safety_mode": "normal",
  "max_turns_remaining": 6,
  "ad_cooldown_satisfied": true
}
```

典型约束规则包括：

* 若当前内容敏感，则 `ad_allowed = false`，并将 `insert_ad` 加入 blocked actions。
* 若刚刚展示过广告，则 `ad_cooldown_satisfied = false`，短期内禁止继续插广告。
* 若用户明确拒绝广告，则短期内禁止或降低广告动作优先级。
* 若接近最大轮数，则提高 `close` 或 `summarize` 类动作的合理性。
* 若某类 topic 被用户明确拒绝，则该 topic 不应再次作为目标候选。

## 2.8 模块串联

完整的状态构造流程可以写成：

$$m_t = M(u_t, R_t^K, a_{t-1})$$

$$B_t = U_B(B_{t-1}, m_t)$$

$$D_t = U_D(D_{t-1}, a_{t-1}, y_{t-1}, u_t, m_t)$$

$$A_t = U_A(\text{AdLog}_{t-1}, \mathcal{A}_t, B_t, m_t)$$

$$K_t = U_K(\text{rules}, \text{safety}_t, \text{AdLog}_{t-1}, t)$$

$$I_t = (B_t, D_t, A_t, K_t)$$

其中：

| 模块                           | 职责                                      |
| ---------------------------- | --------------------------------------- |
| Monitor ($M$)                  | 解释当前用户反馈                                |
| BeliefUpdater ($U_B$)          | 更新用户需求和偏好 belief                        |
| DialogueContextUpdater ($U_D$) | 更新历史摘要、recent turns 和上一轮反馈              |
| AdContextUpdater ($U_A$)       | 更新广告候选、广告负担和广告态度                        |
| ConstraintManager ($U_K$)      | 生成动作约束和安全边界                             |
| StateBuilder                     | 将 $(B_t, D_t, A_t, K_t)$ 组装成 PlannerInput |

最终，Planner 根据 $I_t$ 选择结构化动作：

$$a_t \sim \pi(a \mid I_t)$$

或在确定性 baseline 中：

$$a_t = \pi(I_t)$$

## 2.9 本节最终表述

可以在文档中这样总结：

> REC-DIAL does not directly use the full dialogue history as the Planner input. Instead, at each agent decision turn, the system constructs a compact PlannerInput $I_t = (B_t, D_t, A_t, K_t)$. $B_t$ denotes the current belief about the user’s goal, preferences, constraints, and open needs. $D_t$ represents the compressed dialogue context, including a rolling summary, recent $K$ exchanges, and feedback to the previous action. $A_t$ captures the advertising context, including recent ad load, last ad outcome, user ad attitude, and available ad candidates. $K_t$ specifies hard constraints such as safety restrictions, ad cooldown, and remaining turns. MonitorOutput only provides per-turn interpretation of the latest user response; the final PlannerInput is constructed by integrating MonitorOutput with memory updates, ad context, and constraints.

中文版本：

> REC-DIAL 不直接将完整对话历史作为 Planner 输入，而是在每个 agent decision turn 构造一个紧凑的 PlannerInput，记为 $I_t = (B_t, D_t, A_t, K_t)$。其中，$B_t$ 表示系统当前对用户目标、偏好、约束和未解决需求的 belief；$D_t$ 表示压缩后的对话上下文，包括 rolling summary、最近 $K$ 轮 exchange 和上一轮动作反馈；$A_t$ 表示广告上下文，包括近期广告负担、上一次广告结果、用户广告态度和当前可用广告候选；$K_t$ 表示硬约束，包括安全限制、广告冷却和剩余轮数等。MonitorOutput 只提供当前用户回复的本轮解释，最终 PlannerInput 由 StateBuilder 将 MonitorOutput、对话记忆、广告上下文和约束信息整合得到。
