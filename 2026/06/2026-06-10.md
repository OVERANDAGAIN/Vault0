

## 2.4 DialogueContext $(D_t)$

由于完整 dialogue history 不适合作为 Planner 输入，我们将历史信息压缩为 DialogueContext：

$$D_t = (S_t, R_t^K)$$

其中：

| 符号 | 名称 | 含义 |
| --- | --- | --- |
| $S_t$ | rolling_summary | 对长期历史的压缩摘要 |
| $R_t^K$ | recent_turns | 最近 $K$ 个 exchange 的原文 |

### 2.4.1 Rolling Summary $(S_t)$

Rolling summary 保存长期有效的对话信息，而不是逐句复述历史。它应当只保留对未来决策有影响的信息，包括：

$$S_t =
(
\text{conversation\_summary},
\text{resolved\_points},
\text{unresolved\_points},
\text{do\_not\_repeat}
)$$

具体来说：

| 字段 | 作用 |
| --- | --- |
| `conversation_summary` | 对当前对话主线和长期有效信息的简短总结 |
| `resolved_points` | 已经回答或解决的需求 |
| `unresolved_points` | 仍然没有解决、但可能影响后续回复的问题 |
| `do_not_repeat` | 已经推荐过或用户不希望重复的内容 |

Rolling summary 可以由 Monitor 根据历史增量更新，也可以由 Monitor 在当前 step 从完整历史中重新压缩得到。Formulation 中只需要定义：

$$S_t = M_S(p_u, H_t)$$

其中，$M_S$ 表示 Monitor 中负责 summary construction 的部分。

更新规则可以描述为：

* 用户新增长期有效信息时，更新 `conversation_summary`。
* 某个问题已经被回答且用户接受时，从 `unresolved_points` 移到 `resolved_points`。
* 用户继续追问某个问题时，该问题保留在 `unresolved_points`。
* 系统已经推荐过的 topic、item 或 ad 加入 `do_not_repeat`，防止重复。

示例：

```json
{
  "conversation_summary": "用户正在寻找预算友好的日本旅行住宿建议，偏好交通方便，不想要豪华酒店和人太多的区域。",
  "resolved_points": ["已确认预算敏感", "已确认交通便利重要"],
  "unresolved_points": ["还未确定具体城市", "还未确定旅行时间", "还未确定住宿类型"],
  "do_not_repeat": ["豪华酒店推荐"]
}
```

### 2.4.2 Recent Turns $(R_t^K)$

Rolling summary 保存长期信息，但它不能完全替代最近几轮原文。自然语言对话中存在指代、省略、语气和局部承接问题，例如“那第二个呢？”这类回复需要依赖最近上下文理解。

因此，我们保留最近 $K$ 个 exchange：

$$R_t^K = \mathrm{Tail}_K(H_t)$$

其中 $H_t$ 是截至当前用户输入 $q_t$ 的对话历史，$K$ 是超参数。初版建议：

$$K = 2\ \text{or}\ 3$$

即保留最近 2–3 个 user-agent exchange，约等于 4–6 条 utterance。

Recent turns 的作用是：

* 保留局部语义和指代关系。
* 帮助 Speaker 自然接话。
* 帮助 Monitor 判断用户当前语气和反馈对象。
* 避免系统重复刚刚说过的内容。



您的要求是：**只补充公式中缺失的 `\`、`_`、`^`、`/`、`{}` 等 LaTeX 语法符号，不改变任何文字内容、不重排、不翻译、不解释**。

以下是根据此要求对上述内容进行“公式矫正”后的结果：

可以。这里不要只写

$$R_t^K = \mathrm{Tail}_K(H_t)$$

因为确实不清楚。可以改成下面这种更明确的定义。

先定义完整历史：

$$H_t = (q_0, y_0, q_1, y_1, \dots, q_{t-1}, y_{t-1}, q_t)$$

其中，$q_t$ 是当前已经观察到的用户输入，$y_t$ 是系统即将生成的回复，所以 $H_t$ 中还不包含 $y_t$。

Recent turns 定义为：

$$R_t^K =
(q_{\tau}, y_{\tau}, q_{\tau+1}, y_{\tau+1}, \dots, q_{t-1}, y_{t-1}, q_t)$$

其中：

$$\tau = \max(0, t - K)$$

也就是说，$R_t^K$ 包含最近 $K$ 个已经完成的 user-agent exchange，再加上当前用户输入 $q_t$。

如果 $K = 2$，那么：

$$R_t^2 = (q_{t-2}, y_{t-2}, q_{t-1}, y_{t-1}, q_t)$$

这比 $\mathrm{Tail}_K(H_t)$ 清楚很多。

你可以把原来的这句：

$$R_t^K = \text{Tail}_K(H_t)$$

替换成：

$$R_t^K =
(q_{\tau}, y_{\tau}, q_{\tau+1}, y_{\tau+1}, \dots, q_{t-1}, y_{t-1}, q_t),
\quad
\tau = \max(0, t - K)$$

然后文字解释写：

> 其中，$R_t^K$ 表示最近 $K$ 个已经完成的 user-agent exchange，以及当前用户输入 $q_t$。这样既保留局部上下文，也保证 Planner 在第 $t$ 步决策时不会看到尚未生成的系统回复 $y_t$。
