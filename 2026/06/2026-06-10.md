从 LLM 长对话记忆文献看，比较稳的抽象是：

$$D_t = (m_t, h_t^K)$$

其中：

| 符号 | 含义 | 对应文献思路 |
| --- | --- | --- |
| $m_t$ | long-term dialogue memory / rolling summary | 递归摘要、长期记忆、memory update |
| $h_t^K$ | recent history window / local context | 当前 context window / 最近上下文 |

这个拆法有依据：recursive summarization 方法会用 previous memory 和 following context 递归生成新的 memory；MemoryBank 类方法维护长期 memory 并持续更新；MemGPT 则强调有限 context window 下需要把长期 memory 和当前 working context 区分开。([arXiv][1])

## 1. $h_t^K$：recent history window

这个最简单。它不是总结，而是**最近几轮原文**。

完整历史定义为：

$$H_t = (q_0, y_0, q_1, y_1, \dots, q_{t-1}, y_{t-1}, q_t)$$

其中 $q_t$ 是当前用户输入，$y_t$ 还没有生成。

令：

$$\tau = \max(0, t - K)$$

则最近窗口定义为：

$$h_t^K = (q_{\tau}, y_{\tau}, q_{\tau+1}, y_{\tau+1}, \dots, q_{t-1}, y_{t-1}, q_t)$$

也就是说，$h_t^K$ 包含：

```text
最近 K 个已完成 user-agent exchange
+
当前用户输入 q_t
```

这个设计对应 LLM 里的 **working context / context window**：保留最近原文，用来处理指代、省略、语气、局部承接关系。MemGPT 这类工作强调 fixed context length 是限制，因此需要管理当前 context 与长期 memory 的关系。([arXiv][2])

---

## 2. $m_t$：rolling dialogue memory


$$m_t = M_{\mathrm{mem}}(m_{t-1}, \Delta H_t)$$

其中：

$$\Delta H_t = (y_{t-1}, q_t)$$

也就是：

$$\text{previous memory} + \text{latest interaction} \rightarrow \text{updated memory}$$

这和 recursive summarization 文献中的做法一致：先让 LLM 记住小段 dialogue context，然后用 previous memory 和 following context 递归生成新的 memory。([arXiv][1])

在 REC-DIAL 里，可以把 $m_t$ 表示为一个固定 schema：

$$m_t =
(
\text{brief}_t,
\text{resolved}_t,
\text{pending}_t,
\text{avoid}_t
)$$

其中：

| 字段 | 含义 |
| --- | --- |
| $\text{brief}_t$ | 对当前对话主线的简短总结 |
| $\text{resolved}_t$ | 已经回答、确认或完成的事项 |
| $\text{pending}_t$ | 尚未解决、仍影响后续回复的问题 |
| $\text{avoid}_t$ | 已推荐过、用户拒绝过或不应重复的内容 |

这样 $m_t$ 的作用不是重新表示 $B_t$ 里的用户偏好，也不是表示 $E_t$ 里的广告反馈，而是记录**对话进展**。

---

## 3. $m_t$ 的更新方式

可以写成三步：

$$e_t^{\text{mem}} = \mathrm{Extract}_{\text{mem}}(\Delta H_t)$$

$$\tilde{m}_t = \mathrm{Merge}(m_{t-1}, e_t^{\text{mem}})$$

$$m_t = \mathrm{Compress}(\tilde{m}_t)$$

含义是：

| 操作 | 作用 |
| --- | --- |
| $\mathrm{Extract}_{\text{mem}}$ | 从最新交互中抽取长期有用的信息 |
| $\mathrm{Merge}$ | 和上一轮 memory 合并，更新 resolved / pending / avoid |
| $\mathrm{Compress}$ | 删除冗余、过期、局部细节，保持 memory 简短 |

这比“每轮重新总结完整 $H_t$”更像 rolling memory，也和长对话记忆文献的递归摘要思路一致。MemoryBank 和 Generative Agents 也都强调 memory 不是简单日志，而是需要持续更新、检索或反思得到对后续行为有用的信息。([AAAI Publications][3])

---

## 4. 对我们当前 formulation 的建议

因此，$D_t$ 可以定为：

$$D_t = (m_t, h_t^K)$$

其中：

$$m_t = M_{\mathrm{mem}}(m_{t-1}, y_{t-1}, q_t)$$

$$h_t^K = (q_{\tau}, y_{\tau}, \dots, q_{t-1}, y_{t-1}, q_t),
\quad
\tau = \max(0, t - K)$$

对应文字：

> $m_t$ 表示由 Monitor 维护的长期 dialogue memory，用于记录对话主线、已解决事项、未解决事项和不要重复的内容；$h_t^K$ 表示最近 $K$ 个 exchange 以及当前用户输入，用于保留局部上下文。前者是压缩记忆，后者是原文窗口。
