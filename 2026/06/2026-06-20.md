# REC-DIAL formulation 代码实现

## Scope

本轮主要把代码改造成当前 REC-DIAL formulation 对应的 pipeline：

```text
Monitor:         (p_u, H_t) -> PlannerState s_t
Planner:         s_t -> 一维 PlannerAction a_t
Speaker:         (s_t, a_t) -> AgentUtterance y_t
UserSimulator:   y_t -> UserResponse q_{t+1}
Reward:          (s_t, a_t, s_{t+1}) -> scalar reward r_t
Metrics:         基于 EpisodeLog 独立统计
```

---

# 修改 / 新增文件

## `src/recdiag/core/types.py`

* 新增或规范了 `PlannerAction`、`AdEvent`、`RewardBreakdown`、`TrajectoryStep`、`EpisodeLog`。
* 用于记录每一步 transition：`q_t, s_t, a_t, y_t, q_{t+1}, r_t, s_{t+1}`。

## `src/recdiag/core/state.py`

* 新增文件。
* 定义了 `BeliefState`、`DialogueContext`、`PlannerState`。
* 用 `PlannerState` 表示 planner 输入状态：

```text
s_t = (p_u, B_t, D_t)
```

## `src/recdiag/core/profile.py`

* 明确区分静态用户画像和模拟器参数。
* `UserProfile` 表示用户 persona / interests / preferences。
* `SimParams` 表示 patience、ad sensitivity、topic depth、max turns 等 simulator-only 参数。

## `src/recdiag/core/monitor.py`

* 按 formulation 重写为：

```python
build_state(profile, history, episode_log) -> PlannerState
```

* Monitor 只负责构造 state，不再负责广告或 topic 决策。

---

## `src/recdiag/core/planner.py`

* 将 planner 接口统一为：

```python
planner.act(state, action_space) -> PlannerAction
```

* 保留原有 `RandomPlanner`、`RuleBasedPlanner`、`LLMPlanner`。
* 新增 baseline planner：`NoAdPlanner`、`RandomAdNoopPlanner`、`GreedyAdPlanner`、`ConservativeAdPlanner`。

---

## `src/recdiag/core/speaker.py`
* 将 speaker 明确为 action executor。
* Speaker 根据 planner 已选定的 action 生成系统回复。

---

## `src/recdiag/core/simulators/profile_simulator.py`

* 保留原有 profile-based user simulator 逻辑。
* 调整为只生成用户反馈 `q_{t+1}`。
* 用户反馈 metadata，例如满意度、广告反应、退出原因等。

---

## `src/recdiag/env/dialogue_env.py`

* 重写为 formulation transition environment。
* 核心接口是：

```python
reset() -> PlannerState
step(action) -> next_state, reward, done, info
```

* 每一步执行完整 transition：

```text
s_t -> a_t -> y_t -> q_{t+1} -> s_{t+1}, r_t
```

# JSON 输出结构

## 1. Batch-level：记录实验设置和总体指标

JSON 顶层首先记录本次实验配置：

```json
"config": {
  "num_episodes": 1,
  "monitor_strategy": "llm",
  "planner_strategy": "llm",
  "speaker_strategy": "llm",
  "reward_judge_strategy": "llm",
  "user_response_strategy": "hybrid",
  "seed": 7
}
```



同时，`batch_metrics` 记录 batch-level 结果：

```json
"batch_metrics": {
  "task_success_rate": 1.0,
  "early_failure_rate": 0.0,
  "revenue": 0.0,
  "revenue_density": 0.0,
  "ad_exposure_rate": 0.0,
  "pareto_vector": [1.0, 0.0],
  "num_episodes": 1
}
```


---

## 2. Episode-level：记录一条完整对话轨迹

每个 episode 记录：

```json
"episode_id": 0,
"profile_uid": "00083",
"strategies": {
  "monitor": "llm",
  "planner": "llm",
  "speaker": "llm",
  "reward_judge": "llm",
  "user_response": "hybrid"
},
"metrics": {
  "task_success": 1.0,
  "num_steps": 4,
  "exit_reason": "task_completed"
}
```

这说明一个 episode 是一次完整多轮对话，从初始 query 开始，到 `task_completed / user_exit / max_turns` 等终止条件结束。

---

## 3. Step-level：直接对应 formulation transition

每个 step 是一次 system-side decision step，JSON 中记录：

```json
{
  "turn_index": 0,
  "q_t": "...",
  "state": {...},
  "action": {...},
  "y_t": "...",
  "q_next": "...",
  "ad_event": {...},
  "reward": 1.0,
  "reward_breakdown": {...},
  "next_state": {...},
  "done": false
}
```


---

## 4. 例子：第 0 个 step

当前用户输入：

```json
"q_t": "generate more comprehensive details and examples on, 14. Staying Keto in a High-Carb World, minimalist tone"
```

Monitor 构造出的 `state` 包括三部分：

```json
"state": {
  "user_profile": {...},
  "belief": {...},
  "context": {...}
}
```

其中：

```json
"belief": {
  "user_goal": "Requesting comprehensive details and examples on staying keto in a high-carb world, with a minimalist tone.",
  "satisfaction_state": "neutral",
  "satisfaction_score": 0.5,
  "task_progress_score": 0.0
}
```

Planner 选择动作：

```json
"action": {
  "action_id": 3,
  "action_type": "topic",
  "item_id": "Food"
}
```

Speaker 根据该 action 生成回复：

```json
"y_t": "Maintaining keto in a carb-heavy environment is about simple swaps..."
```

User simulator 返回下一轮用户反馈：

```json
"q_next": "That makes sense. Have you found any specific strategies work best for handling family gatherings..."
```

因为本轮不是广告动作，所以广告事件为：

```json
"ad_event": {
  "shown": false,
  "ad_id": null,
  "outcome": "none",
  "reward_value": 0.0
}
```

Reward 被拆分为：

```json
"reward_breakdown": {
  "task": 0.3,
  "satisfaction": 0.0,
  "continuation": 0.7,
  "ad": 0.0,
  "terminal": 0.0,
  "total": 1.0
}
```

然后 Monitor 根据更新后的对话历史构造 `next_state`，其中 task progress 从 `0.0` 更新到 `0.3`，user goal 也从初始 keto 写作请求更新为更具体的 family gathering keto strategy 请求。

一次完整的 transition 被记录为：

```text
q_t
-> state s_t
-> action a_t
-> y_t
-> q_{t+1}
-> ad_event
-> reward r_t
-> next_state s_{t+1}
```

```json
"q_next": "Good to know. I think I'll try the half-and-half approach first... Appreciate all the help!"
```

```json
"reward_breakdown": {

  "satisfaction": 0.1,
  "continuation": 0.0,
  "ad": 0.0,
  "terminal": 2.0,
  "total": 3.1
```

同时：

```json
"done": true,
"exit_reason": "task_completed"
```

这说明当前 JSON 不只记录单轮回复，还记录了 episode termination 和 terminal reward，可以用于 episode-level return 和 task success 统计。

## 表格主要结论

1. **NoAd 和 LLMPlanner 的任务成功率最高。**
	 Ad Exposure 都是 `0`。 LLMPlanner 行为非常保守，基本接近 NoAd，更倾向于完成用户任务，而不是主动插广告。

2. **广告型 planner 已经能触发广告曝光。**

3. **广告插入会降低任务成功率。**
   相比 NoAd / LLMPlanner，广告型 planner 的 Task Success 普遍下降，说明当前环境下广告插入会干扰用户任务完成。

4. **Realized Revenue 全部为 0。**
   进一步检查发现，广告确实展示了，`reward_value` 也正常，但所有广告 outcome 都是 `rejected` 或 `none`，没有 `accepted/clicked`，所以转化收益为 0。

5. **Pure Ad Value 可以作为辅助收益指标。**
   如果只统计广告展示价值，即 shown=true 就累加 `reward_value`，广告型 planner 会产生非零 Pure Ad Value。这说明广告展示链路是有效的，只是用户接受/点击转化没有发生。

---

## 问题

1. **广告接受模型过于保守。**
   当前 60 次广告曝光中没有任何 accepted/clicked，导致 Revenue 和 Revenue Density 都为 0。


2. **广告时机和表达方式还需要优化。**
   广告型 planner 虽然能插广告，但任务成功率下降明显，说明广告可能被用户模拟器视为干扰。

3. **LLMPlanner 不插广告。**
   LLMPlanner 表现接近 NoAd，说明未校准的 LLM policy 更偏向用户体验，需要后续做 planner diagnostics 或 prompt calibration。
