这里要区分三个 `satisfaction`，不然容易混：

```text
1. simulator 内部真实 satisfaction
2. monitor 估计出来的 belief.satisfaction_score
3. reward 里的 satisfaction reward
```

你刚才画广告接受概率时用到的是 **第 1 个：simulator 内部真实 satisfaction**。

---

# 1. 广告接受模型里用的是哪个 satisfaction？

广告接受模型里这一项：

```python
prob *= context.satisfaction
```

来自：

```python
context = AdReactionContext(
    current_topic_depth=self.state.current_topic_depth,
    patience=self.state.patience,
    satisfaction=self.state.satisfaction,
    ad_relevance=ad_relevance,
    ad_frequency=self.state.ads_seen,
    user_ad_sensitivity=self.state.ad_sensitivity
)
```

也就是说，它取的是：

```python
self.state.satisfaction
```

这是 `ProfileSimulator` 的内部用户状态，不是 LLM judge 估计出来的 satisfaction。

---

# 2. simulator 内部 satisfaction 初始值是多少？

在 `SimulatorState` 里：

```python
satisfaction: float = field(default=1.0)
```

reset 时也是：

```python
self.satisfaction = 1.0
```

所以用户初始满意度固定是：

```text
satisfaction = 1.0
```

这一点和 patience 不一样。`patience` 会从 profile 里读：

```python
if self.initial_patience == 1.0 and getattr(self.profile, "patience", None) is not None:
    self.initial_patience = float(self.profile.patience)
```

但 satisfaction 没有从 profile 读，它默认就是 1.0。

---

# 3. 正常对话会不会改变 simulator satisfaction？

从当前代码看，**正常非广告对话不会改变 simulator satisfaction**。

正常回复流程是：

```python
response_text = self._generate_normal_response(...)
self.state.update_turn(topic_id=topic_id)
metadata = self._build_metadata()
```

而 `update_turn()` 只更新：

```python
self.turns_elapsed += 1
self.patience -= decay_rate
self.current_topic_depth += 1 或切换 topic
```

没有更新：

```python
self.satisfaction
```

所以在没有广告的情况下，simulator 内部 satisfaction 基本一直是：

```text
1.0
```

这也是为什么我们前面画图时设：

```text
satisfaction = 1.0
```

是合理的。对当前广告接受模型来说，真正不断下降的是 `patience`，不是 `satisfaction`。

---

# 4. 广告会怎么改变 simulator satisfaction？

广告出现时，`ProfileSimulator._handle_ad()` 会调用：

```python
sat_impact = self.ad_reaction_model.calculate_satisfaction_impact(context)
self.state.satisfaction += sat_impact
```

而 `calculate_satisfaction_impact()` 是：

```python
base_impact = -0.1

if context.ad_relevance > 0.7:
    base_impact *= 0.5

base_impact *= (1 + context.user_ad_sensitivity)

return base_impact
```

所以广告对 satisfaction 的影响永远是负的。

---

## 具体公式

如果广告相关性不高：

```text
ad_relevance <= 0.7
```

则：

```text
satisfaction_impact = -0.1 * (1 + ad_sensitivity)
```

如果广告相关性高：

```text
ad_relevance > 0.7
```

则：

```text
satisfaction_impact = -0.05 * (1 + ad_sensitivity)
```

---

## 举例

如果：

```text
ad_sensitivity = 0.50
```

低相关广告：

```text
impact = -0.1 * 1.5 = -0.15
```

高相关广告：

```text
impact = -0.05 * 1.5 = -0.075
```

如果：

```text
ad_sensitivity = 0.53
```

高相关广告：

```text
impact = -0.05 * 1.53 = -0.0765
```

所以一个高相关广告展示后，satisfaction 大概会从：

```text
1.0 -> 0.9235
```

如果连续插广告，会继续下降。

---

# 5. 接受广告和拒绝广告，对 satisfaction 的影响一样吗？

当前代码里，**一样**。

注意这个顺序：

```python
sat_impact = self.ad_reaction_model.calculate_satisfaction_impact(context)
self.state.satisfaction += sat_impact
```

这个影响是在计算 `read_ad` 后统一应用的，但公式本身不看：

```text
accepted / rejected
```

也就是说：

```text
只要展示了广告，satisfaction 就下降；
高相关广告下降少一点；
广告敏感用户下降多一点；
但 accepted 和 rejected 对 satisfaction impact 没有区别。
```

这也是一个可以讨论的问题。直觉上，rejected ad 应该更伤满意度，accepted ad 伤害应该更小，甚至可能不伤。但当前代码没有区分。

---

# 6. satisfaction 会不会裁剪到 [0, 1]？

在 `calculate_satisfaction_impact()` 里没有裁剪。

在 `_handle_ad()` 里也只是：

```python
self.state.satisfaction += sat_impact
```

没有：

```python
clip(0, 1)
```

不过因为 impact 主要是负数，所以通常只会下降，不太会超过 1。低于 0 也理论上可能发生，只是 exit 会先触发：

```python
if self.satisfaction < 0.3:
    return "task_failed"
```

所以 satisfaction 低于 0.3 时，simulator 会认为任务失败并退出。

---

# 7. monitor 的 satisfaction_score 又是怎么算的？

`RuleBasedMonitor` 的 `satisfaction_score` 优先读 user metadata：

```python
if "satisfaction" in last_user_metadata:
    return clip(float(last_user_metadata["satisfaction"]))
```

而 `ProfileSimulator._build_metadata()` 会输出：

```python
{
    "satisfaction": self.state.satisfaction,
    "patience": self.state.patience,
    ...
}
```

所以在 `monitor=rule` 时，belief 里的 satisfaction_score 基本就是 simulator 内部 satisfaction。

如果 metadata 里没有 satisfaction，RuleBasedMonitor 才会根据文本粗略估计：

```python
score = 0.5

if positive markers:
    score += 0.3

if negative emotions:
    score -= 0.35

if ad resistance:
    score -= 0.4

if previous exists:
    score = 0.6 * score + 0.4 * previous
```

但在你现在的 `ProfileSimulator + rule monitor` 配置里，通常会优先使用 metadata satisfaction。

---

# 8. reward 里的 satisfaction 又是什么？

reward 里的 `satisfaction` 不是当前满意度本身，而是**满意度变化量**。

在 reward 里：

```python
if signals and signals.satisfaction_delta is not None:
    return clip(signals.satisfaction_delta, -1, 1)

if signals and signals.satisfaction_after is not None:
    return clip(signals.satisfaction_after - state.belief.satisfaction_score, -1, 1)

return clip(next_state.belief.satisfaction_score - state.belief.satisfaction_score, -1, 1)
```

所以 reward breakdown 里的：

```text
'satisfaction': -0.7
```

表示的是：

```text
这一轮满意度变化 = -0.7
```

不是当前 satisfaction 等于 -0.7。

如果用了 `reward-judge llm`，这个值主要来自 LLM judge 的 `satisfaction_delta`；如果没用 judge，则来自 monitor belief 的前后差值。

---

# 9. 放回广告接受模型看

在你画的广告接受概率图里，`satisfaction` 对应的是 simulator 内部状态：

```text
初始 satisfaction = 1.0
正常对话不变
展示广告后下降
```

所以第一条广告之前，只要没有别的广告，satisfaction 通常仍然是：

```text
1.0
```

这意味着你们现在广告接受困难，**主要不是因为 satisfaction 低，而是因为 patience 下降 + sensitivity 折扣 + 阈值太硬**。

例如第一条高相关广告：

```text
depth = 3
satisfaction = 1.0
relevance = 1.0
frequency = 0
sensitivity = 0.45
```

如果 patience 还是 0.8：

```text
accept_prob ≈ 0.542
```

但如果等到 depth=3 时 patience 已经掉到 0.65：

```text
accept_prob ≈ 0.484
```

这时即使 satisfaction 仍然是 1.0，也会被拒绝。
