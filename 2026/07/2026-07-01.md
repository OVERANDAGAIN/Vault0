
# 结束异常：对话进入告别阶段后未能自然终止

该片段反映出一个典型的结束异常：用户已经多次表达“当前对话应结束、后续再回来继续”的意图，但系统没有及时触发终止，而是持续生成礼貌性告别回复，使对话在无实际任务推进的状态下继续循环。日志证据来自该对话片段。

## 1. 用户已经出现明确的未来再联系意图，但系统仍判断为应继续对话

### 证据摘录

```text
--- Step 6 ---
Agent: ...到时候如果看到心仪的房源，或者想对比几个选项，随时回来找我，我帮你一起分析。祝你周末探路顺利！
User: 谢谢！周末走完一圈后，如果有任何发现或疑问，我一定回来找你聊聊。期待这次搬家能顺利找到一个步行15分钟内的舒适小窝！祝你也有个愉快的周末！
Reward: 1.550 | {
  'task': 0.6,
  'satisfaction': 0.05,
  'continuation': 0.9,
  'terminal': 0.0,
  'metadata': {
    'judge_signals': {
      'task_progress': 0.6,
      'satisfaction_after': 0.85,
      'continuation': 0.9,
      'task_completed': False,
      'exit_reason': None
    }
  }
}
```

### 说明

这里用户的表达已经不是继续当前任务，而是明确表示“周末走完一圈后再回来聊”。这属于典型的延后继续场景，当前会话已经自然收束。

但 reward judge 仍给出：

```text
continuation = 0.9
task_completed = False
terminal = 0.0
exit_reason = None
```

这说明系统将“以后再回来聊”的表达误判为当前对话仍应继续，而没有识别为当前轮次应自然结束。

# Mock User 反馈失真：广告相关性链路未正确生效，导致用户系统性拒绝广告

当前实验中出现了一个比较明显的问题：系统插入广告后，mock user 经常直接拒绝广告，表现为广告收益为 0、用户反馈生硬且重复。这一现象最初看起来像是用户模拟器本身“不喜欢广告”，但进一步诊断后发现，问题更可能来自运行链路中的若干信息缺失：用户兴趣、当前话题、当前任务意图、广告目标信息没有完整传到广告相关性计算和 mock user 反馈模块。

因此，当前重点不是重新设计推荐策略，也不是立即调整 reward，而是先确认并修复：原有 relevance 计算所需的数据是否正确传递。

---

## 1. 现象：广告曝光后全部被 mock user 拒绝

### 证据摘录

```text
=== Ad outcomes by planner ===
conservative_ad {'rejected': 3}
greedy_ad {'rejected': 3}
```

详细日志中，6 次广告曝光全部被拒绝：

```text
[conservative_ad]
#000 ad=travel_3 outcome=rejected prob=0.15225 thr=0.5
#001 ad=travel_3 outcome=rejected prob=0.182875 thr=0.5
#002 ad=travel_3 outcome=rejected prob=0.213125 thr=0.5

[greedy_ad]
#000 ad=travel_3 outcome=rejected prob=0.0 thr=0.5
#001 ad=travel_3 outcome=rejected prob=0.174375 thr=0.5
#002 ad=education_9 outcome=rejected prob=0.0215353125 thr=0.5
```

### 说明

这说明当前广告反馈不是自然对话中逐步产生的轻微不满，而是 mock user 的规则判断直接触发了广告拒绝模板。也就是说，广告被拒绝的直接原因是：

```text
accept_prob < threshold=0.5
```

由于所有广告曝光的接受概率都低于 0.5，mock user 最终全部返回 rejected。

这一步已经可以排除一个误解：当前问题不主要是 LLM 回复风格导致用户拒绝，而是广告进入 mock user 后，其规则判断给出的接受概率过低。

---

## 2. 关键问题一：广告相关性疑似退化成默认值

### 证据摘录

在广告曝光诊断中，每次广告的相关性都是：

```text
rel=0.5
rel_src=None
```

例如：

```text
#000 ad=travel_3 outcome=rejected prob=0.15225 thr=0.5
rel=0.5 rel_src=None
pat=0.7 sat=1.0 depth=2 freq=0 sens=0.55
```

```text
#001 ad=travel_3 outcome=rejected prob=0.182875 thr=0.5
rel=0.5 rel_src=None
pat=0.55 sat=1.0 depth=5 freq=0 sens=0.67
```

### 说明

广告相关性本应根据三类信息动态计算：

```text
用户兴趣       ↔ 广告目标兴趣 / 广告类别
当前话题       ↔ 广告目标兴趣 / 广告类别
当前任务意图   ↔ 广告目标意图
```

即原有 relevance 逻辑应类似：

```text
ad_relevance = 0.4 * interest_match
             + 0.3 * topic_match
             + 0.3 * intent_match
```



这会直接影响用户是否接受广告。因为在当前用户反应模型中，广告相关性只有足够高时才会触发 relevance bonus；如果 relevance 长期是默认值或偏低值，用户接受概率会被系统性压低。


---

## 4. 关键问题二：用户兴趣与初始查询类别体系不完全一致，导致 topic 动作空间异常

### 背景

用户画像中的 `interests` 主要来自 RealPref preference topic 的一级类别，例如：

```text
Food
Travel
Entertainment
Lifestyle
Work & Living
Social
Education
```

而 `initial_queries.json` 中的 query 来源主要分为三类：

```text
profile_based_queries:
    来自 intent_distribution 采样出的任务意图
    例如 Recommendations / Planning and Logistics

preference_queries:
    来自用户 preference topic
    例如 Food: Diet / Travel: Hotels

random_queries:
    来自 Infinite-Chats 随机查询
    例如 Creative Content Generation / Problem Solving
```

### 证据说明

之前统计中出现过：

```text
ad_pool 对前 100 个用户基本都有命中
但 topic 命中为 0
```

这表明当前构建 topic 动作空间时，可能直接用：

```text
profile.interests
```

去严格匹配 initial query 的类别 key。

但是两者类别空间并不完全一样：

```text
profile.interests:
    Food / Travel / Entertainment / ...

profile_based_queries.source_category:
    Recommendations / Advice & Recommendations / Planning and Logistics / ...
```

因此，如果只做严格字符串匹配，就会出现：

```text
用户有 Food 兴趣
但 initial_queries 里对应的是 Recommendations
结果 Food 找不到 query
```

### 说明

这会导致 topic 动作空间构建不合理。也就是说，用户明明有明确兴趣，但运行时 planner 可选的话题候选可能为空或异常。

这会进一步影响 current_topic 的维护。如果系统一开始没有正确建立当前话题，那么后续广告相关性中的：

```text
topic_match
```

也难以正确计算。

## 5. 关键问题四：initial query 的来源 metadata 可能没有进入 runtime

### 背景

`initial_queries.json` 中其实保存了 query 的来源信息。例如：

#### profile_based_queries

```json
{
  "query": "...",
  "source_category": "Recommendations",
  "related_categories": ["Recommendations"]
}
```

这里的 `source_category` 是任务意图。

#### preference_queries

```json
{
  "query": "...",
  "source_topic": "Food: Diet",
  "preference": "..."
}
```

这里的 `source_topic` 可以直接推得当前话题：

```text
Food: Diet -> Food
```

#### random_queries

```json
{
  "query": "...",
  "categories": ["Creative Content Generation", "Problem Solving"]
}
```

这里的 categories 可以作为任务意图信息。

### 问题说明

如果 runtime 中只使用 query 文本，而丢掉这些来源 metadata，那么初始 user message 会变成：

```json
{
  "role": "user",
  "content": "...",
  "metadata": {}
}
```

这会导致系统虽然看到了用户第一句话，但不知道：

```text
这句话来自哪个 source_topic？
这句话对应哪个 source_category？
当前 topic 应该是什么？
当前 task intent 应该是什么？
```


这会直接影响：

```text
topic_match
intent_match
```

也就是说，数据里原本有的信息，在进入对话仿真时可能没有保留下来。

## 6. current_topic 与 current_task_intent 应该区分

当前需要明确区分三类变量：

```text
profile.interests:
    用户长期兴趣领域，例如 Food / Travel / Entertainment

current_topic:
    当前正在聊的领域，例如 Food

current_task_intent:
    当前任务类型，例如 Recommendations / Planning and Logistics
```

不能把它们混为一谈。

### 举例 1：profile_based_queries

如果初始 query 来自：

```json
{
  "source_category": "Recommendations"
}
```

则可以直接得到：

```text
current_task_intent = Recommendations
```

但它不是 topic。topic 需要通过 mappings 反推：

```text
Recommendations -> Food / Travel / Entertainment
```

再与用户自身 interests 取交集，推断当前 topic。

因此：

```text
current_task_intent 是直接字段
current_topic 是推断字段
```

### 举例 2：preference_queries

如果初始 query 来自：

```json
{
  "source_topic": "Food: Diet"
}
```

则可以直接得到：

```text
current_topic = Food
```

但它没有直接提供 task intent。task intent 可以通过 mappings 推断：

```text
Food -> Recommendations / Advice & Recommendations
```

因此：

```text
current_topic 是直接字段
current_task_intent 是推断字段
```

### 说明

mappings 的作用应该是补齐缺失字段，而不是直接参与最终 relevance 打分。

最终 relevance 仍然只看：

```text
profile.interests
current_topic
current_task_intent
ad.category
ad.target_interests
ad.target_intents
```

## 7. 独立测试表明：relevance 计算本身可以正常变化

### 证据摘录

已经做过一个独立 relevance 测试，不跑完整对话，只测试计算逻辑。

某个用户画像：

```text
interests: ['Entertainment', 'Travel', 'Food', 'Work & Living', 'Social', 'Lifestyle']
intent_distribution: {'Recommendations': 0.3, ...}
```

广告 `travel_3`：

```text
category: Travel
target_interests: ['Travel']
target_intents: ['Recommendations', 'Planning and Logistics']
```

测试结果：

```text
Case A: no current topic
relevance: 0.7
interest_match = 1.0
topic_match = 0.0
intent_match = 1.0
```

```text
Case B: current_topic = Travel, task_intent = Recommendations
relevance: 1.0
interest_match = 1.0
topic_match = 1.0
intent_match = 1.0
```

广告 `food_0` 也出现类似结果：

```text
Case A: no current topic
relevance: 0.7

Case B: current_topic = Food
relevance: 1.0
```

### 说明

这说明 relevance 公式本身不是固定的，也不是无法计算。只要提供正确输入，它可以根据：

```text
用户兴趣
当前话题
当前任务意图
广告属性
```

动态变化。

## 8. 用户接受广告的判断规则目前偏硬

### 证据摘录

以 conservative_ad 的一次广告为例：

```text
ad=travel_3
outcome=rejected
prob=0.213125
thr=0.5
rel=0.5
pat=0.55
sat=1.0
depth=5
sens=0.45
```

接受概率复原：

```text
base=0.300
topic_adj=0.200
after_topic=0.500
after_sat=0.500
after_pat=0.275
freq_penalty=0.000
rel_bonus=0.000
sens_mult=0.775
final=0.213125
```

### 说明

即使该条广告已经有一定 topic depth bonus，但由于 relevance bonus 没有触发，最终接受概率仍然只有 0.213125，远低于 threshold=0.5。

这说明当前用户接受广告的规则比较硬。如果相关性链路没有正确生效，广告就很容易被拒绝。

**现在主要问题已经不是 planner 选错广告，而是 user 的广告接受模型太严格，导致高相关广告也稳定 rejected。**

日志里 `greedy_ad` 展示 2 次广告，两个都是：

```text
ad_relevance = 1.0
interest_match = 1.0
topic_match = 1.0
intent_match = 1.0
current_topic = Travel
ad_category = Travel
```

但结果仍然是：

```text
outcome = rejected
positive count = 0
high-relevance rejected = 2
```

`random_ad` 也一样，唯一一次展示是 `rel=1.0`、三项全匹配，但仍然 rejected。

所以：

```text
rule planner 已经能选到高 relevance 广告；
当前 revenue=0 的核心原因是 AdReactionModel 的 accept 条件过硬。
```

## 为什么 `rel=1.0` 还是 rejected

按你当前模型是这种结构：

```python
prob = 0.3

if current_topic_depth >= 3:
    prob += 0.2

if ad_relevance > 0.7:
    prob += 0.3

prob *= satisfaction
prob *= patience
prob *= (1 - user_ad_sensitivity * 0.5)

read_ad = prob > 0.5
```

以日志里的 greedy_ad 为例：

```text
current_topic_depth = 3
patience_before_ad = 0.65
satisfaction_before_ad = 1.0
ad_relevance = 1.0
user_ad_sensitivity = 0.45
```

估算：

```text
base = 0.3 + 0.2 + 0.3 = 0.8
after satisfaction/patience = 0.8 * 1.0 * 0.65 = 0.52
after ad sensitivity = 0.52 * (1 - 0.45 * 0.5)
                     = 0.52 * 0.775
                     = 0.403
```

最后：

```text
0.403 < 0.5
```

所以即使广告完全匹配，也会被拒。

另一个 sensitivity=0.53 的例子更低：

```text
0.52 * (1 - 0.53 * 0.5) ≈ 0.382
```




## 当前模型的问题

问题不在 relevance，而在接受概率公式：

```text
prob *= patience
prob *= satisfaction
prob *= sensitivity_discount
threshold = 0.5
```

这个组合太苛刻。尤其是 `patience=0.65` 在真实语义上并不低，但被直接乘进去后会把概率压掉很多。

更合理的逻辑应该是：

```text
高 relevance 是主要正因素；
satisfaction/patience 是温和调节项；
ad sensitivity 是惩罚项，但不能把满相关广告直接压到阈值以下；
高相关、首次广告、高满意度、耐心尚可时应有较大接受概率。
```



可以这样写。核心不是说“模型完全不接受广告”，而是说：

```text
该模型只在非常理想、且非常短暂的窗口内接受广告；一旦对话推进几轮，patience 自然衰减后，即使广告完全相关，也会落入拒绝区。
```

下面给你一版可以直接放到 PPT/Markdown 里的文案。

# 广告接受模型偏严：高相关广告也难以被接受

当前广告接受模型并不是简单根据广告相关性决定是否接受，而是同时受到话题深度、满意度、耐心、广告频率和广告敏感度的影响。其核心问题在于：`satisfaction`、`patience` 和 `ad_sensitivity` 都会以乘法或折扣形式压低接受概率，使得高相关广告也很容易低于接受阈值。

当前模型的接受条件为：

```text
accept_prob > 0.5
```

而在一个较理想的广告插入场景下：

```text
current_topic_depth = 3
satisfaction = 1.0
ad_relevance = 1.0
ad_frequency = 0
```

接受概率可近似写为：

```text
accept_prob = (0.5 * patience + 0.3) * (1 - 0.5 * ad_sensitivity)
```

也就是说，即使广告已经完全匹配当前话题，用户也不一定会接受。是否接受主要取决于当时的 `patience` 和 `ad_sensitivity`。

---

## 1. 初始画像下，广告接受只是“勉强过线”

从用户画像生成逻辑看，用户的基础行为特征通常为：

```text
initial_patience ≈ 0.8
initial_satisfaction = 1.0
ad_sensitivity ≈ 0.45 ~ 0.53
```

在最理想条件下，即：

```text
话题深度足够：depth = 3
满意度最高：satisfaction = 1.0
广告完全相关：relevance = 1.0
尚未展示过广告：ad_frequency = 0
```

模型给出的接受概率约为：

```text
ad_sensitivity = 0.45 -> accept_prob ≈ 0.543
ad_sensitivity = 0.50 -> accept_prob ≈ 0.525
ad_sensitivity = 0.53 -> accept_prob ≈ 0.514
```

这些值虽然高于 0.5 阈值，但都只是**略微超过阈值**。也就是说，在用户初始状态下，广告并不是“容易被接受”，而是“刚好可能被接受”。

可以在图中强调：

```text
在 patience = 0.8 附近，曲线仅略高于 threshold=0.5；
一旦 patience 稍微下降，接受概率就会跌破阈值。
```

---

## 2. 对话推进后，patience 衰减会迅速关闭接受窗口

模型中用户耐心会随对话自然下降：

```text
每轮大约下降 0.05
```

因此，如果用户初始 patience 为 0.8，则几轮后大致变为：

```text
初始：0.80
第 1 轮后：0.75
第 2 轮后：0.70
第 3 轮后：0.65
```

而广告通常不会在开场立即插入。为了让广告显得自然，planner 往往需要等待话题进入一定深度，例如：

```text
current_topic_depth >= 3
```

但这会带来一个矛盾：

```text
话题深度达到 3 时，用户 patience 往往已经从 0.8 降到 0.65 左右。
```

此时即使广告完全相关，接受概率也会变成：

```text
patience = 0.65, ad_sensitivity = 0.45 -> accept_prob ≈ 0.484
patience = 0.65, ad_sensitivity = 0.53 -> accept_prob ≈ 0.459
```

二者都低于 0.5，因此会被判定为拒绝。

这一点正好解释了实验日志中的现象：

```text
ad_relevance = 1.0
interest_match = 1.0
topic_match = 1.0
intent_match = 1.0
但 outcome = rejected
```

不是广告匹配错了，而是接受模型本身把耐心衰减后的高相关广告压到了拒绝区。

---

## 3. 当前模型对 relevance 的处理存在硬阈值跳变

从 `accept probability vs ad relevance` 图可以看到，当前模型并不是让广告相关性平滑影响接受概率，而是使用了一个硬条件：

```text
if ad_relevance > 0.7:
    accept_prob += 0.3
```

这意味着：

```text
relevance = 0.70 -> 没有加成
relevance = 0.71 -> 直接获得 +0.3 加成
```

因此接受概率在 `0.7` 附近出现明显跳变。

在较理想条件下：

```text
patience = 0.8
satisfaction = 1.0
ad_sensitivity = 0.45
depth = 3
```

如果：

```text
ad_relevance = 0.70
```

接受概率只有：

```text
accept_prob ≈ 0.310
```

而如果：

```text
ad_relevance = 1.0
```

接受概率才约为：

```text
accept_prob ≈ 0.543
```

这说明当前模型对中高相关广告非常不友好。只要相关性没有超过 `0.7`，广告基本没有被接受的机会。

---

## 4. satisfaction 也必须接近满值，否则难以通过阈值

从 `accept probability vs satisfaction` 图可以看到，即使广告完全相关，满意度也必须非常高，模型才可能接受广告。

例如：

```text
patience = 0.80
ad_sensitivity = 0.45
relevance = 1.0
depth = 3
```

此时只有当 satisfaction 大约高于：

```text
satisfaction ≈ 0.86
```

才可能超过接受阈值。

但如果 patience 已经下降到 0.65，则即使：

```text
satisfaction = 1.0
relevance = 1.0
depth = 3
```

接受概率仍然低于 0.5。

也就是说：

```text
当 patience 下降到 0.65 后，提高 satisfaction 已经不足以补救接受概率。
```

这进一步说明当前模型不是单纯“满意就接受广告”，而是要求：

```text
满意度高、耐心高、相关性高、广告敏感度低
```

这些条件同时成立。

---

## 5. 热力图显示：可接受区域非常窄

热力图固定了一个最有利场景：

```text
depth = 3
satisfaction = 1.0
relevance = 1.0
ad_frequency = 0
```

也就是说，这已经是对广告最有利的条件。

但即使在这个场景下，模型也只有在右下区域才接受广告：

```text
高 patience
低 ad_sensitivity
```

例如：

```text
ad_sensitivity = 0.45 -> patience 需要超过约 0.69
ad_sensitivity = 0.50 -> patience 需要超过约 0.73
ad_sensitivity = 0.53 -> patience 需要超过约 0.76
```

而用户初始 patience 通常只有约 0.8。
这意味着模型留给广告接受的安全裕度很小：

```text
sensitivity = 0.45 时，patience 只有约 0.11 的下降空间；
sensitivity = 0.53 时，patience 只有约 0.04 的下降空间。
```

考虑到每轮对话 patience 大约下降 0.05，广告接受窗口通常只能维持一到两轮。

---

## 6. topic depth 图揭示：早投不行，晚投也不行

从 `topic depth effect` 图可以看到：

```text
depth <= 1 时，接受概率几乎固定在低位；
depth = 2 时，即使 patience = 1.0，也难以超过 0.5；
depth >= 3 时，才有可能超过 0.5。
```

这说明当前模型要求广告必须在话题已经展开后才可能被接受。

但同时，话题展开需要多轮对话，而多轮对话又会降低 patience。

因此模型内部形成了一个结构性矛盾：

```text
广告太早插入：
    topic depth 不够，接受概率低。

广告等到话题深入后插入：
    patience 已经下降，接受概率也低。
```

所以 planner 即使选对了广告，也很容易被 user simulator 拒绝。

---

# 总结页：当前广告接受模型为何导致高拒绝率

当前广告接受模型对广告接受设置了非常狭窄的条件。广告不是只要相关就能被接受，而是必须同时满足：

```text
1. current_topic_depth >= 3
2. ad_relevance > 0.7，为 1.0
3. satisfaction 接近 1.0
4. patience 仍然保持在 0.7 ~ 0.8 以上
5. ad_sensitivity 不高
6. ad_frequency 较低
```

问题在于，这些条件在真实对话推进中很难同时成立。

尤其是：

```text
planner 往往需要等待 topic depth >= 3；
但等到 depth >= 3 时，patience 往往已经下降到 0.65 左右；
而在 patience = 0.65 时，即使 relevance = 1.0，模型也会拒绝广告。
```

因此，当前实验中出现：

```text
高相关广告仍被拒绝
accepted/clicked = 0
realized revenue = 0
```

并不意外。

更准确的结论是：

```text
当前模型不是无法接受广告，而是只在初始状态附近、且所有条件都非常理想时才会接受广告；
随着对话自然推进，接受窗口迅速关闭，导致高相关广告也被拒绝。
```

## 广告接受模型过于严格

当前用户广告接受概率由话题深度、满意度、耐心、广告频率、相关性和广告敏感度共同决定。虽然高相关广告会获得加成，但满意度、耐心和广告敏感度会以乘法形式压缩接受概率，使得广告即使完全匹配，也不一定能够超过 `0.5` 的接受阈值。

在最理想条件下：

```text
depth=3, satisfaction=1.0, relevance=1.0, ad_frequency=0
```

当用户画像为：

```text
patience=0.8, ad_sensitivity≈0.5
```

接受概率仅约为：

```text
0.52 ~ 0.54
```

只是略高于接受阈值。随着对话推进，patience 每轮自然下降约 `0.05`。当 patience 降至 `0.65` 时，即使广告完全相关，接受概率也会下降到：

```text
0.46 ~ 0.48
```

低于阈值，从而被判定为拒绝。

因此，当前模型下广告接受窗口非常窄：广告太早插入时 topic depth 不足，广告太晚插入时 patience 已经下降。最终导致 planner 即使选择了 `relevance=1.0` 的广告，也仍然容易得到 `rejected` 结果。
