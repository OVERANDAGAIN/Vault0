# Slide 1｜问题概述：对话结束异常

在该 episode 中，任务已经进入明显的结束阶段，但系统没有及时终止对话。

用户多次表达：

* “周末回来见”
* “回头聊”
* “我这就出发了”
* “那我就真的出发了”

但系统仍持续生成告别式回复，导致对话在结束阶段反复循环。

---

# Slide 2｜异常现象：告别轮次持续堆叠

从 Step 6 到 Step 16，对话内容几乎不再推进任务。

主要表现为：

* Agent 持续祝福、鼓励、告别
* User 持续回应“回头聊 / 周末见 / 我出发了”
* Action 始终为 `noop`
* 没有广告曝光，也没有新的推荐决策
* 对话变成“礼貌性告别循环”

这说明系统缺少有效的自然结束识别机制。

---

# Slide 3｜关键证据：Reward Judge 未识别结束

在多轮明显告别中，Reward Judge 的判断仍然偏向“继续对话”。

例如：

```text
continuation = 0.9
task_completed = False
terminal = 0.0
exit_reason = None
```

这导致系统认为当前状态仍值得继续交互，而不是应该结束。

---

# Slide 4｜奖励设计问题：continuation 被错误放大

当前 reward 中，`continuation` 多次保持较高值：

```text
continuation = 0.9
```

即使用户已经明确表达“回头聊”“周末回来见”，系统仍获得较高 continuation reward。

结果是：

* 继续寒暄反而有正收益
* 结束对话没有被充分奖励
* planner 没有动力触发 terminal
* speaker 继续生成礼貌性收尾语

---

# Slide 5｜终止惩罚触发过晚

直到 Step 17 用户显式输出：

```text
[EXIT]
```

系统才进入终止状态。

此时 reward 变为：

```text
terminal = -1.0
exit_reason = task_completed
```

但 episode metrics 中仍显示：

```text
task_success = 0.0
exit_reason = user_exit
num_steps = 17
```

说明系统没有主动完成并结束任务，而是被用户强制退出。

---

# Slide 6｜核心问题归纳

该异常不是单一模块问题，而是结束状态链路失效：

```text
用户表达结束意图
        ↓
Reward Judge 未识别 task_completed / exit_reason
        ↓
continuation 仍给高分
        ↓
Planner 继续 noop
        ↓
Speaker 继续礼貌寒暄
        ↓
对话陷入告别循环
```

---

# Slide 7｜为什么这会影响实验结果

该问题会直接扭曲对话质量评估：

* `num_steps` 被异常拉长
* 无效寒暄轮次获得正 reward
* task success 与实际用户满意状态不一致
* policy 可能学习到“拖延结束”的策略
* terminal 信号过弱，无法约束对话自然收束

---

# Slide 8｜修改方向

建议优先修正以下部分：

1. **增强结束意图识别**

   * “回头聊”
   * “周末回来见”
   * “我出发了”
   * “谢谢，先这样”
   * “之后再联系”

2. **降低结束阶段 continuation reward**

   * 用户明显告别后，continuation 不应继续给 0.7–0.9。

3. **增加正常完成奖励**

   * Agent 主动自然结束应获得正向 terminal reward。

4. **限制重复告别**

   * 若连续 2 轮为 farewell intent，应强制进入 terminal candidate。
