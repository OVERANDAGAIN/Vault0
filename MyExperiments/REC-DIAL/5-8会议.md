## 5.8 会议纪要：

### 主要内容

当前 Reward 与  Mock User 主流程已初步接入。讨论重点由单模块功能转向整体系统框架设计，包括 Mock User 模拟逻辑、Monitor / Planner / Speaker 模块边界、整体代码结构同步。

### 一、Monitor 模块

1. Monitor 与 Planner 在部分功能上存在一定重合。Monitor 负责满意度、疲劳度等信号输出，与后续planner决策高度关联。因此：Monitor 与 Planner 是否可合并，或者 还是保持“Monitor 提供状态评估、Planner 负责策略决策”的辅助关系。

2. Monitor 输出信号的合理性与可信度。包括满意度、广告敏感度等指标是否足够敏感、数值是否稳定，以及指标分布是否能够反映真实用户情况。所以怎么得到一个好的monitor。

### 二、Planner 模块

1. Planner 与 Speaker 的关系。当前 Reward Function 实际上直接反馈的是 Speaker 的最终输出，再间接影响 Planner，因此两者的统一性很关键，例如联合优化或分开训练，或者是否可能出现planner策略与speaker表达之间的不一致。

2. Planner 本身作为高层决策模块，其训练复杂度相对更高。Planner 涉及的话题切换、广告插入等决策空间较大，如何设计合理的训练方式需进一步讨论。

### 三、整体框架同步

目前 Monitor、Planner、Speaker 三者之间的边界仍在讨论阶段，因此整体代码框架需要尽快同步和敲定，明确模块关系。
