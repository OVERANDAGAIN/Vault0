同意。这里应当按 formulation 重建一条清晰主线，而不是在旧 `Monitor.evaluate_turn()` 外面再包 `state_builder`。否则代码会变成：

```text
Monitor.evaluate_turn() -> signals
StateBuilder.build() -> state
Planner.act() -> action
```

这会制造两个问题：

```text
1. Monitor 的职责被拆碎：既不像旧代码的 signal detector，也不像 formulation 中的 state constructor。
2. 后续 reward / metrics / env 都不知道应该依赖 signals 还是 state。
```

所以后面方案改为：**Monitor 直接负责构造 PlannerState；不再额外引入 StateBuilder。**

---

# 总体改造目标

最终代码结构应该直接对应 formulation：

```text
Episode start
  ↓
DialogueEnv.reset()
  ↓
Monitor.build_state(p_u, H_t) -> s_t
  ↓
Planner.act(s_t, action_space) -> a_t
  ↓
Speaker.generate(s_t, a_t) -> y_t
  ↓
UserSimulator.generate_response(H_t, y_t) -> q_{t+1}
  ↓
EpisodeLog append y_t, q_{t+1}, ad_event
  ↓
Monitor.build_state(p_u, H_{t+1}) -> s_{t+1}
  ↓
RewardCalculator.compute(s_t, a_t, s_{t+1}) -> r_t
  ↓
TrajectoryStep(s_t, a_t, r_t, s_{t+1}, done_t)
```

这和 formulation 中的 transition 完全一致：Monitor 把 observation 压缩成 (s_t=(p_u,B_t,D_t))，Planner 输出一维动作，Speaker 实现动作，Mock User 返回 raw feedback，RewardCalculator 再基于状态转移计算 reward。

# 一、代码设计原则

## 1. 以 formulation 为主线，不保留两套并行逻辑

后续不建议继续维护：

```text
旧 Agent.act(history)
旧 Planner.should_insert_ad()
旧 Planner.select_next_topic()
旧 Monitor.evaluate_turn()
```

因为这些函数对应的是旧 pipeline：

```text
monitor signals -> ad decision -> topic decision -> speaker
```

而 formulation 是：

```text
state -> one-dimensional action -> speaker
```

两者决策边界不同。继续兼容旧逻辑会让代码越来越绕。

---

## 2. 少加 class，必要对象集中放置

不做过度抽象。不引入：

```text
StateBuilder
FormulationSpeakerAdapter
PlannerAdapter
TransitionManager
EpisodeManager
```

这些都可以避免。

建议只保留几个必要模块：

```text
types / state / monitor / planner / speaker / env / reward / metrics
```

每个模块有明确职责，不互相抢职责。

---

## 3. 每次修改都要能对应 formulation 的一个部分

后续每一步改代码都按这个清单说明：

```text
文件：
修改内容：
对应 formulation：
为什么必须改：
是否影响其他模块：
```

这会作为后面逐步打勾的标准。

## 2. Action space 设计

### 修改位置

```text
core/planner.py
```

或者如果为了清晰，也可以放在：

```text
core/types.py
```

但我更建议放在 `planner.py`，因为 action space 是 Planner 的输入约束。

---

### 设计目标

Planner action 是一维离散动作：

```text
[-1, ad_0, ad_1, ..., ad_m, topic_0, topic_1, ..., topic_n]
```

其中：

```text
-1 = noop
0 <= a < |D| = ad action
|D| <= a < |D| + |T| = topic action
```

这和 formulation 完全一致。

---

### 需要实现

```python
class ActionSpace:
    def __init__(self, ad_ids: list[str], topic_ids: list[str]):
        ...

    def decode(self, action_id: int) -> PlannerAction:
        ...

    def valid_action_ids(self) -> list[int]:
        ...
```

这里可以接受一个小 class，因为它不是额外抽象，而是 formulation 中 action mapping 的必要实现。

关键点：

```text
1. action_id 是 RL 训练保存的动作。
2. item_id 是代码执行时使用的真实 topic_id / ad_id。
3. ads 必须在 topics 前面。
4. -1 必须作为 noop 保留。
```

## 3. Monitor 设计

### 修改位置

```text
core/monitor.py
```

### 核心变化

旧接口：

```python

```

新接口：

```python

```

这是本次重构的核心。Monitor 不再只是 signal detector，而是 formulation 中的状态构造器：

```text
M(p_u, H_t) -> s_t
```

formulation 明确 Monitor 的职责是读取用户画像和完整对话历史，抽取用户需求与偏好，维护 dialogue memory，保留最近上下文，构造广告事件序列，并输出 Planner 可用的压缩状态。

---

### 第一版实现方式

第一版可以 rule-based，不必一开始就上 LLM judge。

输入：

```python

history: list[dict]
episode_log: EpisodeLog | None
```

输出：

```python
PlannerState
```

内部做：

```text
1. 从 history 最后一条 user message 提取 user_goal 的简短描述
2. 从 profile.interests / preferences 初始化 positive_preferences
3. 从用户最近回复中提取 negative_preferences
4. 根据关键词和 simulator metadata 估计 satisfaction_state / satisfaction_score
5. recent_turns = 最近 K 个 history turn
6. ad_events = episode_log.steps 中的 ad_event
7. unresolved_items 暂时用规则或空列表
```

后续可替换为 LLM Monitor，但接口不变。

旧的这些不作为主路径：

原因：action 已经不是“先判断广告，再判断 topic”，而是一维动作选择。

## 8. RewardCalculator 设计

### 修改位置

```text
reward.py
```

### 输入输出

```python
compute(
    state: PlannerState,
    action: PlannerAction,
    next_state: PlannerState,
    info: dict,
) -> float
```

为了 debug，可以额外返回 breakdown，但主训练接口只用 scalar：

```python
RewardResult(
    total: float,
    task: float,
    satisfaction: float,
    continuation: float,
    ad: float,
    terminal: float,
)
```

这里要注意：breakdown 只用于训练日志，不给 metrics 用。

---

### 公式

按 formulation 第一版严格实现：

```text
r_t =
λ_task * η_task
+ λ_sat * ξ_sat
+ λ_cont * κ_cont
+ λ_ad * α_ad
```

其中四项分别是任务推进、满意度变化、对话持续性、广告商业收益。

---

### 第一版计算方式

```text
η_task:
    使用 next_state.belief.task_progress_score
    或 next_state - state 的 progress delta

ξ_sat:
    next_state.belief.satisfaction_score
    - state.belief.satisfaction_score

κ_cont:
    done=False -> 1.0
    done=True 且 exit_reason=task_completed -> 0.5 或 0.0
    done=True 且 exit_reason in {ad_annoyance, task_failed} -> 0.0

α_ad:
    if action.action_type == "ad":
        topic_pool.get_item(action.item_id).reward_value
    else:
        0.0
```

### terminal reward

在 episode 结束时加：

```text
task_completed -> +R_success
ad_annoyance -> -R_annoy
task_failed -> -R_fail
neutral / max_turns -> 0 or weak penalty
```

reward 文件里明确 terminal reward 用于区分任务完成、广告反感退出、任务未完成退出和中性结束，避免把所有结束都视为负反馈。

## 9. MetricsCalculator 设计

### 修改位置

```text
metrics.py
```

### 输入

只接受：

```python

```

不能接受：

```text
total_reward
reward_breakdown
discounted_return
avg_task_reward
```

metrics 文件明确说，metrics 不能直接依赖自定义 reward，否则等于用方法自己的目标函数评价自己。

---

### 第一版指标

需要实现：

```text
TaskSuccessRate
EarlyFailureRate
Revenue
RevenueDensity
AdExposureRate
ParetoVector
```

具体：

```text
TaskSuccessRate:
    exit_reason == task_completed 的 episode 占比

EarlyFailureRate:
    T < T_max 且 exit_reason in {task_failed, ad_annoyance}

Revenue:
    sum I(ad_outcome in {clicked, accepted}) * v(ad_id)

RevenueDensity:
    Revenue / T

AdExposureRate:
    ad_exposure_count / T

ParetoVector:
    (TaskSuccessRate, RevenueDensity)
```

metrics 设计文件里这几项都有明确公式，Revenue 只在广告 outcome 为 clicked 或 accepted 时计入，RevenueDensity 用于避免策略靠拖长对话堆总收益，AdExposureRate 衡量广告负担。

## 10. Episode / exit_reason 设计



## 12. run_simulation.py 设计

### 修改位置

```text
scripts/run_simulation.py
```

### 新主循环

旧逻辑：

```python
bot_utterance = agent.act(history)
history, reward, done, info = env.step(bot_utterance)
agent.observe(user_response)
```

新逻辑：

```python
state = env.reset(profile, initial_query)

    action = planner.act(state, env.action_space)
    next_state, reward, done, info = env.step(action)
    planner.update(state, action, reward, next_state, done)
    state = next_state

metrics = MetricsCalculator.compute([env.episode_log])
```

# 五、几个实现中需要保留讨论的点

## 1. 广告 reward 是否只看展示

当前 reward 设计里，广告商业收益是：

```text
action 是广告 -> α_ad = v(ad)
```

也就是展示即有商业收益。metrics 里 Revenue 则是 clicked / accepted 才计入。这个差异是合理的，但实验中可能导致 Planner 偏向插广告。第一版按 formulation 写；后续如果发现过度插广告，再讨论是否把 reward 的广告项改成 outcome-dependent。

---

## 2. task_completed 如何判断

这是目前代码落地最不确定的地方。可以有三种方案：

```text
1. simulator metadata 显式返回 task_completed
2. Monitor rule-based 判断
3. LLM judge 判断
```

第一版建议先做 rule-based + metadata fallback，后续再加 LLM judge。

---

## 3. DialogueAgent 是否保留

我的建议是：主流程不再使用 `DialogueAgent`。

理由是 formulation 中 `DialogueEnv.step(action)` 应该显式包含 Speaker、UserSim、Monitor、RewardCalculator。如果继续让 `DialogueAgent.act()` 包起来，就会把 `s_t -> a_t -> y_t` 的边界藏掉，不利于 RL trajectory 收集。

可以暂时保留文件，但 `run_simulation.py` 不再使用它。

---
