## 1. 五个 processed 文件的总体关系

可以先把它理解成下面这张逻辑图：

```text
RealPref 用户数据
   │
   ├─→ user_profiles.json
   │       ├─ uid
   │       ├─ interests
   │       ├─ preferences
   │       ├─ intent_distribution
   │       ├─ patience / ad_sensitivity / topic_depth_pref / max_turns
   │       │
   │       └─ uid 对应 initial_queries.json
   │
   ├─→ initial_queries.json
   │       ├─ profile_based_queries：根据 intent_distribution 从 Infinite-Chats 采样
   │       ├─ preference_queries：根据用户前两个偏好生成
   │       └─ random_queries：从 Infinite-Chats 随机采样
   │
   ├─→ topic_pool.json
   │       ├─ Infinite-Chats 的全局意图/话题类别
   │       └─ RealPref 的偏好主题，例如 Food: Diet
   │
   └─→ mappings.json
           ├─ preference_to_intent
           └─ intent_to_preference


ad_pool.json
   │
   ├─ 广告类别 category
   ├─ 广告子类 subcategory
   ├─ target_interests
   ├─ target_intents
   └─ reward_value / pitch_template

AdMatcher:
   user_profile.interests
   + current_topic
   + current_intent
   + ad.target_interests / ad.category / ad.target_intents
   → ad.relevance
   → ProfileSimulator 的 AdReactionModel
   → read_ad / reject_ad / satisfaction impact
```

## 2. `user_profiles.json` 结构

这是用户画像主表，格式是：

```python
{
  "00001": {
    "uid": "00001",  # 用户唯一编号
    
    "demographics": {...},  # 人口统计信息，例如姓名、出生年份、性别、教育、职业、经济状态、婚姻状态等
    
    "persona": "...",  # 自然语言形式的用户画像描述，用于模拟用户背景、兴趣、行为风格和生活经历
    
    "interests": [...],  # 用户的粗粒度兴趣类别，例如 Food、Travel、Entertainment、Lifestyle 等
    
    "interest_strength": {...},  # 每个兴趣类别对应的强度分数，例如 {"Food": 0.53, "Travel": 0.65}
    
    "preferences": [...],  # 用户的细粒度偏好条目列表；每条通常包含 Type、Topic、Preference
    
    "preference_history": [...],  # 偏好形成历史或事件依据，通常解释某个 preference 是如何由过去经历产生的
    "life_events": [...],  # 用户人生经历事件，例如教育、工作、旅行、获奖、生活变化等
    
    "intent_distribution": {...},  # 用户初始查询意图分布，例如 Recommendations、Skill Development 等；用于生成 profile_based_queries
    
    "patience": 0.8,  # 用户耐心参数

    "ad_sensitivity": 0.45,  # 用户广告敏感度

    "topic_depth_pref": 4,  # 用户偏好的话题深入程度

    "max_turns": 19  # 用户最大可持续对话轮数
  },

  ...  # 其他用户画像，例如 00002、00003，一直到 00147
}
```

### 数量统计

| 项目                     |                数量 |
| ---------------------- | ----------------: |
| 用户总数                   |               147 |
| 用户 ID 范围               | `00001` 到 `00147` |
| 有 10 条 preferences 的用户 |               100 |
| preferences 为空的用户      |                47 |
| 平均每个用户兴趣数              |            约 4.82 |
| `ad_sensitivity` 范围    |       0.40 到 0.67 |
| `patience` 范围          |       0.80 到 0.90 |
| `topic_depth_pref` 范围  |             2 到 5 |
| `max_turns` 范围         |           15 到 30 |

### 用户兴趣类别

`interests` 主要来自 RealPref 偏好主题的一级类别，例如：

```text
Food
Travel
Entertainment
Lifestyle
Work & Living
Social
Education
```

统计如下：

| 兴趣            | 出现用户数 |
| ------------- | ----: |
| Entertainment |   132 |
| Lifestyle     |   115 |
| Work & Living |   109 |
| Travel        |   103 |
| Food          |   100 |
| Social        |    86 |
| Education     |    64 |

注意：**用户兴趣中没有 `Fitness` 和 `Beauty` 这两个一级兴趣**，但是广告池里有 `Fitness` 和 `Beauty` 类广告。这会导致后面广告匹配出现覆盖问题。

## 3. `initial_queries.json` 结构

这是每个用户的初始查询候选库，和 `user_profiles.json` 是 ==**uid 一一对应** 的==。

格式是：

```python
{
  "00001": {
    "profile_based_queries": [
      {
        "query": "...",
        "source_category": "Recommendations",
        "related_categories": ["Recommendations"]
      }
    ],
    "preference_queries": [
      {
        "query": "...",
        "source_topic": "Food: Diet",
        "preference": "..."
      }
    ],
    "random_queries": [
      {
        "query": "...",
        "categories": ["Creative Content Generation", "Problem Solving"]
      }
    ]
  }
}
```

### 数量统计

| 类型                    |   总数 | 说明                                             |
| --------------------- | ---: | ---------------------------------------------- |
| profile_based_queries |  496 | 根据用户 `intent_distribution` 从 Infinite-Chats 采样 |
| preference_queries    |  200 | 根据用户前两个 RealPref preferences 生成                |
| random_queries        |  441 | 每个用户固定 3 条随机查询                                 |
| 总查询数                  | 1137 | 全部用户合计                                         |

每个用户查询数并不完全一样，范围是 4 到 10 条，平均约 7.73 条。

### 生成逻辑

在 `scripts/process_data.py` 中：

```python
for uid, profile in user_profiles.items():
    # 1. 根据 intent_distribution 采样 5 次
    # 2. 根据 profile.preferences[:2] 生成 preference_queries
    # 3. 从 Infinite-Chats 随机采样 3 条 random_queries
```

所以：

```text
user_profiles.intent_distribution
    → profile_based_queries

user_profiles.preferences[:2]
    → preference_queries

Infinite-Chats 全局查询库
    → random_queries
```

但当前 `IntegratedCorpus._load_processed_data()` 里有一个问题：它加载 `initial_queries.json` 时，只扁平化了 `profile_based_queries` 和 `preference_queries`，**没有使用 `random_queries`**。而且 `preference_queries` 被统一放入 `"Other"` 类别，丢失了原来的 `source_topic` 分类结构。

## 4. `mappings.json` 结构

这个文件是偏好领域和意图类别之间的映射。结构是：

```python
{
  "preference_to_intent": {
    "Education": ["Skill Development", "Concept Explanations"],
    "Food": ["Recommendations", "Advice & Recommendations"],
    ...
  },
  "intent_to_preference": {
    "Skill Development": ["Education", "Work & Living"],
    "Recommendations": ["Food", "Travel", "Entertainment"],
    ...
  }
}
```

### `preference_to_intent`

| RealPref 一级偏好领域 | 映射到的 Infinite-Chats 意图                       |
| --------------- | -------------------------------------------- |
| Education       | Skill Development, Concept Explanations      |
| Food            | Recommendations, Advice & Recommendations    |
| Travel          | Recommendations, Planning and Logistics      |
| Entertainment   | Creative Content Generation, Recommendations |
| Lifestyle       | Personal Advice, Advice & Recommendations    |
| Work & Living   | Problem Solving, Skill Development           |
| Social          | Personal Advice, Communication Styles        |

这个映射主要用于两件事：

```text
1. 从用户 preferences 推断 intent_distribution
2. 根据 intent_distribution 生成 profile_based_queries
```

例如用户有很多 `Food:*` 偏好，则其 `Recommendations` 和 `Advice & Recommendations` 的概率会上升。

但是注意：**当前运行时代码并没有真正读取 `mappings.json`。** 实际使用的是 `src/recdiag/data/processors/intent_mapper.py` 里硬编码的 `PREFERENCE_TO_INTENT`。所以你改 `mappings.json`，不一定会影响运行，除非代码也改成读取这个文件。

## 5. `topic_pool.json` 结构

格式是：

```python
{
  "topics": [
    {
      "topic_id": "topic_ic_0",
      "name": "Abstract Conceptual Questions",
      "category": "Education",
      "source": "infinite_chats",
      "sample_queries": [...],
      "query_count": 2602
    },
    {
      "topic_id": "topic_rp_333",
      "name": "Food: Diet",
      "category": "Food",
      "source": "realpref",
      "sample_queries": [...],
      "query_count": 311
    }
  ]
}
```

### 数量统计

| 来源             |  数量 |
| -------------- | --: |
| Infinite-Chats | 333 |
| RealPref       |  39 |
| 总话题数           | 372 |

### topic 的字段含义

| 字段               | 含义                                      |
| ---------------- | --------------------------------------- |
| `topic_id`       | 话题 ID，例如 `topic_ic_0`、`topic_rp_333`    |
| `name`           | 原始话题名，例如 `Food: Diet`                   |
| `category`       | 通过 `_infer_topic_category()` 粗略推断出的上层类别 |
| `source`         | `infinite_chats` 或 `realpref`           |
| `sample_queries` | 该话题下的样例 query                           |
| `query_count`    | 该话题关联 query 数量                          |

### 关键问题

`topic_pool.json` 是离线处理产物，但在当前 `run_simulation.py` 中，并没有直接读取它。

当前运行时是这样构造 `TopicPool` 的：

```python
pool = TopicPool()
all_queries = self.corpus.get_all_queries()

for topic in profile.interests:
    sample_queries = all_queries.get(topic, [])
    if sample_queries:
        pool.add_topic(topic, sample_queries[0])
```

也就是说，运行时不是把 `topic_pool.json` 里的 372 个 topic 加进动作空间，而是根据用户兴趣 `profile.interests` 去 `all_queries` 里找同名 key。

这会产生一个潜在错位：

```text
profile.interests: Food / Travel / Education / Lifestyle / ...
all_queries keys: Recommendations / Skill Development / Creative Content Generation / Other / ...
```

如果 `all_queries` 里没有 `"Food"` 这个 key，那么 `Food` 兴趣不会产生对应 topic。

因此，**当前代码里的 `topic_pool.json` 更像是一个已处理数据文件，但没有真正接入运行主流程。**

## 7. `AdMatcher` 到底怎么计算 relevance

核心代码在：

```text
src/recdiag/data/processors/ad_matcher.py
```

`AdMatcher.match_ads()` 输入：

```python
match_ads(
    user_profile,
    current_topic="",
    current_intent=""
)
```

输出是广告列表，每个广告带一个运行时字段：

```python
ad.relevance
```

### 7.1 候选广告召回逻辑

当前候选广告不是直接扫描所有广告，而是分三步：

```python
# 1. 根据用户兴趣召回
for interest in user_profile.interests:
    if interest in self._category_index:
        for ad_id in self._category_index[interest]:
            matched_ads.append(ad)
            break

# 2. 根据 current_topic 召回
if current_topic in self._category_index:
    add ads under current_topic

# 3. 根据 current_intent 召回
if current_intent in self._category_index:
    add ads under current_intent
```

其中 `self._category_index` 的 key 是广告的 `category`，也就是：

```text
Food
Travel
Fitness
Beauty
Education
```

所以第一步实际上要求：

```text
user_profile.interests 中的某个兴趣
必须刚好等于广告 category
```

例如：

```text
user interest = Food      → 可以召回 Food 广告
user interest = Travel    → 可以召回 Travel 广告
user interest = Education → 可以召回 Education 广告
user interest = Lifestyle → 不能直接召回，因为 ad category 没有 Lifestyle
user interest = Social    → 不能召回
user interest = Work & Living → 不能召回
user interest = Entertainment → 不能召回
```

这就是一个关键问题：**`target_interests` 没有用于候选广告召回，只用于后续 relevance 打分。**

例如 Peloton 的 `target_interests` 包含 `Lifestyle`：

```python
target_interests=["Fitness", "Health", "Lifestyle"]
```

但由于它的广告 `category` 是 `Fitness`，而用户兴趣里通常没有 `Fitness`，所以很多 Lifestyle 用户并不会召回 Peloton 广告。

### 7.2 relevance 计算公式

代码公式是：

```python
relevance = 0.4 * interest_match
          + 0.3 * topic_match
          + 0.3 * intent_match
```

具体：

```python
interest_match = 1
if any(user_interest in ad.target_interests)

topic_match = 1
if current_topic.lower() in ad.category.lower()

intent_match = 1
if any(ad_target_intent in current_intent.lower())
```

也就是：

| 分量               |  权重 | 匹配依据                                                  |
| ---------------- | --: | ----------------------------------------------------- |
| `interest_match` | 0.4 | `user_profile.interests` 是否和 `ad.target_interests` 相交 |
| `topic_match`    | 0.3 | `current_topic` 是否是 `ad.category` 的子串                 |
| `intent_match`   | 0.3 | `current_intent` 是否包含广告的 `target_intents`             |

### 7.3 当前代码中的实际问题

这里有几个明显问题。

第一，`trigger_keywords` 虽然在广告里定义了，也建立了 `_keyword_index`，但当前 `match_ads()` **完全没有使用 trigger_keywords**。所以“用户当前说了 hungry / hotel / course”不会直接触发广告。

第二，`current_intent` 在 `BasePlanner.should_insert_ad()` 中传的是空字符串：

```python
available_ads = self.ad_matcher.match_ads(
    self.user_profile,
    current_topic=self._get_current_topic_id() or "",
    current_intent=""
)
```

所以运行时 `intent_match` 通常恒为 0。

第三，`intent_match` 的写法本身也有大小写 bug：

```python
if intent in current_intent.lower():
```

`intent` 是 `"Recommendations"`，而 `current_intent.lower()` 是 `"recommendations"`，大小写不一致，所以即使传入 `"Recommendations"`，也匹配不上。更合理的写法应该是：

```python
if intent.lower() in current_intent.lower():
```

第四，`current_intent in self._category_index` 这一步基本无效，因为 `_category_index` 的 key 是广告类别 `Food / Travel / Fitness / Beauty / Education`，而不是意图类别 `Recommendations / Skill Development / ...`。

第五，候选召回时每个兴趣类别只取第一个广告：

```python
for ad_id in self._category_index[interest]:
    matched_ads.append(ad)
    break
```

所以一个 Food 用户一般只会先召回 `food_0`，而不是 Food 下的全部 3 个广告。除非 `current_topic="Food"`，才会把 Food 类别下的其他广告也补进来。

## 8. 当前 relevance 的实际分布很可能偏低

我按当前 `ad_matcher` 逻辑对 147 个用户做了静态模拟：只传 `profile`，不传 `current_topic` 和 `current_intent`。

结果是：

| 结果               |      数量 |
| ---------------- | ------: |
| 能匹配到至少一个广告的用户    |     118 |
| 完全匹配不到广告的用户      |      29 |
| 匹配到的广告 relevance | 全部是 0.4 |

原因是：

```text
没有 current_topic → topic_match = 0
没有 current_intent → intent_match = 0
只要兴趣和 target_interests 相交 → interest_match = 1

所以 relevance = 0.4
```

也就是说，在当前默认调用下：

```python
relevance = 0.4
```

很常见。

而 `AdReactionModel` 里 relevance 只有在大于 0.7 时才会提供接受概率加成：

```python
if context.ad_relevance > 0.7:
    prob += 0.3
```

所以如果 relevance 长期是 0.4 或默认 0.5，那么广告接受概率不会得到 relevance bonus。

这可以解释你前面怀疑的现象：**ad_relevance 没有发挥作用，确实可能是广告接受率偏低的重要原因之一。**

## 9. AdMatcher 不直接计算满意度，它只计算 ad_relevance

需要区分三个东西：

```text
AdMatcher:
    计算 ad.relevance

Planner:
    根据 Monitor 信号判断是否插广告

ProfileSimulator / AdReactionModel:
    根据 ad_relevance + 用户状态 决定是否 read_ad，并更新 satisfaction / patience
```

`AdReactionModel` 的输入是：

```python
AdReactionContext(
    current_topic_depth,
    patience,
    satisfaction,
    ad_relevance,
    ad_frequency,
    user_ad_sensitivity
)
```

接受概率大致是：

```python
prob = 0.3

if current_topic_depth >= 3:
    prob += 0.2
elif current_topic_depth <= 1:
    prob -= 0.3

prob *= satisfaction
prob *= patience

if ad_frequency > 2:
    prob -= 0.2 * (ad_frequency - 2)

if ad_relevance > 0.7:
    prob += 0.3

prob *= (1 - user_ad_sensitivity * 0.5)
```

所以 relevance 的影响是阈值型的：

```text
ad_relevance <= 0.7 → 没有接受概率加成
ad_relevance > 0.7  → +0.3 接受概率
```

这很关键。因为当前 AdMatcher 里 relevance 常见值是：

```text
0.4：只兴趣匹配
0.7：兴趣 + topic 匹配
1.0：兴趣 + topic + intent 匹配
```

但代码判断是：

```python
if context.ad_relevance > 0.7
```

不是：

```python
>= 0.7
```

所以 `0.7` 也拿不到 bonus。只有 `1.0` 才稳定拿到 bonus。但由于 `intent_match` 目前基本失效，`1.0` 很难出现。

## 10. 当前代码里还有几个重要结构问题

### 问题 1：`user_profiles.json` 和 `UserProfile` dataclass 不一致

`user_profiles.json` 里有：

```python
patience
ad_sensitivity
topic_depth_pref
max_turns
preference_history
life_events
```

但是 `src/recdiag/core/profile.py` 里的 `UserProfile` 只定义了：

```python
uid
demographics
persona
interests
interest_strength
preferences
intent_distribution
```

而 `IntegratedCorpus._load_processed_data()` 又试图这样构造：

```python
UserProfile(
    ...
    patience=...,
    ad_sensitivity=...,
    topic_depth_pref=...,
    max_turns=...
)
```

在这个 zip 版本里，这会导致 `TypeError: unexpected keyword argument`。除非你们当前工作区已经修过，否则这里是一个明确的结构不一致问题。

### 问题 2：`ProfileSimulator` 用的是 `profile.ad_sensitivity`，不是 `SimulatorState.ad_sensitivity`

`SimulatorState` 里有：

```python
ad_sensitivity: float = 0.5
```

但广告反应时用的是：

```python
user_ad_sensitivity=self.profile.ad_sensitivity
```

所以如果 `UserProfile` 没有 `ad_sensitivity` 字段，会直接出问题。即使 `sim_params` 里传了 `ad_sensitivity`，也不会被 `_handle_ad()` 用到。

### 问题 3：`Planner` 在 `run_simulation.py` 里没有接入 `ad_matcher` 和 `user_profile`

`BasePlanner.should_insert_ad()` 开头有：

```python
if self.ad_matcher is None or self.user_profile is None:
    return AdInsertionDecision(False, reason="no_ad_matcher_or_profile")
```

但是 `run_simulation.py` 里是这样创建 Planner 的：

```python
planner = LLMPlanner(topic_pool=pool, generator=generator, monitor=monitor)
```

没有传：

```python
ad_matcher=self.ad_matcher
user_profile=profile
```

所以专门的广告插入路径实际上会被关掉。

当前广告仍可能作为 `TopicPool` 里的普通 action 被 LLMPolicy 选中，但这种路径生成的 `AgentUtterance` metadata 只有：

```python
metadata={"topic_id": self.current_topic_id}
```

没有：

```python
"ad_relevance": ad.relevance
```

于是 `ProfileSimulator._handle_ad()` 会使用默认值：

```python
ad_relevance = bot_utterance.metadata.get("ad_relevance", 0.5)
```

也就是说，**如果广告是作为普通 TopicPool action 被选中的，用户反应模型看到的 ad_relevance 很可能恒定为 0.5。**

这就是一个非常关键的断裂点。

## 11. 最核心的对应关系总结

### 用户侧

```text
user_profiles.json
    uid
    interests: Food / Travel / Lifestyle / ...
    preferences: Food: Diet / Travel: Accommodation / ...
    intent_distribution: Recommendations / Skill Development / ...
    ad_sensitivity
    patience
```

### 查询侧

```text
initial_queries.json
    uid 对应 user_profiles.uid

    profile_based_queries:
        来源于 intent_distribution

    preference_queries:
        来源于 preferences[:2]

    random_queries:
        来源于 Infinite-Chats 随机采样
```

### 映射侧

```text
mappings.json
    preference domain → intent category

例如:
    Food → Recommendations, Advice & Recommendations
    Travel → Recommendations, Planning and Logistics
```

### 话题侧

```text
topic_pool.json
    全局话题池
    372 个 topic
    333 来自 Infinite-Chats
    39 来自 RealPref

但当前主运行流程没有直接加载 topic_pool.json。
```

### 广告侧

```text
ad_pool.json
    10 个广告
    category: Food / Travel / Fitness / Beauty / Education
    target_interests: 面向哪些用户兴趣
    target_intents: 面向哪些用户意图
    reward_value: 点击收益
    pitch_template: 广告话术
```

### 匹配侧

```text
AdMatcher.match_ads(profile, current_topic, current_intent)

候选召回:
    user_profile.interests ∩ ad.category
    current_topic == ad.category
    current_intent == ad.category  # 这个逻辑实际不合理

相关性:
    0.4 * 用户兴趣是否命中 ad.target_interests
  + 0.3 * current_topic 是否命中 ad.category
  + 0.3 * current_intent 是否命中 ad.target_intents
```

## 12. 判断：当前设计意图是合理的，但实现没有完全对齐

设计上它想表达的是：

```text
用户画像兴趣
+ 当前对话主题
+ 当前对话意图
+ 广告目标兴趣
+ 广告目标意图
→ 广告相关性
→ 用户广告反应
→ 接受/拒绝/满意度变化
```

这个思路是合理的。

但当前实现里：

```text
1. topic_pool.json 没有真正接入运行主流程；
2. mappings.json 没有真正作为运行时配置读取；
3. ad_pool.json 也不是从 processed 文件加载，而是从代码 AD_TEMPLATES 重建；
4. target_interests 没有参与候选召回，只参与 relevance；
5. target_intents 基本没有生效；
6. trigger_keywords 完全没有生效；
7. ad_relevance 经常是 0.4 或默认 0.5；
8. AdReactionModel 只有 relevance > 0.7 才给接受概率加成；
9. run_simulation.py 创建 Planner 时没有传 ad_matcher 和 user_profile，专门广告插入路径可能根本没开；
10. UserProfile 数据字段和 dataclass 定义不一致。
```

所以你的直觉是对的：**当前 user、topic、intent、interest、ad 之间的结构关系在数据层是存在的，但在运行时代码里没有被充分使用。尤其是 ad_relevance 与用户接受概率之间的链路，很可能没有按设计发挥作用。**
